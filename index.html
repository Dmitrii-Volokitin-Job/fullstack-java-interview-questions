Interview Strategy

Tactics that separate seniors from juniors

🎯

STAR Method

Situation → Task → Action → Result. Use this structure for every behavioural question to sound concrete and credible.

🤔

Think Aloud

Verbalise your reasoning before and during coding. Interviewers hire for thinking process, not just final answer.

⚖️

Trade-offs First

Senior engineers don't say "use X". They say "X solves A but costs B; given your constraints I'd choose X because…"

❓

Ask Clarifying Qs

For any system design prompt, spend 2 min asking about scale, SLAs, read/write ratio before proposing anything.

🔄

Iterate Visibly

Start with brute force or naive solution, acknowledge limitations, then refine. Shows intellectual honesty.

📝

Prepare Questions

Always have 3 smart questions for the interviewer: team tech debt approach, deployment cadence, on-call culture.

Quick Reference Cheat Sheet

Concepts interviewers love to ask about

Concept Explanation — what it means, why it matters, what to say in an interview Topic
@Transactional propagation Controls what happens when one @Transactional method calls another. REQUIRED (default): if a transaction already exists, join it — if not, create one. REQUIRES_NEW: always suspend the current transaction and open a brand new one, even if one is already running — useful when you want to commit something independently (e.g. an audit log entry) even if the outer transaction rolls back. Common interview trap: calling a @Transactional method on the same class bypasses the proxy → propagation doesn't apply. Spring
N+1 Problem A performance bug caused by lazy loading in JPA/Hibernate. If you load 100 orders and each order lazily loads its items, Hibernate runs 1 query for the orders + 100 separate queries for items = 101 queries total. Fix: use JOIN FETCH in JPQL (SELECT o FROM Order o JOIN FETCH o.items) or add @EntityGraph to the repository method. This merges it into a single SQL JOIN. If you're not sure it's happening: enable SQL logging (spring.jpa.show-sql=true) and count the queries. JPA
Change Detection (Angular) Angular's mechanism for updating the DOM when data changes. Default strategy: Angular checks every component in the entire tree on every async event (click, HTTP response, timer). OnPush strategy: Angular only checks a component when its @Input reference changes or an Observable it subscribes to emits — much faster for large apps. Signals (Angular 17+) make change detection even more precise: only the specific DOM node that reads a signal is updated when the signal changes. Angular
Subject vs BehaviorSubject Both are RxJS Observables that you can push values into manually (.next(value)). The difference is what a new subscriber sees. Subject: new subscriber gets nothing — it only sees future values, not the current state. BehaviorSubject: requires an initial value; new subscriber immediately gets the last emitted value as soon as they subscribe. Use BehaviorSubject for application state (e.g. current user, cart contents) so any component that subscribes can immediately read the current state. RxJS
Pod vs Container In Kubernetes, a Pod is the smallest deployable unit — it's a wrapper around one or more containers that share the same network namespace (they can reach each other via localhost) and the same storage volumes. You almost always have one container per pod. The pod gets one IP address; Kubernetes handles scheduling the pod onto a node. A container is just the Docker image running inside the pod. If the container crashes, Kubernetes restarts it; if the node dies, Kubernetes reschedules the pod on a different node. K8s
Liveness vs Readiness probe Two separate health checks Kubernetes uses. Liveness: "Is this pod still alive?" — if it fails, K8s kills and restarts the pod. Keep this simple (just check the process is responding). Readiness: "Is this pod ready to receive traffic?" — if it fails, K8s removes the pod from the Service's load balancer but does NOT restart it. Use readiness to check external dependencies (DB, Kafka). This way, if your database goes down, pods are taken out of rotation without being restarted (which wouldn't help anyway). K8s
CAP Theorem In a distributed system (multiple servers), network partitions (servers losing connection to each other) will always happen. When that occurs, you can only guarantee one of: Consistency (every read gets the most recent write — all nodes agree) or Availability (the system always responds, even if the response might be stale). Example: PostgreSQL/ZooKeeper choose CP — they refuse writes if they can't reach a quorum. Cassandra/DynamoDB choose AP — they accept writes and reads even when partitioned, and sync up later. Design
Eventual Consistency A consistency model used by AP databases (Cassandra, DynamoDB, Redis Cluster): after a write, different replicas may temporarily return different values, but they are guaranteed to converge to the same value eventually (usually within milliseconds to seconds). The trade-off: you get high availability and low latency, but a user might briefly read stale data after a write. Acceptable for: social media feeds, product catalogue, recommendations. Not acceptable for: bank balance, inventory counts, payment status. Design
CQRS Command Query Responsibility Segregation. Instead of one data model that handles both writes and reads, you split them: Commands (create order, cancel order) write to a normalised, consistent database (PostgreSQL). Queries (search orders, get dashboard stats) read from a separate, denormalised read model (Elasticsearch, Redis, a materialised view) that's optimised purely for reading. The read model is updated asynchronously from the write model via events (Kafka). Use it when your read requirements are too complex or performance-demanding to be served efficiently by the write database. Design
Saga Pattern A way to manage multi-step business processes across multiple microservices without a distributed transaction (which doesn't scale). Each service does its own local database transaction and publishes an event. If a later step fails, each previous service runs a compensating transaction to undo its work (e.g. refund the payment, release the inventory reservation). Two styles: Choreography — services react to each other's events directly (simple, no single point of failure, hard to see the whole flow). Orchestration — a central Saga coordinator calls each service in sequence and handles compensation (easier to debug, but the coordinator is a bottleneck). Design

Abbreviations & Glossary

Every acronym used in this course — explained in plain language

Abbreviation Full name Plain-language explanation Topic
SLA Service Level Agreement A contract between a service provider and a customer that defines the minimum acceptable performance — for example "99.9% uptime" or "response time under 200ms". If the SLA is breached, there are usually financial penalties (credits, refunds). For an API this might mean: "our API will respond within 500ms for 99% of requests." General
SLO Service Level Objective An internal target your engineering team sets — slightly stricter than the SLA. If the SLA is 99.9% uptime, your SLO might be 99.95% so you have a safety margin before the SLA is breached. SLOs drive the alerting and on-call decisions. General
SLI Service Level Indicator The actual measured metric used to track your SLO — e.g., "percentage of HTTP requests that returned 2xx within 300ms over the last 30 days." SLI is the measurement; SLO is the target; SLA is the contract. General
STAR Situation, Task, Action, Result A structured interview storytelling framework. Situation: set the scene. Task: what was your responsibility. Action: what YOU specifically did (use "I", not "we"). Result: the concrete outcome with numbers. Amazon and Google require STAR stories for behavioral questions. Interview
LP Leadership Principle Amazon's 16 core values (Customer Obsession, Ownership, Dive Deep…). Every Amazon behavioral interview question maps to one or more of these. Interviewers are assigned specific LPs to probe in each round. Interview
PR Pull Request A request to merge a feature branch into the main branch. The PR is reviewed by teammates before merging. Also called MR (Merge Request) in GitLab. General
JVM Java Virtual Machine The runtime that executes compiled Java bytecode. It handles memory management (GC), JIT compilation (converting bytecode to native CPU instructions at runtime), and platform abstraction. When you see "tune the JVM", it means adjusting heap size, GC algorithm, thread counts, etc. Java
GC Garbage Collector / Garbage Collection The JVM process that automatically frees memory occupied by objects no longer in use. "GC pause" means the app stops briefly while GC runs. Modern collectors (G1, ZGC) minimise pauses. OOMKilled means the app ran out of memory before GC could free enough. Java
IoC Inversion of Control Instead of your code creating its own dependencies (new DatabaseService()), a container (Spring) creates and injects them for you. You declare what you need; the framework decides when and how to provide it. This makes code easier to test and change. Spring
DI Dependency Injection The most common implementation of IoC — dependencies are "injected" (passed in) via constructor, setter, or field. Spring does this automatically based on @Autowired / constructor scanning. Constructor injection is preferred because it makes dependencies explicit and classes testable without a Spring context. Spring
AOP Aspect-Oriented Programming A way to add cross-cutting behaviour (logging, security checks, transaction management) to methods without changing the method's own code. Spring's @Transactional uses AOP: it wraps your method with begin/commit/rollback logic transparently. Spring
JPA Java Persistence API A Java specification for ORM — it defines how Java objects map to database tables. Spring Data JPA uses Hibernate as its JPA implementation. You write @Entity classes and Repository interfaces; JPA + Hibernate generate the SQL. JPA
ORM Object-Relational Mapping A technique that lets you work with database records as Java objects instead of writing raw SQL. Hibernate is the most popular Java ORM. It maps a "User" class to a "users" table and handles INSERT/UPDATE/SELECT automatically. JPA
DTO Data Transfer Object A simple object used to carry data between layers (controller → service → client) without exposing your full database entity. A UserDTO might have only {id, email, name} while the User entity also has {passwordHash, auditLog, internalFlags}. Spring
JWT JSON Web Token A compact, self-contained token for transmitting identity information. Structured as three Base64-encoded parts: header.payload.signature. The server signs the payload with a secret key; anyone with the public key can verify it hasn't been tampered with — so services don't need to query a database to validate the user. Security
OAuth Open Authorization A protocol for delegated access — "log in with Google" is OAuth. An app gets an access token on behalf of a user without ever seeing their password. OAuth 2.0 is the current standard; JWTs are commonly used as the access token format. Security
OIDC OpenID Connect An identity layer on top of OAuth 2.0 that adds user authentication (who you are) on top of OAuth's authorisation (what you can do). When you "sign in with Google", OIDC gives you an ID Token (a JWT with user info like email, name). Security
SQL Structured Query Language The standard language for relational databases (PostgreSQL, MySQL). SELECT, INSERT, UPDATE, DELETE. Parameterised queries (PreparedStatement) prevent SQL injection by separating the query structure from the user-supplied values. Database
OOM Out of Memory The JVM ran out of heap space and threw OutOfMemoryError. In Kubernetes, "OOMKilled" means the container exceeded its memory limit and was killed by the OS. Fix: increase memory limit, reduce object retention, fix memory leaks, or tune GC. Java
SOLID 5 OOP design principles Single Responsibility — one class, one reason to change. Open/Closed — open for extension, closed for modification. Liskov Substitution — subclasses must be usable wherever base class is expected. Interface Segregation — small focused interfaces over one big one. Dependency Inversion — depend on abstractions, not concrete classes. Design
ACID Atomicity, Consistency, Isolation, Durability The four properties that guarantee reliable database transactions. Atomicity: all-or-nothing (no partial commits). Consistency: only valid data is written. Isolation: concurrent transactions don't see each other's in-progress changes. Durability: committed data survives crashes. Database
API Application Programming Interface A defined contract for how software components talk to each other. A REST API is one where you make HTTP requests (GET, POST…) and get back JSON. "API design" means deciding which endpoints exist, what parameters they take, and what they return. General
REST Representational State Transfer An architectural style for APIs: resources are identified by URLs (/users/42), state is transferred via HTTP methods (GET=read, POST=create, PUT=replace, PATCH=partial update, DELETE=remove), and responses are stateless (server doesn't remember the client between calls). Design
gRPC Google Remote Procedure Call A high-performance framework for service-to-service communication. Uses Protocol Buffers (binary format, much smaller than JSON) over HTTP/2 (faster than HTTP/1.1). Common for internal microservice calls where performance matters. Requires .proto schema files; generates client and server code automatically. Design
CQRS Command Query Responsibility Segregation Separating write operations (Commands: create order, update address) from read operations (Queries: list orders, search products) into separate models or even separate databases. Reads can be optimised projections (denormalised, fast); writes enforce business rules on normalised data. Design
DDD Domain-Driven Design A software design approach where the structure of code mirrors the structure of the real-world business domain. Key concepts: Bounded Context (a boundary within which a term has one precise meaning), Aggregate (a cluster of objects treated as one unit), Entity (has identity), Value Object (defined by its value, no identity). Design
CAP Consistency, Availability, Partition Tolerance A theorem: a distributed system can guarantee at most 2 of these 3 properties simultaneously. In practice, network partitions always happen, so you choose between CP (consistent but may be unavailable during a partition — e.g., ZooKeeper) or AP (available but may return stale data — e.g., Cassandra). Design
BASE Basically Available, Soft state, Eventually consistent The alternative to ACID for distributed/NoSQL systems. Basically Available: the system responds even if some nodes are down. Soft state: state may change over time even without new input (due to replication). Eventually consistent: all nodes will agree eventually, but not necessarily right now. DynamoDB, Cassandra are BASE systems. Design
ACL Anti-Corruption Layer A translation layer between two systems with incompatible models. Prevents the poor design of a legacy system from "leaking" into your clean new service. Your new code works with its own clean types; the ACL translates to/from the legacy model at the boundary. Design
BFF Backend For Frontend A separate API gateway (or aggregation layer) built specifically for one type of client — mobile BFF, web BFF, partner API BFF. Each BFF aggregates data from microservices in the shape the client actually needs, instead of one generic API trying to serve all clients equally badly. Design
CDC Change Data Capture Reading the database's internal write log (WAL — Write-Ahead Log) to stream every INSERT/UPDATE/DELETE to a message broker in real time. Debezium is the most popular CDC tool. Used to implement the Outbox pattern and to sync data between services without polling. Design
WAL Write-Ahead Log PostgreSQL's internal sequential log of every data change before it's applied to the data files. Used for crash recovery. Debezium reads this log to implement CDC — it sees every row change the moment it's committed, without any application code changes needed. Database
mTLS Mutual Transport Layer Security Normal TLS (HTTPS) verifies only the server's certificate. mTLS adds mutual verification — the client also presents a certificate. Used between microservices to prove identity at the network level: only services with a valid certificate (issued by your internal CA) can call other services. Security
K8s Kubernetes An open-source container orchestration platform. It runs your Docker containers, handles scheduling (which container runs on which server), restarts crashed containers, scales them up/down, and routes network traffic. The name comes from Greek for "helmsman"; k-8letters-s = K8s. K8s
HPA Horizontal Pod Autoscaler A Kubernetes controller that automatically adds or removes pod replicas based on CPU, memory, or custom metrics (e.g., Kafka consumer lag). "Horizontal" means more instances of the same pod, as opposed to VPA which gives each pod more CPU/memory. K8s
VPA Vertical Pod Autoscaler Automatically adjusts the CPU and memory requests/limits for individual pods based on their actual usage history. Useful for workloads that can't scale horizontally (e.g., stateful singletons). Can be disruptive because it restarts pods to apply new limits. K8s
CI/CD Continuous Integration / Continuous Delivery (or Deployment) CI: every code commit triggers automated tests — you always know the codebase is in a working state. CD: after tests pass, the code is automatically deployed to staging (Continuous Delivery) or all the way to production (Continuous Deployment). Tools: GitHub Actions, Jenkins, GitLab CI, ArgoCD. DevOps
YAML YAML Ain't Markup Language (recursive acronym) A human-readable data serialisation format used for Kubernetes manifests, Docker Compose files, Spring Boot configuration (application.yml), and GitHub Actions pipelines. Indentation is significant (like Python) — a missing space breaks the file. DevOps
ISR In-Sync Replicas The set of Kafka broker replicas that are fully caught up with the partition leader. When a producer writes with acks=all, Kafka waits until all ISR replicas acknowledge the message — guaranteeing no message loss even if the leader crashes immediately after. Kafka
DLT Dead Letter Topic A Kafka topic where messages that failed processing (after all retries) are sent instead of being discarded. Allows you to inspect, fix, and replay failed messages later without losing them or blocking the main topic. Kafka
Avro Apache Avro A binary data serialisation format used with Kafka. Schema is defined in a JSON file and compiled to Java classes. Much smaller and faster than JSON. Used with Confluent Schema Registry to enforce schema compatibility — you can't publish a message that would break existing consumers. Kafka
PII Personally Identifiable Information Data that can identify a specific person: name, email address, phone number, IP address, passport number. GDPR regulates how PII must be stored, processed, and deleted. Never log PII (passwords, full credit card numbers, SSNs). Mask or hash PII in logs. Security
GDPR General Data Protection Regulation EU regulation governing how personal data is collected, stored, and processed. Requires explicit consent, the right to be forgotten, data minimisation, and breach notification within 72 hours. Violations can result in fines of up to €20M or 4% of global revenue. Security
OWASP Open Worldwide Application Security Project A non-profit that publishes the "OWASP Top 10" — the ten most critical web application security risks (SQL injection, XSS, broken authentication, etc.). Interviewers frequently ask "how do you prevent the OWASP Top 10?" It's the industry-standard security checklist. Security
XSS Cross-Site Scripting An attack where an attacker injects malicious JavaScript into a web page that gets executed in other users' browsers. Prevention: never use innerHTML with user-supplied content; use textContent or a sanitiser; set Content-Security-Policy headers. Security
