Kubernetes Core Concepts

Pods, deployments, statefulsets, services

1
What is the difference between a Pod, Deployment, ReplicaSet, and StatefulSet?
Medium▼
ObjectPurposeKey property
PodSmallest schedulable unit; 1+ containers sharing network/storageEphemeral — if a Pod dies, it's not recreated automatically
ReplicaSetEnsures N copies of a Pod are runningManaged by Deployment; rarely created directly
DeploymentManages ReplicaSets; enables rolling updates and rollbacksStateless workloads — use for web apps, APIs
StatefulSetLike Deployment but Pods get stable identity (pod-0, pod-1)Persistent volume per Pod; ordered startup/shutdown; use for DBs, Kafka
DaemonSetRuns one Pod per Node (or subset of nodes)Node-level agents: log collectors, monitoring, network plugins
Job/CronJobRun-to-completion tasksBatch processing, scheduled cleanup
YAML
# Deployment — stateless web service
apiVersion: apps/v1
kind: Deployment
metadata:
  name: user-service
spec:
  replicas: 3
  selector:
    matchLabels:
      app: user-service
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1         # allow 1 extra pod during update
      maxUnavailable: 0   # zero downtime
  template:
    metadata:
      labels:
        app: user-service
    spec:
      containers:
      - name: user-service
        image: myregistry/user-service:1.2.0
        ports:
        - containerPort: 8080

What is the difference between Liveness, Readiness, and Startup probes?
Hard▼
ProbeWhat happens on failureUse for
LivenessContainer is restartedDetect deadlock/crash that doesn't exit — "is the app still alive?"
ReadinessPod is removed from Service endpoints (not killed)Signal "not ready for traffic yet" — warming up cache, DB migration
StartupContainer is restarted if startup takes too longSlow-starting apps (JVM warm-up) — disables liveness during startup
YAML
containers:
- name: api
  image: myapp:latest
  startupProbe:            # gives app 5*12=60s to start before liveness kicks in
    httpGet:
      path: /actuator/health
      port: 8080
    failureThreshold: 5
    periodSeconds: 12

  livenessProbe:           # restart if app crashes/deadlocks
    httpGet:
      path: /actuator/health/liveness
      port: 8080
    initialDelaySeconds: 0 # startup probe protects the early window
    periodSeconds: 10
    failureThreshold: 3

  readinessProbe:          # remove from LB until ready
    httpGet:
      path: /actuator/health/readiness
      port: 8080
    periodSeconds: 5
    failureThreshold: 3
💡
Spring Boot integration
Spring Boot Actuator exposes /actuator/health/liveness and /actuator/health/readiness out of the box. Enable with management.endpoint.health.probes.enabled=true.
3
What happens when you run `kubectl apply -f deployment.yaml`? Walk through the Kubernetes API flow.
Hard▼
  1. kubectl sends an HTTP PUT/PATCH to the API Server
  2. API Server authenticates (certificates/tokens), authorizes (RBAC), validates (admission controllers), and persists the desired state in etcd
  3. The Deployment Controller (in control plane) detects the change and creates/updates a ReplicaSet
  4. The ReplicaSet Controller detects that desired Pod count doesn't match and creates Pod objects
  5. The Scheduler watches for unscheduled Pods and assigns each to a Node based on resources, affinity, taints
  6. The kubelet on the assigned Node watches for Pods assigned to it, calls the container runtime (containerd) to pull the image and start the container
  7. kubelet reports status back; API Server updates Pod status; Readiness probe gates traffic
💡
Key interview insight
Everything in K8s is a reconciliation loop (controller pattern): watch desired state → compare with actual → take action to converge. This is the fundamental design philosophy of Kubernetes.
4
How do rolling deployments work? How do you do a rollback?
Easy▼

A rolling update gradually replaces old Pods with new ones, controlled by maxSurge (extra Pods allowed during update) and maxUnavailable (how many Pods can be down). With maxUnavailable: 0 you get zero-downtime deployments.

Bash
# Update image — triggers rolling update
kubectl set image deployment/user-service user-service=myregistry/user-service:1.3.0

# Watch rollout progress
kubectl rollout status deployment/user-service

# View rollout history
kubectl rollout history deployment/user-service

# Rollback to previous version
kubectl rollout undo deployment/user-service

# Rollback to specific revision
kubectl rollout undo deployment/user-service --to-revision=2

# Pause a rollout mid-way (canary check)
kubectl rollout pause deployment/user-service
kubectl rollout resume deployment/user-service

6
How does Kubernetes Ingress work? What is an IngressClass?
Medium▼

Ingress is an API object that defines HTTP routing rules. An Ingress Controller (nginx, Traefik, HAProxy) watches Ingress objects and programs the actual load balancer/reverse proxy.

YAML
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  ingressClassName: nginx        # which controller handles this
  tls:
  - hosts:
    - api.example.com
    secretName: api-tls          # cert-manager populates this Secret
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /api/users
        pathType: Prefix
        backend:
          service:
            name: user-service
            port: { number: 80 }
      - path: /api/orders
        pathType: Prefix
        backend:
          service:
            name: order-service
            port: { number: 80 }
7
How does Kubernetes DNS work? How do services communicate by name?
Easy▼

CoreDNS runs as a Deployment in the cluster and serves DNS for all Pods. Every Service gets a DNS entry:

<service-name>.<namespace>.svc.cluster.local

Bash
# Full DNS name (always works across namespaces)
http://user-service.production.svc.cluster.local:8080/api/users

# Short name works within the SAME namespace
http://user-service:8080/api/users

# From order-service in 'staging' namespace calling user-service in 'production'
http://user-service.production.svc.cluster.local:8080

# Verify DNS from inside a pod
kubectl exec -it <pod> -- nslookup user-service
8
What are Network Policies and how do you implement micro-segmentation?
Hard▼

Network Policies are firewall rules at the Pod level. By default, all Pods can talk to each other. A Network Policy enables a default-deny posture for a namespace, then explicitly allows specific traffic.

YAML
# Default deny all ingress in a namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: production
spec:
  podSelector: {}   # selects all pods in namespace
  policyTypes: [Ingress]
---
# Allow order-service to talk to user-service only
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-order-to-user
spec:
  podSelector:
    matchLabels:
      app: user-service
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: order-service
    ports:
    - protocol: TCP
      port: 8080
⚠️
Requires a CNI plugin
Network Policies require a CNI plugin that enforces them — Calico, Cilium, or Weave Net. Flannel does NOT enforce NetworkPolicies. Always verify your CNI supports them in interviews about production security.

ConfigMaps & Secrets

Configuration management, secret rotation

9
What is the difference between ConfigMap and Secret? How do you inject them into Pods?
Medium▼

ConfigMap — stores non-sensitive config (env vars, config files, feature flags). Stored in plaintext in etcd.

Secret — stores sensitive data (passwords, tokens, certs). Stored base64 encoded (NOT encrypted by default — enable etcd encryption at rest or use external secret manager).

YAML
# ConfigMap
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  LOG_LEVEL: INFO
  DB_URL: jdbc:postgresql://postgres:5432/mydb

---
# Secret (base64-encoded values)
apiVersion: v1
kind: Secret
metadata:
  name: app-secrets
type: Opaque
data:
  DB_PASSWORD: cGFzc3dvcmQxMjM=  # echo -n 'password123' | base64

---
# Inject into Pod as env vars
spec:
  containers:
  - name: app
    image: myapp:latest
    envFrom:
    - configMapRef:
        name: app-config
    - secretRef:
        name: app-secrets
    # Or mount Secret as a file (for certs, kubeconfig)
    volumeMounts:
    - name: tls-certs
      mountPath: /etc/ssl/certs
      readOnly: true
  volumes:
  - name: tls-certs
    secret:
      secretName: app-tls
🚫
Security note
Base64 is encoding, not encryption. For production: enable etcd encryption, use external secret managers (AWS Secrets Manager, HashiCorp Vault) with the Secrets Store CSI Driver or External Secrets Operator. Never commit Secret manifests to Git.
10
How do you manage secrets at scale in Kubernetes? Explain External Secrets Operator.
Hard▼

External Secrets Operator (ESO) pulls secrets from external stores (AWS Secrets Manager, Vault, GCP Secret Manager) and creates Kubernetes Secrets automatically. Secrets are never committed to Git.

YAML
# ExternalSecret — syncs from AWS Secrets Manager
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
spec:
  refreshInterval: 1h          # auto-rotate every hour
  secretStoreRef:
    name: aws-secret-store
    kind: ClusterSecretStore
  target:
    name: db-credentials       # creates this K8s Secret
    creationPolicy: Owner
  data:
  - secretKey: DB_PASSWORD
    remoteRef:
      key: /production/myapp/db
      property: password

Alternative: Vault Agent Injector — HashiCorp Vault injects secrets as files into pods via sidecar container annotations.

HPA, VPA, resource limits, node affinity

11
What is the difference between HPA, VPA, and KEDA? How do you configure HPA?
Hard▼
TypeWhat it scalesTrigger
HPANumber of Pods (horizontal)CPU, memory, custom metrics (Prometheus)
VPACPU/memory requests per Pod (vertical)Historical usage — recommends or auto-applies resource adjustments
KEDAPods (including scale-to-zero)Event-driven: Kafka lag, queue depth, HTTP request rate, cron
Cluster AutoscalerNumber of Nodes in the clusterPods that can't be scheduled due to insufficient capacity

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: user-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: user-service
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70       # scale up if avg CPU > 70%
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # wait 5min before scaling down
12
What are resource requests and limits? What happens when a Pod exceeds its memory limit?
Medium▼

Requests — the guaranteed resource amount. Used by the scheduler to decide which node to place the Pod on. Pod is always allocated at least this much.

Limits — the maximum. CPU is throttled (not killed). Memory: if exceeded, the container is OOMKilled (Out Of Memory Kill) and restarted.

YAML
resources:
  requests:
    memory: "256Mi"    # scheduler uses this to find a node
    cpu: "250m"        # 250 millicores = 0.25 CPU
  limits:
    memory: "512Mi"    # OOMKilled if exceeded
    cpu: "1000m"       # throttled (not killed) if exceeded
⚠️
JVM tip
Java doesn't know about container memory limits — it sees the full host RAM and sets heap accordingly. Always set -XX:MaxRAMPercentage=75.0 (Java 10+) so the JVM respects the container limit. Without this you'll get OOMKilled.
13
What are Taints, Tolerations, and Node Affinity? When do you use them?
Hard▼

Taints — applied to Nodes: "repel all Pods unless they tolerate this taint". Prevents pods from being scheduled on specialist nodes (GPU, high-memory).

Tolerations — applied to Pods: "I can run on nodes with this taint".

Node Affinity — attracts pods to nodes matching certain labels (required or preferred).

YAML
# Node taint — only GPU-tolerant pods can run here
kubectl taint nodes gpu-node-1 hardware=gpu:NoSchedule

# Pod toleration + affinity to prefer GPU nodes
spec:
  tolerations:
  - key: hardware
    operator: Equal
    value: gpu
    effect: NoSchedule

  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: hardware
            operator: In
            values: [gpu]
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 1
        preference:
          matchExpressions:
          - key: zone
            operator: In
            values: [eu-west-1a]

Helm

Charts, values, lifecycle, best practices

14
What is Helm and how does it work? Explain Charts, Releases, and Values.
Medium▼

Helm is the package manager for Kubernetes. A Chart is a templated package of K8s manifests. A Release is an installed instance of a chart. Values are configuration overrides that parameterise the templates.

Bash
# Install a chart from a repository
helm repo add bitnami https://charts.bitnami.com/bitnami
helm install my-postgres bitnami/postgresql \
  --set auth.postgresPassword=secret \
  --set primary.persistence.size=20Gi

# Install your own chart with values file
helm install user-service ./charts/user-service \
  -f values/production.yaml \
  --namespace production

# Upgrade a release
helm upgrade user-service ./charts/user-service \
  --set image.tag=1.4.0 -f values/production.yaml

# Rollback to previous version
helm rollback user-service 1

# Dry run — render templates without applying
helm template user-service ./charts/user-service -f values/production.yaml
YAML
# templates/deployment.yaml — Helm template syntax
spec:
  replicas: {{ .Values.replicaCount }}
  template:
    spec:
      containers:
      - name: {{ .Chart.Name }}
        image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
        resources:
          {{- toYaml .Values.resources | nindent 12 }}
        {{- if .Values.env }}
        env:
          {{- range $key, $val := .Values.env }}
          - name: {{ $key }}
            value: {{ $val | quote }}
          {{- end }}
        {{- end }}

How do you structure Helm charts for multiple environments (dev/staging/prod)?
Medium▼
Bash
# Chart structure
charts/user-service/
├── Chart.yaml
├── values.yaml          # defaults (dev-safe)
├── values/

│   └── production.yaml  # production overrides (more replicas, limits)

    ├── configmap.yaml
    └── _helpers.tpl     # named template functions

image:
  repository: myregistry/user-service
  tag: latest
resources:
  limits:
    cpu: 500m
    memory: 256Mi

# production.yaml (overrides)
replicaCount: 3
image:
  tag: 1.4.0              # pinned version, not latest

  limits:
    cpu: "2"
    memory: 1Gi

CI/CD & Docker

Pipelines, Docker best practices, GitOps

16
Write a production-ready Dockerfile for a Spring Boot application.
Medium▼
Dockerfile
# Multi-stage build — builder stage uses full JDK, runtime uses JRE only
FROM eclipse-temurin:21-jdk-alpine AS builder
WORKDIR /app

# Copy dependency files first (layer cache: only re-runs if pom.xml changes)
COPY pom.xml .
COPY .mvn .mvn
COPY mvnw .
RUN ./mvnw dependency:go-offline -q

COPY src src
RUN ./mvnw package -DskipTests -q

# Extract layered jar — enables Docker layer caching for dependencies
RUN java -Djarmode=layertools -jar target/*.jar extract --destination target/extracted

# ── Runtime stage ────────────────────────────────────────
FROM eclipse-temurin:21-jre-alpine AS runtime

# Security: non-root user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

WORKDIR /app

# Layered jar: dependencies, spring-boot-loader, snapshot-dependencies, application
COPY --from=builder /app/target/extracted/dependencies/ ./
COPY --from=builder /app/target/extracted/spring-boot-loader/ ./
COPY --from=builder /app/target/extracted/snapshot-dependencies/ ./
COPY --from=builder /app/target/extracted/application/ ./

EXPOSE 8080

# JVM tuned for containers
ENTRYPOINT ["java",
  "-XX:MaxRAMPercentage=75.0",
  "-XX:+UseContainerSupport",
  "-Djava.security.egd=file:/dev/./urandom",
  "org.springframework.boot.loader.launch.JarLauncher"]
💡
Layer caching tip
Layered jar + multi-stage build means Docker only rebuilds the application layer when your code changes (a few MB), not the dependencies layer (hundreds of MB). This dramatically speeds up CI builds.
17
Describe a complete CI/CD pipeline for a Java microservice deployed to Kubernetes.
Hard▼

A mature pipeline has these stages:

YAML
# GitHub Actions example
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4

    - name: Setup Java 21
      uses: actions/setup-java@v4
      with:
        java-version: '21'
        distribution: 'temurin'
        cache: 'maven'

    - name: Run tests             # unit + integration
      run: ./mvnw verify

    - name: SonarQube analysis    # code quality gate
      run: ./mvnw sonar:sonar -Dsonar.token=${{ secrets.SONAR_TOKEN }}

    - name: Build Docker image
      run: |
        docker build -t $IMAGE:${{ github.sha }} .
        docker push $IMAGE:${{ github.sha }}

  deploy-staging:
    needs: build-and-test
    environment: staging
    steps:
    - name: Helm upgrade staging
      run: |
        helm upgrade --install user-service ./charts/user-service \
          --set image.tag=${{ github.sha }} \
          -f values/staging.yaml \
          --namespace staging --wait --timeout 5m

    - name: Smoke test
      run: curl -f https://staging.api.example.com/actuator/health

  deploy-production:
    needs: deploy-staging
    environment: production       # requires manual approval
    steps:
    - name: Helm upgrade production
      run: |
        helm upgrade --install user-service ./charts/user-service \
          --set image.tag=${{ github.sha }} \
          -f values/production.yaml \
          --namespace production --wait
18
What is GitOps? How do ArgoCD or Flux implement it?
Hard▼

GitOps — Git is the single source of truth for cluster state. Every change goes through a PR; the cluster automatically reconciles toward the Git-declared state. Benefits: audit trail, easy rollback (git revert), self-healing.

ArgoCD watches a Git repo for Kubernetes manifests/Helm charts. When a change is pushed, it applies it to the cluster. If someone manually changes something via kubectl, ArgoCD detects drift and either alerts or auto-syncs back.

YAML
# ArgoCD Application CRD
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: user-service
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/k8s-configs
    targetRevision: main
    path: services/user-service/production
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true     # delete resources removed from git
      selfHeal: true  # auto-revert manual kubectl changes
    syncOptions:
    - CreateNamespace=true

How do you implement a blue-green or canary deployment in Kubernetes?
Hard▼
StrategyHowRisk
Rolling (default)Gradually replace old pods with newBoth versions run simultaneously during update
Blue-GreenRun both in parallel; switch Service selector from blue to green instantlyDouble resource cost; instant traffic switch (no gradual)
CanarySend small % of traffic to new version; monitor errors; gradually increaseMore complex; needs traffic splitting (Istio, Argo Rollouts)
YAML
# Blue-Green: two deployments, one Service
# Switch by changing Service selector
apiVersion: v1
kind: Service
metadata:
  name: user-service
spec:
  selector:
    app: user-service
    version: blue    # change to 'green' to cut over instantly
  ports:
  - port: 80
    targetPort: 8080

# Canary with Argo Rollouts (weights)
apiVersion: argoproj.io/v1alpha1
kind: Rollout
spec:
  strategy:
    canary:
      steps:
      - setWeight: 10    # 10% traffic to new version
      - pause: {duration: 5m}
      - setWeight: 50
      - pause: {duration: 10m}
      - setWeight: 100   # full traffic if no errors
      analysis:
        templates:
        - templateName: error-rate-check

21
How do you troubleshoot a Pod that is stuck in CrashLoopBackOff or Pending?
Medium▼
Bash
# CrashLoopBackOff — container starts then immediately exits
# Step 1: describe the pod — look at Events section
kubectl describe pod <pod-name> -n <namespace>

# Step 2: read logs from crashed container
kubectl logs <pod-name> --previous     # -p = previous container run
kubectl logs <pod-name> -c <container> --previous

# Step 3: exec into running container to investigate
kubectl exec -it <pod-name> -- /bin/sh

# Pending — pod can't be scheduled
# Check events — common causes:
#   Insufficient cpu/memory: nodes don't have enough
#   No nodes match nodeSelector/affinity
#   PersistentVolumeClaim not bound
#   Taint has no matching toleration
kubectl get events -n <namespace> --sort-by='.lastTimestamp'
kubectl describe node <node-name> | grep -A5 Conditions

# ImagePullBackOff — wrong image name, auth issue
kubectl describe pod <pod-name> | grep -A10 Events
22
What is a PersistentVolume and PersistentVolumeClaim? Explain the StorageClass.
Medium▼

PersistentVolume (PV) — a piece of storage provisioned by an admin or dynamically. Lifecycle independent of any Pod.

PersistentVolumeClaim (PVC) — a request for storage by a user (size, access mode, StorageClass). K8s binds the PVC to a matching PV.

StorageClass — defines how PVs are dynamically provisioned (AWS EBS, GCP PD, NFS, etc.).

YAML
# PVC — request 20Gi SSD storage
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
spec:
  accessModes:
  - ReadWriteOnce       # RWO: one node; RWX: many nodes (NFS)
  storageClassName: gp3  # AWS EBS gp3 StorageClass
  resources:
    requests:
      storage: 20Gi

# Mount in StatefulSet
volumeClaimTemplates:   # each replica gets its own PVC automatically
- metadata:
    name: data
  spec:
    accessModes: [ReadWriteOnce]
    storageClassName: gp3
    resources:
      requests:
        storage: 20Gi
23
What is the difference between a Docker image layer and a container? Explain image security best practices.
Medium▼

An image is a read-only stack of layers (each RUN/COPY/ADD instruction creates a layer). A container is a running image with a writable layer on top.

Security best practices:

  • Use distroless or Alpine base images (smaller attack surface)
  • Run as non-root user (USER appuser)
  • Don't install dev tools in production images (multi-stage)
  • Use .dockerignore to exclude secrets, .git, test data
  • Pin base image versions (alpine:3.19 not alpine:latest)
  • Scan images for CVEs: docker scout cves, Trivy, Snyk
  • Never embed secrets — use runtime environment variables
  • Read-only root filesystem: securityContext.readOnlyRootFilesystem: true
