Java 8 — Lambdas & Functional Interfaces: explain with examples
Medium ▼

A lambda is a short anonymous function: (parameters) -> body. It can be used wherever the expected type is a functional interface — any interface with exactly one abstract method (SAM = Single Abstract Method). The annotation @FunctionalInterface enforces this at compile time.

Java 8 — Lambda syntax forms
// 1. Full form: parameter types declared, block body
Comparator<String> c1 = (String a, String b) -> {
    return a.compareTo(b);
};

// 2. Inferred types + expression body (most common)
Comparator<String> c2 = (a, b) -> a.compareTo(b);

// 3. Single parameter — parens optional
Predicate<String> isEmpty = s -> s.isEmpty();

// 4. No parameters
Runnable r = () -> System.out.println("running");

Built-in functional interfaces — memorise these four, they appear constantly:

InterfaceMethodTakesReturnsTypical use
Predicate<T>test(T)Tbooleanfilter(), anyMatch()
Function<T,R>apply(T)TRmap(), transform
Consumer<T>accept(T)TvoidforEach(), side effects
Supplier<T>get()—Tlazy init, orElseGet()
BiFunction<T,U,R>apply(T,U)T, URmerge, combine two values
UnaryOperator<T>apply(T)TT (same type)map when type stays same
Java 8 — Method references (shorter than lambdas)
// Instead of: x -> System.out.println(x)
list.forEach(System.out::println);          // instance method on specific object

// Instead of: s -> s.toUpperCase()
list.stream().map(String::toUpperCase);     // instance method on parameter

// Instead of: () -> new ArrayList<>()
Supplier<List> s = ArrayList::new;       // constructor reference

// Instead of: (a, b) -> Integer.compare(a, b)
Comparator<Integer> cmp = Integer::compare; // static method reference

// Practical: sort list of employees by salary then name
employees.sort(
    Comparator.comparingInt(Employee::getSalary)
              .thenComparing(Employee::getName)
);
Java 8 — Variable capture (effectively final rule)
int threshold = 50_000;  // effectively final — never reassigned
Predicate<Employee> highEarner = e -> e.getSalary() > threshold; // ✅ OK

int counter = 0;
list.forEach(e -> counter++); // ❌ compile error: counter is not effectively final

// Fix for mutable state in lambdas: use AtomicInteger or collect to a result
AtomicInteger count = new AtomicInteger();
list.forEach(e -> count.incrementAndGet()); // ✅ object reference is effectively final
💡
Interview angle
Interviewers often ask "what is a functional interface?" — the answer is any interface with exactly one abstract method. It can have multiple default or static methods. Examples: Runnable (run), Callable (call), Comparator (compare), all java.util.function types.

Java 8 — Streams: complete API, Collectors, flatMap, reduce — with examples
Hard ▼

A Stream is a one-time-use pipeline over a data source. Operations are intermediate (lazy, return a new Stream) or terminal (eager, trigger the whole pipeline and produce a result).

Java 8 — Creating streams
// From a collection
list.stream()
list.parallelStream()

// From values
Stream.of("a", "b", "c")

// Infinite stream — generate a value on demand
Stream.generate(Math::random).limit(10)

// Infinite stream — iterate from a seed
Stream.iterate(0, n -> n + 2).limit(5)  // 0, 2, 4, 6, 8

// From a range of ints (IntStream)
IntStream.rangeClosed(1, 10)           // 1..10 inclusive
Java 8 — Intermediate operations (lazy, return Stream)
employees.stream()
  .filter(e -> e.getDept().equals("Engineering"))  // keep matching
  .map(Employee::getSalary)                          // transform each element
  .mapToInt(Integer::intValue)                       // box to primitive stream
  .distinct()                                        // remove duplicates
  .sorted()                                          // natural order (or pass Comparator)
  .peek(n -> log.debug("salary: {}", n))           // side effect without consuming
  .limit(10)                                        // take first 10
  .skip(2)                                          // skip first 2
  .sum();                                            // terminal (IntStream)
Java 8 — flatMap: flattening nested collections
// Problem: each Order has a List<OrderItem> — you want all items across all orders
List<Order> orders = getOrders();

// WITHOUT flatMap: Stream<List<OrderItem>> — wrong, need Stream<OrderItem>
orders.stream().map(Order::getItems); // ❌ nested

// WITH flatMap: flattens each List into the stream → Stream<OrderItem>
List<String> allProductNames = orders.stream()
    .flatMap(order -> order.getItems().stream()) // flatten
    .map(OrderItem::getProductName)
    .distinct()
    .sorted()
    .collect(Collectors.toList());
Java 8 — reduce: aggregate to a single value
// Sum all salaries (identity = starting value)
int totalSalary = employees.stream()
    .map(Employee::getSalary)
    .reduce(0, Integer::sum);              // 0 + e1 + e2 + e3 + ...

// Without identity — result is Optional (stream could be empty)
Optional<Integer> max = salaries.stream()
    .reduce(Integer::max);

// For simple aggregations, prefer IntStream methods — cleaner
int total = employees.stream()
    .mapToInt(Employee::getSalary)
    .sum();                                // same as reduce but more readable
double avg = employees.stream()
    .mapToInt(Employee::getSalary)
    .average().orElse(0);
Java 8 — Collectors (the most important ones)
// Group employees by department → Map<String, List<Employee>>
Map<String, List<Employee>> byDept =
    employees.stream()
        .collect(Collectors.groupingBy(Employee::getDepartment));

// Count per department → Map<String, Long>
Map<String, Long> countByDept =
    employees.stream()
        .collect(Collectors.groupingBy(Employee::getDepartment, Collectors.counting()));

// Average salary per department
Map<String, Double> avgByDept =
    employees.stream()
        .collect(Collectors.groupingBy(
            Employee::getDepartment,
            Collectors.averagingInt(Employee::getSalary)));

// Split into two groups: earns > 70k vs not
Map<Boolean, List<Employee>> split =
    employees.stream()
        .collect(Collectors.partitioningBy(e -> e.getSalary() > 70_000));

// Join names as comma-separated string
String names = employees.stream()
    .map(Employee::getName)
    .collect(Collectors.joining(", ", "[", "]"));  // [Alice, Bob, Charlie]

// Collect to unmodifiable map
Map<Long, String> idToName = employees.stream()
    .collect(Collectors.toUnmodifiableMap(Employee::getId, Employee::getName));
🚫
Common mistakes
1. Reusing a stream — streams are single-use; a second terminal operation throws IllegalStateException. 2. Using parallelStream() for I/O work — it uses ForkJoinPool.commonPool which has few threads; starving it blocks other parts of the app. 3. sorted() on an unbounded stream — it must see all elements before producing output, causing OOM.

Java 8 — Date/Time API: LocalDate, ZonedDateTime, Duration, Period — why was it introduced?
Medium ▼

Why the old API was replaced: java.util.Date is mutable (can be changed after creation → threading bugs), confuses months (January = 0 — countless production bugs), has no timezone concept built in cleanly, and has terrible API design. Calendar was even worse. The new java.time API (inspired by Joda-Time) is immutable, thread-safe, and expressive.

Java 8 — java.time core classes
// ── LocalDate: a date only, NO time, NO timezone ──
LocalDate today   = LocalDate.now();
LocalDate birthday = LocalDate.of(1990, Month.MAY, 15);  // Month enum avoids 0-index bug
LocalDate nextWeek = today.plusDays(7);
LocalDate parsed   = LocalDate.parse("2024-06-01");

boolean isBefore = birthday.isBefore(today);               // true
int dayOfYear   = today.getDayOfYear();
DayOfWeek dow    = today.getDayOfWeek();                    // MONDAY, TUESDAY...

// ── LocalTime: a time only, NO date, NO timezone ──
LocalTime now      = LocalTime.now();
LocalTime meeting  = LocalTime.of(14, 30);                 // 14:30:00
LocalTime inAnHour = now.plusHours(1).truncatedTo(ChronoUnit.MINUTES);

// ── LocalDateTime: date + time, still NO timezone ──
LocalDateTime dt  = LocalDateTime.of(today, meeting);
LocalDateTime dt2 = LocalDateTime.now().minusMonths(3);
Java 8 — Timezone-aware types
// ZonedDateTime = LocalDateTime + timezone. Use for user-facing datetimes.
ZoneId vienna   = ZoneId.of("Europe/Vienna");
ZonedDateTime zdt = ZonedDateTime.now(vienna);
ZonedDateTime utc = zdt.withZoneSameInstant(ZoneOffset.UTC);  // convert TZ

// Instant = machine time (epoch milliseconds). Use for DB timestamps, Kafka offsets.
Instant now    = Instant.now();
long epochMs   = now.toEpochMilli();
Instant before = now.minus(5, ChronoUnit.MINUTES);

// Convert Instant ↔ ZonedDateTime
ZonedDateTime from = now.atZone(vienna);
Instant back       = from.toInstant();
Java 8 — Duration, Period, DateTimeFormatter
// Duration = time-based (hours, minutes, seconds, nanos)
Duration d = Duration.between(start, end);   // between two Instants or LocalDateTimes
long minutes = d.toMinutes();
Duration twoHours = Duration.ofHours(2);

// Period = date-based (years, months, days)
Period p = Period.between(birthday, today);  // between two LocalDates
int age = p.getYears();
Period threeMonths = Period.ofMonths(3);

// ── Rule: Duration for times, Period for dates ──

// DateTimeFormatter — formatting and parsing
DateTimeFormatter fmt = DateTimeFormatter.ofPattern("dd.MM.yyyy HH:mm");
String formatted  = dt.format(fmt);           // "01.06.2024 14:30"
LocalDateTime parsed = LocalDateTime.parse("01.06.2024 14:30", fmt);

// Thread-safe: DateTimeFormatter is immutable — safe as a static field
private static final DateTimeFormatter ISO = DateTimeFormatter.ISO_LOCAL_DATE;

Database / JPA integration
Spring Data JPA maps LocalDate → SQL DATE, LocalDateTime → SQL TIMESTAMP, Instant → SQL TIMESTAMP (UTC). Always store timestamps as Instant in the DB and convert to the user's timezone in the presentation layer. This avoids DST bugs when servers or databases are in different timezones.

Java 8 — Default & static interface methods: what problem do they solve? What is the diamond problem?
Medium ▼

Before Java 8, adding a new method to a published interface was a breaking change — every class implementing it would fail to compile. default methods solve this: they provide an implementation directly in the interface, so existing implementors inherit it without any changes.

Java 8 — Default methods
interface Greeter {
    String getName();       // abstract — must implement

    default String greet() {          // default — inheritors get this for free
        return "Hello, " + getName();
    }

    static Greeter formal(String name) {   // static — called on the interface itself
        return () -> "Dear " + name;
    }
}

class EnglishGreeter implements Greeter {
    public String getName() { return "Alice"; }
    // greet() is inherited — no need to override
}

// Real-world example: Java itself added List.sort(), Map.forEach(), Iterable.forEach()
// without breaking every existing List / Map / Iterable implementation
List<String> names = ...;
names.sort(Comparator.naturalOrder()); // List.sort() is a default method added in Java 8
names.forEach(System.out::println);     // Iterable.forEach() — also default
Java 8 — Diamond problem: what to do when two interfaces conflict
interface A {
    default String hello() { return "Hello from A"; }
}
interface B {
    default String hello() { return "Hello from B"; }
}

class C implements A, B {
    // ❌ Compile error: C inherits unrelated defaults for hello() from A and B
    // ✅ Must override and explicitly resolve:
    public String hello() {
        return A.super.hello();  // explicitly pick A's version
    }
}

// Rule: class implementation always wins over interface default
// If class C extends a concrete class that also has hello() — that wins over both A and B
💡
Compose with default methods
Comparator, Predicate, and Function all have default methods for chaining: Predicate.and(), Predicate.or(), Predicate.negate(), Function.andThen(), Function.compose(). Example: isActive.and(hasEmail).and(isVerified) — much cleaner than nested ifs.

Java 14–17: Records, Pattern Matching instanceof, Sealed Classes, Text Blocks — all with examples
Medium ▼

Records (Java 16 final) — immutable data carriers. A record auto-generates: canonical constructor, accessors (no getX prefix — just name()), equals(), hashCode(), toString(). Replaces 20+ lines of boilerplate DTO/POJO.

Java 16 — Records
// Declaration: that's it — all the boilerplate is generated
record Point(int x, int y) {}

Point p = new Point(3, 4);
p.x();               // 3  (accessor, NOT getX())
p.y();               // 4
p.toString();        // "Point[x=3, y=4]"

// Compact constructor — validates without repeating assignments
record EmailAddress(String value) {
    EmailAddress {    // no parens — this is the compact constructor
        Objects.requireNonNull(value, "email must not be null");
        if (!value.contains("@")) throw new IllegalArgumentException("invalid email");
        value = value.toLowerCase();  // normalise — compact constructor can reassign params
    }
}

// Records CAN: implement interfaces, have static fields/methods, add custom methods
record UserDto(Long id, String name) implements Serializable {
    static UserDto anonymous() { return new UserDto(0L, "anonymous"); }
    boolean isAnonymous() { return id == 0; }
}

// Records CANNOT: extend classes, be abstract, have mutable (non-final) instance fields

Pattern Matching for instanceof (Java 16 final) — eliminates the explicit cast after an instanceof check.

Java 16 — Pattern Matching instanceof
// Old way — repetitive cast
if (obj instanceof String) {
    String s = (String) obj;   // redundant cast
    System.out.println(s.length());
}

// New way — type test + binding variable in one step
if (obj instanceof String s) {
    System.out.println(s.length());  // s is in scope here, already cast
}

// Can combine with conditions in the same if
if (obj instanceof String s && s.length() > 5) {
    System.out.println(s.toUpperCase());
}

// Useful in equals() — eliminates all casts
@Override
public boolean equals(Object o) {
    return o instanceof Point other
        && x == other.x && y == other.y;
}

Text Blocks (Java 15 final) — multiline strings without escape hell.

Java 15 — Text Blocks
// Old: string concatenation nightmare
String json = "{\n" + "  \"name\": \"" + name + "\"\n}";

// New: Text Block — triple quotes, indentation is stripped automatically
String json = """
        {
          "name": "%s",
          "active": true
        }
        """.formatted(name);

// SQL query — no more string concatenation
String sql = """
        SELECT u.id, u.email, o.total
        FROM users u
        JOIN orders o ON o.user_id = u.id
        WHERE u.active = true
          AND o.created_at > ?
        ORDER BY o.total DESC
        """;

// The closing """ position controls indentation stripping
// Trailing """ on its own line = no trailing newline issue
💡
Records + Spring
Records work perfectly as DTOs with Jackson: @JsonDeserialize works via the canonical constructor. For request bodies, add @RequestBody UserDto dto — Jackson maps JSON to record. For response, just return the record directly. Don't use records for JPA @Entity classes — Hibernate needs mutable objects and a no-arg constructor.

Java 17 — Sealed Classes: what problem do they solve and how do they pair with pattern matching?
Medium ▼

Before sealed classes: a class hierarchy was either completely open (any class can extend) or completely closed (final — nothing can extend). There was no middle ground. Sealed classes let you define exactly which classes are permitted to extend or implement a type — giving you a controlled, known hierarchy.

Java 17 — Sealed classes & permits
// Sealed class: only these three classes may extend Shape
public sealed class Shape permits Circle, Rectangle, Triangle {}

// Each permitted subclass must be one of: final, sealed, or non-sealed
public final class Circle    extends Shape { double radius; }
public final class Rectangle extends Shape { double width, height; }
public non-sealed class Triangle extends Shape {} // anyone can extend Triangle
Java 21 — Sealed classes + pattern switch = exhaustiveness
// The compiler knows ALL possible subtypes, so this switch is exhaustive
// No default needed — compiler warns if you miss a case
double area(Shape s) {
    return switch (s) {
        case Circle    c -> Math.PI * c.radius * c.radius;
        case Rectangle r -> r.width * r.height;
        case Triangle  t -> calculateTriangleArea(t);
    }; // if you add a 4th subclass, this switch won't compile → safety
}

// Practical pattern: Result type (Success or Failure — like Rust's Result<T, E>)
public sealed interface Result<T> permits Result.Success, Result.Failure {
    record Success<T>(T value)           implements Result<T> {}
    record Failure<T>(String errorMsg)  implements Result<T> {}
}

// Usage — exhaustive handling
return switch (processPayment(req)) {
    case Result.Success<Order> s -> ResponseEntity.ok(s.value());
    case Result.Failure<Order> f -> ResponseEntity.badRequest().body(f.errorMsg());
};
💡
Why this matters for APIs
Sealed types express a complete, known set of variants — like an enum but with data. They model domain concepts like PaymentMethod (CreditCard | BankTransfer | Crypto), NotificationChannel (Email | SMS | Push), or HTTP response variants. Adding a new subclass forces you to update every switch — the compiler becomes your safety net.

Java 21 — Virtual Threads (Project Loom): how do they work, and when do they outperform platform threads?
Hard ▼

The problem with platform threads: each Java platform thread maps 1:1 to an OS thread. OS threads are expensive (~1MB stack, kernel context switch). A typical JVM can handle ~2000-10000 concurrent platform threads before memory and scheduling overhead kills throughput. Blocking I/O (DB query, HTTP call) wastes the thread — it just sits there waiting.

Virtual threads are JVM-managed, very lightweight threads (~few KB). The JVM multiplexes thousands of virtual threads onto a small pool of platform "carrier" threads. When a virtual thread blocks (I/O, sleep, lock), the JVM unmounts it from the carrier thread — the carrier is immediately free to run another virtual thread. No OS thread is wasted waiting.

Java 21 — Creating and using virtual threads
// 1. Create a single virtual thread
Thread vt = Thread.ofVirtual().name("my-vt").start(() -> {
    System.out.println("Running in: " + Thread.currentThread().isVirtual()); // true
});

// 2. Executor that creates a new virtual thread per task — the main pattern
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    List<Future<String>> futures = urls.stream()
        .map(url -> executor.submit(() -> httpClient.fetchBody(url)))
        .collect(Collectors.toList());
    // All 10,000 URLs fetched concurrently with just virtual threads
    // Carrier thread pool is typically # of CPU cores (e.g. 8)
}

// 3. Spring Boot 3.2+ — enable with one property (replaces Tomcat thread pool)
// application.yml:
// spring.threads.virtual.enabled: true
// That's it — Tomcat, @Async, @Scheduled all use virtual threads automatically
Java 21 — Virtual threads: what to watch out for
// ⚠️ Pinning: synchronized blocks PREVENT unmounting — the carrier thread is blocked!
synchronized (this) {
    Thread.sleep(1000);  // carrier thread is PINNED for 1 second — defeats the purpose
}
// Fix: replace synchronized with java.util.concurrent.locks.ReentrantLock
private final ReentrantLock lock = new ReentrantLock();
lock.lock();
try { Thread.sleep(1000); } finally { lock.unlock(); } // ✅ virtual thread can unmount

// ⚠️ ThreadLocal at scale: millions of virtual threads × ThreadLocal data = memory pressure
// Java 21 adds ScopedValue as the preferred alternative
ScopedValue<String> USER_ID = ScopedValue.newInstance();
ScopedValue.where(USER_ID, "user-42")
           .run(() -> processRequest());  // USER_ID visible within this scope only

// ⚠️ CPU-bound work: virtual threads don't help — they don't add more CPU
// For CPU work, use parallel streams or a bounded thread pool

The big picture
Virtual threads make the "one-thread-per-request" model viable at very high concurrency — the same model Spring MVC has always used. This is why reactive programming (WebFlux, Project Reactor) is less necessary with Java 21+. The blocking code stays readable, and scalability comes from the JVM runtime, not from async callbacks.

Java 21 — Pattern Matching for switch and Sequenced Collections
Medium ▼

Pattern Matching for switch (Java 21 final) — extends instanceof pattern matching into switch, making it the clean way to deconstruct sealed hierarchies and heterogeneous data.

Java 21 — Pattern switch with type patterns and guards
// Old style: if-else instanceof chain — verbose and error-prone
static String describe(Object obj) {
    if (obj instanceof Integer i) return "int: " + i;
    else if (obj instanceof String s) return "str: " + s;
    else return "unknown";
}

// New: pattern switch — clean, exhaustive (compiler checks for sealed types)
static String describe(Object obj) {
    return switch (obj) {
        case Integer i when i < 0 -> "negative int: " + i;   // guarded pattern
        case Integer i              -> "positive int: " + i;
        case String  s when s.isEmpty() -> "empty string";
        case String  s              -> "str of length " + s.length();
        case null                    -> "null value";  // explicit null handling
        default                      -> "something else";
    };
}

// With sealed types — no default needed, compiler verifies exhaustiveness
static double area(Shape s) {  // Shape is sealed: Circle | Rectangle | Triangle
    return switch (s) {
        case Circle    c             -> Math.PI * c.radius() * c.radius();
        case Rectangle r             -> r.width() * r.height();
        case Triangle  t when t.isRight() -> 0.5 * t.base() * t.height();
        case Triangle  t             -> calculateTriangleArea(t);
    }; // no default — compiler ensures all Shape subtypes are covered
}

// Record deconstruction pattern (also Java 21) — extract record components in case
if (event instanceof OrderCreated(Long orderId, Long customerId, BigDecimal total)) {
    log.info("Order {} for customer {} worth {}", orderId, customerId, total);
}

Sequenced Collections (Java 21) — a new interface hierarchy that adds getFirst(), getLast(), addFirst(), addLast(), reversed() to List, Deque, SortedSet, LinkedHashSet.

Java 21 — SequencedCollection
List<String> list = new ArrayList<>(List.of("a", "b", "c"));

// Before Java 21 — awkward
list.get(0);                     // first
list.get(list.size() - 1);       // last — ugly
new LinkedList<>(list).peekFirst(); // only on Deque

// After Java 21 — every List, Deque, SortedSet has these:
list.getFirst();      // "a" — replaces get(0)
list.getLast();       // "c" — replaces get(size-1)
list.addFirst("z");   // inserts at front
list.addLast("z");    // appends to end
list.removeFirst();   // removes and returns first
list.removeLast();    // removes and returns last
list.reversed();      // returns a reversed VIEW (no copy) — lazy

What is the Stream API and how does lazy evaluation work?
Medium ▼

Streams are a pipeline of operations over a data source. Operations are either intermediate (lazy — return a new Stream, not executed until terminal) or terminal (eager — trigger the pipeline).

Java
var result = employees.stream()
    .filter(e -> e.getSalary() > 50_000)  // intermediate (lazy)
    .map(Employee::getName)                // intermediate (lazy)
    .sorted()                              // intermediate (lazy)
    .collect(Collectors.toList());         // terminal — runs everything NOW

// Short-circuit terminal stops early:
var first = employees.stream()
    .filter(e -> e.getSalary() > 100_000)
    .findFirst()                           // stops after first match
    .orElseThrow();

// Parallel stream — use only for CPU-bound work on large data
long count = bigList.parallelStream()
    .filter(this::isExpensive)
    .count();
🚫
Pitfall
Parallel streams use ForkJoinPool.commonPool. For I/O-bound work (DB calls, HTTP) they can degrade performance by exhausting the shared pool. Use virtual threads or reactive instead.

Explain Optional. When should you NOT use it?
Medium ▼

Optional<T> is a container that may or may not hold a non-null value. It forces callers to handle the absent case explicitly.

Java
// ✅ Good use — method return type signals "may be absent"
public Optional<User> findById(long id) {
    return repo.findById(id); // JPA returns Optional already
}

// ✅ Chain without null checks
findById(id)
    .map(User::getEmail)
    .map(Email::getDomain)
    .orElse("unknown");

// 🚫 Never use Optional as a field or constructor param
private Optional<String> email; // BAD — not serializable, memory overhead

// 🚫 Never call .get() without isPresent() — defeats the purpose
user.get().getName(); // throws NoSuchElementException if empty

// ✅ Prefer orElseThrow with meaningful exception
user.orElseThrow(() -> new UserNotFoundException(id));

Don't use Optional for: collections (return empty list instead), method parameters, fields, or primitive performance-critical code (OptionalInt exists but is rarely worth it).

How does Spring's IoC container work? What is the difference between IoC and DI?
Medium ▼

IoC (Inversion of Control) is the principle: the framework calls your code, not the other way around. You relinquish control of object creation and lifecycle.

DI (Dependency Injection) is one concrete implementation of IoC: dependencies are supplied externally (by the container) rather than created inside the class.

Explain @Transactional — propagation, isolation levels, and common mistakes.
Hard ▼

Propagation — what happens when a @Transactional method calls another:

PropagationBehaviour
REQUIRED (default)Joins existing tx; creates new if none
REQUIRES_NEWSuspends current tx, always starts fresh tx (commits/rolls back independently)
NESTEDSavepoint within current tx — rollback to savepoint only
SUPPORTSRuns in tx if one exists; non-transactional if not
NOT_SUPPORTEDSuspends tx and runs non-transactionally
NEVERThrows if called inside a tx
MANDATORYRequires existing tx; throws if none
Java
@Transactional  // REQUIRED + READ_COMMITTED (default)
public void processOrder(Order order) {
    repo.save(order);
    auditService.log(order); // runs in the SAME tx
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void logAudit(AuditEntry entry) {
    // Commits independently — audit is saved even if outer tx rolls back
    auditRepo.save(entry);
}

// ⚠️ Self-invocation — @Transactional is IGNORED here
public void outer() {
    this.inner(); // proxy is bypassed — no transaction!
}
@Transactional
public void inner() { ... }

// Fix: inject self, or extract inner() to another bean
🚫
Top Spring mistake
Self-invocation skips the AOP proxy. @Transactional and @Cacheable on private methods are also silently ignored because proxies can't intercept them.

What is Spring AOP? What is the difference between @Before, @After, @Around, and @AfterThrowing?
Medium ▼

AOP (Aspect-Oriented Programming) extracts cross-cutting concerns (logging, transactions, security) into reusable Aspects. Spring uses proxy-based AOP — it wraps the target bean in a proxy that intercepts calls.

Java
@Aspect @Component
public class LoggingAspect {

    // Runs before the method (can't prevent execution here)
    @Before("execution(* com.app.service.*.*(..))")
    public void before(JoinPoint jp) {
        log.info("Calling: {}", jp.getSignature());
    }

    // Runs ONLY when method throws — can inspect exception
    @AfterThrowing(pointcut = "execution(* com.app.*.*(..))", throwing = "ex")
    public void onError(Exception ex) { log.error("Error", ex); }

    // Most powerful — wraps method, can modify args/return value/suppress exceptions
    @Around("@annotation(com.app.Timed)")
    public Object time(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        Object result = pjp.proceed(); // call the actual method
        log.info("Took {}ms", System.currentTimeMillis() - start);
        return result;
    }
}

How does JWT-based authentication work in Spring Boot? Walk me through the full flow.
Hard ▼

Flow: Client sends credentials → Server validates → Server issues signed JWT → Client stores token → Client sends token in Authorization: Bearer <token> header → Server validates signature on every request (stateless).

Java
// 1. SecurityFilterChain — disable sessions, add JWT filter
@Configuration @EnableWebSecurity
public class SecurityConfig {
    @Bean
    public SecurityFilterChain chain(HttpSecurity http, JwtFilter jwtFilter)
        throws Exception {
        return http
            .sessionManagement(s -> s.sessionCreationPolicy(STATELESS))
            .csrf(AbstractHttpConfigurer::disable)
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()
                .anyRequest().authenticated())
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class)
            .build();
    }
}

// 2. JWT filter — validates token on every request
@Component @RequiredArgsConstructor
public class JwtFilter extends OncePerRequestFilter {
    private final JwtService jwtService;

    @Override
    protected void doFilterInternal(HttpServletRequest req, ...) {
        String header = req.getHeader("Authorization");
        if (header == null || !header.startsWith("Bearer ")) {
            filterChain.doFilter(req, res); return;
        }
        String token = header.substring(7);
        String username = jwtService.extractUsername(token); // validates signature
        if (username != null && SecurityContextHolder.getContext().getAuthentication() == null) {
            var auth = new UsernamePasswordAuthenticationToken(username, null, roles);
            SecurityContextHolder.getContext().setAuthentication(auth);
        }
        filterChain.doFilter(req, res);
    }
}
⚠️
Security consideration
Store the JWT secret in environment variables or a secrets manager (Vault), never in application.properties. Use RS256 (asymmetric) for microservices so services can verify tokens with the public key without accessing the private key.

What is the N+1 select problem and how do you fix it?
Hard ▼

When fetching N entities that each have a lazy relationship, Hibernate fires 1 SELECT for the parent list plus N additional SELECTs for each child collection — N+1 queries total.

Java
// 🚫 Problem: fetches authors, then N queries for books
List<Author> authors = authorRepo.findAll();
authors.forEach(a -> System.out.println(a.getBooks().size())); // N extra queries!

// ✅ Fix 1: JOIN FETCH in JPQL
@Query("SELECT a FROM Author a LEFT JOIN FETCH a.books")
List<Author> findAllWithBooks();

// ✅ Fix 2: @EntityGraph — avoids modifying query string
@EntityGraph(attributePaths = {"books", "books.tags"})
List<Author> findAll(); // override JpaRepository method

// ✅ Fix 3: @BatchSize — lazy but loads in batches, not 1-by-1
@OneToMany @BatchSize(size = 25)
private List<Book> books;

// ✅ Fix 4: DTO projection with JOIN in query (most efficient)
@Query("SELECT new com.app.AuthorSummaryDto(a.id, a.name, COUNT(b)) "
     + "FROM Author a LEFT JOIN a.books b GROUP BY a.id, a.name")
List<AuthorSummaryDto> findSummaries();
💡
Detect it
Add spring.jpa.show-sql=true or use Hibernate's statistics to spot N+1. In production use p6spy or datasource-proxy to log slow query patterns.
17
Explain the difference between FetchType.LAZY and EAGER. Which should you default to?
Medium ▼

LAZY — loads the related entity only when you access the field. Uses a proxy object. Default for @OneToMany and @ManyToMany.

EAGER — always joins and loads related data with the parent. Default for @ManyToOne and @OneToOne.

🚫
The EAGER trap
Never change @ManyToOne/@OneToOne to EAGER=false and then override to EAGER on queries — this is confusing. Instead, default everything to LAZY and use @EntityGraph or JOIN FETCH at query level to load what you need.

LazyInitializationException — accessing lazy data outside a transaction (after the session is closed). Fix: use @Transactional on the service method, load within the session, or use DTO projections.

What is Hibernate's First-Level and Second-Level Cache? When would you use second-level caching?
Hard ▼

First-level cache — always on, scoped to the EntityManager/Session. Within a single transaction, if you load entity with id=5 twice, Hibernate hits DB only once.

Second-level cache — optional, scoped to the SessionFactory (shared across transactions). Stores entity state in memory (Caffeine, Ehcache, Redis). Good for reference data that is read often and rarely changes.

What is the Circuit Breaker pattern? How do you implement it with Resilience4j?
Hard ▼

Circuit Breaker prevents cascading failures: if a downstream service is failing repeatedly, the breaker opens and fast-fails requests for a cooldown period, then half-opens to test recovery.

States: Closed (normal) → Open (failing, return fallback) → Half-Open (probe) → Closed (if probes succeed).

Java
# application.yml

    instances:
      paymentService:
        slidingWindowSize: 10
        failureRateThreshold: 50      # open if 50%+ fail in last 10 calls
        waitDurationInOpenState: 10s  # wait before half-open
        permittedNumberOfCallsInHalfOpenState: 3

// Java usage
@CircuitBreaker(name = "paymentService", fallbackMethod = "paymentFallback")
@Retry(name = "paymentService")
public PaymentResult charge(ChargeRequest req) {
    return paymentClient.charge(req);
}

public PaymentResult paymentFallback(ChargeRequest req, Exception ex) {
    log.error("Payment service unavailable", ex);
    return PaymentResult.pending("QUEUED_FOR_RETRY");
}

Explain Java's memory model: stack vs heap, and what causes a memory leak?
Medium▼

Stack — per-thread, stores local variables and method call frames. Automatically cleaned up when method returns. Stack overflow = infinite recursion.

Heap — shared, stores all objects. GC reclaims unreachable objects. OOM = objects reachable but never used again.

Memory leaks in Java (object is reachable but logically unused):

  • Static collections that keep growing: static Map<K,V> that is never cleared
  • Listeners/callbacks registered but never de-registered
  • ThreadLocal variables not removed after request completes (thread pool reuse)
  • Long-living cache without eviction policy
  • Inner class holding implicit reference to outer class

What is CQRS and Event Sourcing? When would you use them?
Hard▼

Mockito: @Mock vs @Spy vs @InjectMocks — what's the difference?
Medium ▼

@Mock creates a complete stub — all methods return defaults (null, 0, false, empty list). You define behaviour with when(...).thenReturn(...). No real code runs.

@Spy wraps a real instance — real code runs by default, but you can selectively stub methods with doReturn(...).when(spy).method(). Use for partial mocking of legacy code.

@InjectMocks creates the class under test and injects @Mock/@Spy fields into it via constructor, setter, or field injection.

Java — Mockito annotations
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    OrderRepository orderRepo; // pure stub — no DB

    @Mock
    EmailService emailService; // stub — no emails sent

    @Spy
    PriceCalculator calculator = new PriceCalculator(); // real logic runs

    @InjectMocks
    OrderService orderService; // gets mocks injected automatically

    @Test
    void placeOrder_sendsConfirmationEmail() {
        // Arrange
        Order order = new Order("item-1", 2);
        when(orderRepo.save(any())).thenReturn(order);

        // Act
        orderService.placeOrder(order);

        // Assert — verify interaction
        verify(emailService).sendConfirmation(order);
    }

    @Test
    void spy_partialMock() {
        // Spy: stub only the expensive part, real logic for the rest
        doReturn(100.0).when(calculator).fetchTaxRate();
        BigDecimal total = calculator.calculateTotal(500.0); // real method runs
        assertThat(total).isEqualByComparingTo("600.0");
    }
}
⚠️
@Spy pitfall
With @Spy, use doReturn(...).when(spy) NOT when(spy.method()).thenReturn(...) — the second form calls the real method during stubbing setup, which can cause NullPointerExceptions or side effects.

Testing Spring Security — @WithMockUser and SecurityMockMvcRequestPostProcessors
Medium ▼

Testing secured endpoints requires authenticating in the test context. Spring Security Test provides two main approaches:

@WithMockUser — class/method-level annotation that injects a mock principal into the SecurityContext before the test runs. Simple and readable for most cases.

SecurityMockMvcRequestPostProcessors — fluent API applied per request. More flexible — can set JWT claims, OAuth2 tokens, or OIDC details.

Java — Spring Security test approaches
@WebMvcTest(AdminController.class)
class AdminControllerTest {
    @Autowired MockMvc mvc;

    // Approach 1: @WithMockUser — sets username + roles in SecurityContext
    @Test
    @WithMockUser(roles = "ADMIN")
    void adminEndpoint_allowsAdmin() throws Exception {
        mvc.perform(get("/admin/dashboard"))
           .andExpect(status().isOk());
    }

    @Test
    void adminEndpoint_deniesAnonymous() throws Exception {
        mvc.perform(get("/admin/dashboard"))
           .andExpect(status().isUnauthorized()); // 401
    }

    // Approach 2: RequestPostProcessors — per-request, more control
    @Test
    void withJwt() throws Exception {
        mvc.perform(get("/api/me")
               .with(jwt().jwt(j -> j
                   .claim("sub", "user-123")
                   .claim("roles", List.of("ROLE_USER")))))
           .andExpect(status().isOk());

    // Testing 403 (authenticated but wrong role)
    @Test
    @WithMockUser(roles = "USER")
    void adminEndpoint_forbidsUser() throws Exception {
        mvc.perform(get("/admin/dashboard"))
           .andExpect(status().isForbidden()); // 403
    }

Medium ▼

TDD (Test-Driven Development) means writing the test before the implementation. The cycle has three phases:

  • Red — write a failing test that captures the desired behaviour
  • Green — write the minimum code to make it pass
  • Refactor — clean up without breaking the test
Java — TDD example: UserRegistrationService
// STEP 1 — RED: write the test first. It won't even compile yet.
class UserRegistrationServiceTest {
    @Mock UserRepository userRepo;
    @Mock EmailService emailService;
    @InjectMocks UserRegistrationService service;

    @Test
    void register_withDuplicateEmail_throwsException() {

        assertThatThrownBy(() -> service.register("alice@example.com", "pass"))
            .isInstanceOf(DuplicateEmailException.class)
            .hasMessage("Email already registered");
    }

    @Test
    void register_withNewEmail_savesUserAndSendsWelcomeEmail() {
        given(userRepo.existsByEmail(any())).willReturn(false);
        User saved = new User("alice@example.com");
        given(userRepo.save(any())).willReturn(saved);

        service.register("alice@example.com", "pass");

        verify(userRepo).save(any(User.class));
        verify(emailService).sendWelcome("alice@example.com");
    }
}

// STEP 2 — GREEN: minimal implementation to make tests pass
@Service
public class UserRegistrationService {
    private final UserRepository userRepo;
    private final EmailService emailService;

    public void register(String email, String password) {
        if (userRepo.existsByEmail(email)) {
            throw new DuplicateEmailException("Email already registered");
        }
        User user = new User(email, passwordEncoder.encode(password));
        userRepo.save(user);
        emailService.sendWelcome(email);
    }
}

// STEP 3 — REFACTOR: extract constants, improve naming, no test changes needed

TDD forces you to design the API from the consumer's perspective before implementation. If the test is hard to write, the API is probably too coupled — it's cheaper to fix that during TDD than after.

What is a "test slice"? What beans load in @WebMvcTest?
Easy ▼

A test slice is a Spring Boot annotation that loads only the subset of beans needed for a specific layer, rather than the full application context. This dramatically reduces test startup time and isolates failures to the correct layer.

What @WebMvcTest loads:

  • Controllers (@Controller, @RestController, @ControllerAdvice)
  • Filters and HandlerInterceptors
  • Jackson ObjectMapper and JSON converters
  • Spring Security (if on classpath) — full filter chain is active
  • MockMvc auto-configured

What @WebMvcTest does NOT load:

  • Services, Repositories, JPA entities — you must @MockBean them
  • Component scan excludes @Service, @Repository, @Component

Testing asynchronous code — CompletableFuture, @Async, and Awaitility
Hard ▼

Async code is tricky to test because the assertion might run before the async operation completes. Three strategies exist:

1. CompletableFuture.get() — block the test thread until the future completes. Simple but can hang forever if the future never completes.

2. Awaitility — polls a condition with a timeout. Works for any async trigger including @Async, Kafka listeners, event handlers.

3. SyncTaskExecutor — configure a synchronous executor in the test profile so @Async methods execute in the same thread (no real async).

Java — async test strategies
// Strategy 1: CompletableFuture.get() — blocks until done
@Test
void asyncFetch_returnsResult() throws Exception {
    CompletableFuture<String> future = asyncService.fetchData();
    String result = future.get(5, TimeUnit.SECONDS); // timeout guard
    assertThat(result).isEqualTo("expected");
}

// Strategy 2: Awaitility — poll until condition or timeout
// testImplementation 'org.awaitility:awaitility:4.2.0'
@Test
void asyncKafkaConsumer_processesMessage() {
    kafkaTemplate.send("orders", "order-1");

    Awaitility.await()
        .atMost(10, TimeUnit.SECONDS)
        .pollInterval(500, TimeUnit.MILLISECONDS)
        .untilAsserted(() -> {
            assertThat(orderRepository.findByOrderId("order-1")).isPresent();
        });
}

// Strategy 3: SyncTaskExecutor in test config — disables actual async
@TestConfiguration
class SyncExecutorConfig {
    @Bean
    @Primary
    public TaskExecutor taskExecutor() {
        return new SyncTaskExecutor(); // @Async runs synchronously in tests
    }
}
ℹ️
Awaitility is the gold standard
Awaitility's untilAsserted() retries the assertion block on every poll interval, which handles flakiness naturally. Unlike Thread.sleep(), it passes as soon as the condition is true — no unnecessary waiting.

JaCoCo test coverage — what to measure, what NOT to measure, common pitfalls
Medium ▼

JaCoCo (Java Code Coverage) measures which lines/branches of code were executed during tests. High coverage is a hygiene signal, not a quality guarantee — you can reach 100% line coverage with tests that make no assertions.

What to measure:

  • Business logic — service methods, domain rules, validation
  • Branch coverage — ensure both true/false paths of conditions are tested
  • Error handling paths — exception cases, null checks

What NOT to measure (exclude):

  • Generated code — Lombok-generated getters/setters, MapStruct mappers
  • DTOs and POJOs with no logic
  • Configuration classes (@Configuration, @Bean methods)
  • Main application class (*Application.java)

Coverage theater
Chasing 100% coverage leads to testing getters/setters and constructors that have no logic. A well-tested service class at 70% coverage with meaningful assertions is more valuable than 95% coverage with trivial tests. Focus on mutation testing (PIT) for true quality signals.

REST best practices — Richardson Maturity Model, idempotency, statelessness
Medium ▼

The Richardson Maturity Model (RMM) defines four levels of REST compliance:

  • Level 0 — Single URI, POST everything (SOAP-like). POST /api?action=getUser
  • Level 1 — Multiple resources (URIs) but still only POST. POST /users/1
  • Level 2 — HTTP verbs used correctly (GET, POST, PUT, DELETE, PATCH) + status codes. This is what most "REST APIs" actually are.
  • Level 3 (HATEOAS) — Responses include hypermedia links to related actions. Rarely used in practice.
HTTP — Level 2 REST conventions

POST   /users          → create user (201 Created, Location header)

PATCH  /users/42       → partial update (200 or 204)
DELETE /users/42       → delete (204 No Content)

// Idempotency: same request → same result (safe to retry)
// Idempotent: GET, PUT, DELETE, HEAD, OPTIONS
// NOT idempotent: POST (creates new resource each time)
// PATCH is NOT guaranteed idempotent (depends on operation)

// Statelessness: server stores no session; each request is self-contained
// Auth via JWT in Authorization header, not server-side session cookie
Authorization: Bearer eyJhbGciOiJSUzI1NiJ9...

Common REST mistakes
Using verbs in URIs (/getUser, /deleteOrder), returning 200 for errors with error body, using POST for idempotent operations without idempotency keys, ignoring 409 Conflict for duplicate resources, and not returning the Location header after 201 Created.

GraphQL N+1 problem — what is it and how does DataLoader solve it?
Hard ▼

The N+1 problem is the most common GraphQL performance issue. When you query a list of N items and each item's resolver makes a separate DB call for a related entity, you end up with 1 + N database queries instead of 1.

Example: query 100 users, each user's department field triggers a separate SQL call → 101 queries.

DataLoader solves this by batching and caching — it collects all the IDs requested within one event loop tick, then makes a single batched query.

Java — DGS (Netflix GraphQL) DataLoader example
// Without DataLoader: N+1 problem
@DgsComponent
class UserDataFetcher {
    @DgsData(parentType = "User", field = "department")
    public Department getDepartment(DgsDataFetchingEnvironment dfe) {
        User user = dfe.getSource();
        return deptRepo.findById(user.getDeptId()); // 1 query per user = N+1!
    }
}

// With DataLoader: single batched query
@DgsDataLoader(name = "departments")
class DepartmentDataLoader implements BatchLoader<Long, Department> {
    @Override
    public CompletionStage<List<Department>> load(List<Long> deptIds) {
        // Called ONCE with ALL dept IDs from the current query
        return CompletableFuture.supplyAsync(
            () -> deptRepo.findAllById(deptIds) // 1 query total
        );
    }
}

@DgsData(parentType = "User", field = "department")
public CompletableFuture<Department> getDepartment(DgsDataFetchingEnvironment dfe) {
    DataLoader<Long, Department> loader = dfe.getDataLoader("departments");
    User user = dfe.getSource();
    return loader.load(user.getDeptId()); // batched — returns CompletableFuture
}
✅
DataLoader caching
DataLoader also caches within a request — if department 5 is requested by multiple users in the same query, it's only fetched once. The cache is per-request, not global, so it won't serve stale data.

gRPC in Java — protobuf definition, streaming types, when to prefer over REST
Hard ▼

gRPC uses HTTP/2 + Protocol Buffers (protobuf) — a binary serialisation format. It's 5-10x faster than REST+JSON for high-throughput internal communication.

4 streaming types:

  • Unary — one request, one response (like REST)
  • Server streaming — one request, stream of responses (e.g. log tailing)
  • Client streaming — stream of requests, one response (e.g. file upload)
  • Bidirectional streaming — both sides stream (e.g. chat)
Protobuf + Java — gRPC service definition
// order.proto — generates Java stubs via protoc plugin
syntax = "proto3";
package com.example.orders;

service OrderService {
    rpc GetOrder (OrderRequest) returns (OrderResponse);              // unary
    rpc StreamOrders (OrderFilter) returns (stream OrderResponse);   // server stream
    rpc CreateOrders (stream CreateOrderRequest) returns (OrderSummary); // client stream
}

message OrderRequest  { string order_id = 1; }
message OrderResponse { string order_id = 1; double amount = 2; string status = 3; }

// Java implementation (Spring Boot + grpc-spring-boot-starter)
@GrpcService
class OrderGrpcService extends OrderServiceGrpc.OrderServiceImplBase {

    @Override
    public void getOrder(OrderRequest req, StreamObserver<OrderResponse> observer) {
        Order order = orderRepo.findById(req.getOrderId())
            .orElseThrow(() -> new StatusRuntimeException(Status.NOT_FOUND));
        observer.onNext(OrderResponse.newBuilder()
            .setOrderId(order.getId())
            .setAmount(order.getAmount())
            .setStatus(order.getStatus())
            .build());
        observer.onCompleted();
    }
}
ℹ️
When to choose gRPC over REST
Choose gRPC for internal microservice communication where: (1) latency matters (trading systems, real-time analytics), (2) you need streaming (live data feeds), (3) strong typing and backward-compatible schema evolution via protobuf field numbers is valuable. Avoid for public APIs — browser support requires a grpc-web proxy.

JWT structure and validation in Spring Security — header.payload.signature, claims, exp
Medium ▼

A JWT (JSON Web Token) is a self-contained credential — the server doesn't need a session store because the token carries the claims itself.

Structure: three Base64URL-encoded parts separated by dots: header.payload.signature

  • Header — algorithm (alg: RS256) and type (typ: JWT)
  • Payload — claims: sub (subject/user ID), exp (expiry timestamp), iat (issued at), roles (custom)
  • Signature — HMAC or RSA signature over header+payload. Prevents tampering.
Java — Spring Security JWT validation
// Spring Security 6 — JWT resource server configuration
@Configuration
@EnableWebSecurity
class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(AbstractHttpConfigurer::disable)
            .sessionManagement(s -> s.sessionCreationPolicy(STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/auth/**").permitAll()
                .anyRequest().authenticated())
            .oauth2ResourceServer(oauth2 -> oauth2
                .jwt(jwt -> jwt.jwtAuthenticationConverter(jwtConverter())))
            .build();
    }

    @Bean
    public JwtDecoder jwtDecoder() {
        // Validates signature, expiry (exp), not-before (nbf) automatically
        return NimbusJwtDecoder
            .withPublicKey(rsaPublicKey()) // or .withJwkSetUri() for OIDC
            .build();
    }

    private JwtAuthenticationConverter jwtConverter() {
        var converter = new JwtAuthenticationConverter();
        converter.setJwtGrantedAuthoritiesConverter(jwt -> {
            // Extract custom "roles" claim → GrantedAuthority list
            List<String> roles = jwt.getClaimAsStringList("roles");
            return roles.stream()
                .map(r -> (GrantedAuthority) () -> "ROLE_" + r)
                .collect(toList());
        });
        return converter;
    }
}
⚠️
JWT security pitfalls
(1) Never use alg=none — reject tokens with no algorithm. (2) Short expiry (15min) + refresh token pattern reduces exposure window. (3) Sensitive data (passwords, PII) must NOT go in the payload — it's Base64-encoded, not encrypted. (4) JWTs cannot be invalidated before expiry — use a token blocklist or short expiry for logout.

OAuth2 flows — Authorization Code (PKCE), Client Credentials, when to use which
Hard ▼

OAuth2 is an authorization framework, not an authentication protocol. It defines several flows (grants) for different scenarios.

Authorization Code + PKCE — the secure flow for user-facing applications (web apps, SPAs, mobile). The user logs in at the identity provider; the app receives an authorization code and exchanges it for tokens. PKCE prevents authorization code interception attacks.

Client Credentials — machine-to-machine. No user involved. A service authenticates with its own client ID + secret to get an access token. Used for microservice-to-microservice calls.

Java — Spring Security OAuth2 client config
# application.yml — Client Credentials flow (M2M)
spring:
  security:
    oauth2:
      client:
        registration:
          my-service:
            authorization-grant-type: client_credentials
            client-id: ${CLIENT_ID}
            client-secret: ${CLIENT_SECRET}
            scope: api.read
        provider:
          my-service:
            token-uri: https://auth.example.com/oauth/token

@Bean
public WebClient webClient(OAuth2AuthorizedClientManager manager) {
    var interceptor = new ServletOAuth2AuthorizedClientExchangeFilterFunction(manager);
    interceptor.setDefaultClientRegistrationId("my-service");
    return WebClient.builder()
        .baseUrl("https://api.partner.com")
        .apply(interceptor.oauth2Configuration())
        .build();
    // Spring auto-fetches & refreshes the token transparently
}

API versioning strategies — URI path vs Accept header vs query param, pros/cons
Medium ▼

API versioning lets you evolve the API without breaking existing clients. Three main strategies:

StrategyExampleProsCons
URI Path/api/v2/usersSimple, cacheable, visible in logsNot pure REST (URI should identify resource, not version)
Accept HeaderAccept: application/vnd.api.v2+jsonPurest REST, same URIHard to test in browser, not cached by default
Query Param/users?version=2Easy to test, optionalBreaks caching, pollutes query string
Java — URI path versioning in Spring
// Most common in practice — URI path versioning
@RestController
@RequestMapping("/api/v1/users")
class UserControllerV1 { // legacy contract preserved
    @GetMapping("/{id}")
    public UserResponseV1 getUser(@PathVariable Long id) { ... }
}

@RestController
@RequestMapping("/api/v2/users")
class UserControllerV2 { // new contract
    @GetMapping("/{id}")
    public UserResponseV2 getUser(@PathVariable Long id) { ... }
}

// Sunset old versions gracefully — add deprecation header
response.addHeader("Deprecation", "Sat, 01 Jan 2025 00:00:00 GMT");
response.addHeader("Sunset", "Sat, 01 Jul 2025 00:00:00 GMT");
response.addHeader("Link", "</api/v2/users>; rel=\"successor-version\"");
ℹ️
Industry preference
URI path versioning (/api/v2/) is the industry default — used by Stripe, GitHub, Twilio. The REST-purity argument for header versioning rarely wins in practice because discoverability and simplicity matter more. Header versioning is used by GitHub's API as an alternative format but most teams choose URI.
