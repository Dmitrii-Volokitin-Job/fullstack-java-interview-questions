Decomposition & Domain-Driven Design

How to split a system into services correctly — and the traps when you get it wrong

1
How do you decide where to draw service boundaries? What is a Bounded Context?
Hard▼

A Bounded Context (DDD) is a linguistic boundary: within it, a term like "customer" has one precise, agreed-upon meaning. Across contexts, "customer" in the billing service vs the shipping service may be different data models with different responsibilities. Service boundaries should match bounded context boundaries.

Rules for good boundaries:

  • High cohesion within — a service owns all data and logic for one business capability
  • Low coupling across — changing one service doesn't force changes in others
  • Independent deployability — you can deploy service A without deploying service B
  • Team ownership — Conway's Law: your service boundaries will mirror your org chart. Design the org and services together
Text — Decomposition example: e-commerce
Monolith "order"  →  microservices:
  OrderService     — create order, track status, order history
  PaymentService   — charge, refund, reconciliation (payment is its OWN domain)
  InventoryService — stock levels, reservations, warehouse routing
  NotificationService — email/SMS/push, templates, retry
  ShippingService  — carriers, tracking numbers, dispatch

What's wrong with splitting too small:
  "AddressService" — addresses are data, not a domain. Put in CustomerService.
  "TaxService" — tax calculation is logic inside OrderService unless regulations
                 are complex enough to warrant a team dedicated to it.
💡
DDD heuristic
Ask: "would a domain expert in department A understand the concepts in department B without translation?" If yes, they're in the same context. If they need a translator, they're in different contexts — and should be different services.
2
What are the signs you've split your microservices too finely (nano-services)?
Medium▼

Nano-service symptoms:

  • Every feature change requires changes in 3+ services simultaneously
  • Services are chatty — a single user request triggers a cascade of 10+ synchronous HTTP calls
  • No single team owns a coherent capability; ownership is blurry
  • Data consistency problems are everywhere because entities are split across services
  • Deployments are always coordinated ("we must deploy A, B, and C together")
  • Network overhead dominates CPU/IO overhead

Rule of thumb: a service should be as small as possible but no smaller than a single team's ownership boundary. If two services are always deployed together, they should be one service.

Merger strategy: if two services are co-deployed more than 50% of the time, merge them. The operational complexity of a separate service must be justified by its independent change rate.

3
Explain the API Gateway pattern. What does it do beyond simple routing?
Medium▼

An API Gateway is the single entry point for all external clients. Beyond routing, it handles cross-cutting concerns that shouldn't live in every service:

  • Authentication — validates JWT/OAuth token centrally; services receive a verified identity claim
  • Rate limiting — token bucket per client/IP before requests hit services
  • SSL termination — TLS at the edge; internal traffic can use plaintext (or mTLS at service mesh layer)
  • Request aggregation (BFF pattern) — one gateway call fans out to 3 services and aggregates the response, reducing mobile client round-trips
  • Protocol translation — external REST → internal gRPC; external HTTP → internal Kafka
  • Logging/tracing injection — adds correlation IDs to every request
YAML — Spring Cloud Gateway route with rate limiter
spring:
  cloud:
    gateway:
      routes:
        - id: order-service
          uri: lb://order-service          # Kubernetes service DNS via LoadBalancer
          predicates:
            - Path=/api/orders/**
          filters:
            - StripPrefix=1
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 100
                redis-rate-limiter.burstCapacity: 200
                key-resolver: "#{@userKeyResolver}"
            - name: CircuitBreaker
              args:
                name: orderCB
                fallbackUri: forward:/fallback/orders
💡
BFF pattern
Backend For Frontend — instead of one generic gateway, you run a separate gateway per client type: mobile BFF aggregates data differently than web BFF or partner API BFF. Avoids the "Swiss army knife" problem where one gateway tries to serve all clients and ends up serving none well.
4
What is a service mesh and when do you actually need one?
Hard▼

A service mesh (Istio, Linkerd) is a dedicated infrastructure layer for service-to-service communication. It deploys a sidecar proxy (Envoy) alongside every service pod that intercepts all inbound/outbound traffic.

What the mesh gives you:

  • mTLS between every service pair (zero-trust networking) — without any code changes
  • Circuit breaking, retry policies, timeouts — configured as CRDs, not code
  • Distributed tracing — automatic span propagation
  • Traffic splitting for canary deploys at L7
  • Observability: request rates, error rates, latency per service-pair

When you actually need it:

  • ≥10 services with complex inter-service traffic patterns
  • Compliance requiring mTLS between services
  • You want canary/traffic splitting without changing application code

When you don't need it:

  • ≤5 services — operational overhead (Istio's learning curve is steep) exceeds benefit
  • Services already use Resilience4j for resilience and Micrometer for observability
  • Team is already stretched — a mesh adds a whole new operational domain

Inter-Service Communication

Synchronous vs async, gRPC, Kafka, service discovery, and the choreography/orchestration trade-off

5
When do you choose synchronous REST/HTTP over asynchronous Kafka messaging between services?
Hard▼
CriterionUse REST/gRPC (sync)Use Kafka (async)
Result needed immediately✅ Validate payment now❌ Can't block caller
Fan-out (1→N consumers)❌ Call each individually✅ Publish once, N consume
Consumer can be slow/offline❌ You'll time out✅ Messages buffer in topic
Simple query (read data)✅ GET /user/42❌ Overkill
Guaranteed processing order❌ No queue, no order✅ Within a partition
Tight coupling acceptable✅ Internal service, same team❌ Prefer loose coupling
💡
Golden rule
Async is the default for cross-domain communication (different bounded contexts). Sync is acceptable within a domain or when the caller genuinely cannot proceed without the response (e.g., real-time fraud check on payment submission).
6
Explain Saga pattern: choreography vs orchestration. When do you choose each?
Hard▼

A Saga is a sequence of local transactions coordinated across services — with compensating transactions for rollback. It replaces distributed 2PC (which doesn't scale).

Choreography Saga:

  • Services communicate via events — no central coordinator
  • Each service listens for an event, does its work, publishes the next event
  • ✅ No single point of failure; services are more loosely coupled
  • ❌ Hard to visualize end-to-end flow; debugging "where did this saga get stuck" is painful; compensating transactions are scattered across services
Text — Choreography flow: order processing
OrderService publishes → OrderCreated
  PaymentService consumes → charges card → publishes PaymentCompleted | PaymentFailed
    InventoryService consumes PaymentCompleted → reserves stock → publishes InventoryReserved
      ShippingService consumes InventoryReserved → schedules dispatch

On failure: PaymentFailed → OrderService consumes and publishes OrderCancelled
            InventoryService consumes OrderCancelled → releases reservation

Orchestration Saga:

  • A central Saga Orchestrator drives the flow — calls each service and reacts to results
  • ✅ Entire flow visible in one place; easy to add new steps; compensation logic is centralized
  • ❌ Orchestrator becomes a bottleneck; more coupling; the orchestrator is smart (violates "dumb pipes, smart endpoints")
💡
Choose by complexity
≤4 steps with simple compensation → choreography. ≥5 steps, complex conditional logic, or compliance audit trail required → orchestration (use Camunda/Temporal as the orchestrator). Never mix — pick one style per saga.
7
What is gRPC and when does it beat REST for internal microservice calls?
Medium▼

gRPC is a high-performance RPC framework using HTTP/2 + Protocol Buffers (binary serialization). It was designed for internal service-to-service calls where performance matters.

gRPC advantages over REST:

  • Protobuf binary encoding is 3-10x smaller than JSON — less bandwidth, faster serialization
  • HTTP/2 multiplexing — multiple calls on one connection; no head-of-line blocking
  • Bidirectional streaming — server can push data to client continuously
  • Strongly typed contracts — .proto files generate client/server stubs in any language; no swagger drift
  • Built-in deadline propagation — client sets a deadline; it flows through all service hops
Protobuf — UserService.proto
syntax = "proto3";

service UserService {
  rpc GetUser (GetUserRequest) returns (UserResponse);
  rpc StreamActivityFeed (UserRequest) returns (stream ActivityEvent); // server streaming
}

message GetUserRequest { int64 user_id = 1; }
message UserResponse {
  int64  id         = 1;
  string email      = 2;
  string name       = 3;
  bool   active     = 4;
}
// Generated: UserServiceGrpc.UserServiceBlockingStub (Java), userServiceClient (Go/Python/etc.)

When to use REST instead: public APIs (browsers don't support gRPC natively without grpc-web), team unfamiliar with proto toolchain, simple CRUD with no performance concern.

8
How does service discovery work in Kubernetes vs with Eureka/Consul?
Medium▼

Kubernetes DNS service discovery (built-in):

  • Every Service gets a DNS name: order-service.default.svc.cluster.local
  • kube-dns / CoreDNS resolves this to the ClusterIP
  • kube-proxy handles load balancing at the IP level via iptables rules
  • No sidecar or library needed — just call http://order-service:8080

Eureka (Spring Cloud Netflix):

  • Services register themselves on startup with their IP + port + health check URL
  • Clients fetch the registry and do client-side load balancing (Spring Cloud LoadBalancer)
  • Required before Kubernetes was standard; adds operational overhead
💡
Modern answer
If you're on Kubernetes, use Kubernetes DNS — it's native and zero-config. Eureka is a Spring-ecosystem holdover from the pre-K8s era. Consul is still valuable in hybrid environments (VMs + containers) where K8s DNS doesn't reach all nodes. Remove Eureka from K8s-native stacks; it duplicates functionality the platform already provides.

Data Patterns

Database-per-service, CQRS, Event Sourcing, and the distributed data consistency problem

9
Why database-per-service? What do you lose and how do you compensate?
Hard▼

A shared database is the death of microservice independence. If two services share a DB schema, any schema change requires coordinating both services — you've lost independent deployability.

What you gain:

  • Services can be deployed, scaled, and replaced independently
  • Each service can choose the right DB technology (PostgreSQL for orders, Redis for sessions, Elasticsearch for search)
  • Schema changes in service A don't affect service B

What you lose:

  • JOIN queries across services → use API composition (call both services, join in code) or CQRS read models
  • ACID transactions across services → use Saga pattern with compensating transactions
  • Cross-service reporting → build a reporting/analytics service that consumes events from all services and maintains a denormalised read store (Kafka → data warehouse or Elasticsearch)
🚫
Anti-pattern to name
"Integration database" — multiple services write to the same DB. Teams start adding direct cross-service JOIN queries, then adding schema constraints that span services. Within 6 months you have a distributed monolith: all the operational complexity of microservices with none of the independence benefits.
10
Explain CQRS. When does it help and when is it over-engineering?
Hard▼

CQRS (Command Query Responsibility Segregation) separates the write model (Commands) from the read model (Queries) — usually implemented as separate data stores.

Java — CQRS with Spring + Kafka
// WRITE SIDE: normalised PostgreSQL, optimised for writes and consistency
@Service
public class OrderCommandService {
    public Order createOrder(CreateOrderCmd cmd) {
        Order order = orderRepository.save(new Order(cmd));
        eventPublisher.publish(new OrderCreatedEvent(order)); // → Kafka
        return order;
    }
}

// READ SIDE: denormalised Elasticsearch document, optimised for complex queries
@KafkaListener(topics = "order-events")
public void on(OrderCreatedEvent e) {
    // Build search document with customer name, product names, etc. — data from multiple services
    orderSearchIndex.index(OrderDocument.from(e, customerService.get(e.customerId())));
}

// Read model queries Elasticsearch, not PostgreSQL → no JOINs, no N+1, O(1) search
@GetMapping("/orders/search")
public List<OrderSummary> search(@RequestParam String q) {
    return orderSearchIndex.search(q);
}

When CQRS helps: complex query requirements that can't be served by the write model efficiently; high read/write ratio with different scaling needs; cross-service read views.

When it's over-engineering: simple CRUD app; read queries are cheap on the write DB; team is small and the eventual consistency complexity will cause bugs.

11
What is Event Sourcing and when does it provide real value?
Hard▼

Event Sourcing stores the state of an entity as a sequence of events, not as the current state. To get current state, replay all events.

Java — Event Sourcing concept
// Traditional: single row updated in place — history lost
UPDATE orders SET status = 'SHIPPED', updated_at = NOW() WHERE id = 42;

// Event Sourced: immutable append-only event log
events table:
  (42, OrderCreated,  {customerId:7, items:[...], total:99.99}, 2024-01-10)
  (42, PaymentTaken,  {amount:99.99, transactionId:'tx_abc'},   2024-01-10)
  (42, ItemPicked,    {warehouseId:3, pickedAt:'2024-01-11'},   2024-01-11)
  (42, Shipped,       {carrier:'DHL', trackingNo:'1234567'},    2024-01-12)

// Current state = replay all events → Order.apply(events)

Real value in:

  • Audit trail is a hard requirement (banking, healthcare, legal)
  • Temporal queries — "what was the account balance on 2023-06-15?"
  • Debugging — you can replay events to reproduce any production state
  • Event-driven integrations — the event log IS the Kafka topic

Costs: querying current state requires replaying (use snapshots for large aggregates); eventual consistency in projections; schema evolution for old events is hard.

12
How does the Transactional Outbox pattern solve the dual-write problem?
Hard▼

Dual-write problem: you can't atomically write to your database AND publish to Kafka. If the DB write succeeds but Kafka publish fails, your event is lost — the rest of the system never knows the order was created.

Java + SQL — Outbox pattern implementation
// Step 1: Write order AND outbox event in the SAME database transaction
@Transactional
public Order createOrder(CreateOrderRequest req) {
    Order order = orderRepo.save(new Order(req));

    outboxRepo.save(OutboxEvent.builder()
        .aggregateType("Order")
        .aggregateId(order.getId().toString())
        .eventType("OrderCreated")
        .payload(objectMapper.writeValueAsString(order))
        .build());
    return order; // both committed or both rolled back
}

// Step 2: Outbox poller — separate thread/job reads and publishes
@Scheduled(fixedDelay = 500)
public void pollOutbox() {
    List<OutboxEvent> events = outboxRepo.findUnpublished();
    events.forEach(e -> {
        kafkaTemplate.send(e.getEventType(), e.getAggregateId(), e.getPayload());
        e.markPublished();
    });
    outboxRepo.saveAll(events);
}
// Or use Debezium CDC — no polling needed, reads PostgreSQL WAL directly

Debezium is the preferred approach — it reads the DB WAL (write-ahead log) in real time. Zero polling delay, no missed events, works without modifying application code.

Resilience Patterns

Circuit Breaker, Bulkhead, Retry, Timeout — and how to combine them correctly

13
Explain the Circuit Breaker pattern. What are its states and how does it prevent cascade failures?
Hard▼

A Circuit Breaker wraps a remote call. When the downstream service starts failing, the CB "trips open" and immediately returns an error (or fallback) instead of calling the failing service — preventing thread pool exhaustion and cascade failures.

Three states:

  • CLOSED — normal operation. Calls pass through; failures counted
  • OPEN — failure threshold exceeded. All calls fail immediately with CallNotPermittedException; no calls reach downstream
  • HALF_OPEN — after a wait duration, allows a limited number of probe calls. If they succeed → CLOSED. If they fail → OPEN again
Java — Resilience4j Circuit Breaker
@CircuitBreaker(name = "inventoryService", fallbackMethod = "inventoryFallback")
@TimeLimiter(name = "inventoryService")
@Retry(name = "inventoryService")
public CompletableFuture<StockLevel> getStock(String productId) {
    return inventoryClient.getStock(productId);
}

private CompletableFuture<StockLevel> inventoryFallback(String productId, Exception ex) {
    log.warn("Inventory service unavailable, returning cached/default for {}", productId);
    return CompletableFuture.completedFuture(stockCache.get(productId)
        .orElse(StockLevel.unknown()));
}
YAML — Resilience4j config (application.yml)
resilience4j:
  circuitbreaker:
    instances:
      inventoryService:
        failureRateThreshold: 50          # trip at 50% failure rate
        slowCallRateThreshold: 80         # also trip on 80% slow calls
        slowCallDurationThreshold: 2000ms
        waitDurationInOpenState: 30s
        permittedNumberOfCallsInHalfOpenState: 5
        slidingWindowSize: 20
  timelimiter:
    instances:
      inventoryService:
        timeoutDuration: 1s
💡
Order matters
Apply decorators in order: @CircuitBreaker wraps @Retry wraps @TimeLimiter. The CB sees the result of all retries, not individual attempts. Don't retry on CallNotPermittedException — the circuit is open for a reason.
14
What is the Bulkhead pattern and how does it isolate failures?
Medium▼

Named after ship bulkheads that prevent one flooded compartment from sinking the whole ship. In software: each downstream service gets its own thread pool or semaphore, so a slow/unavailable service can't exhaust the shared thread pool and block all other operations.

Java — Resilience4j Bulkhead (thread pool isolation)
resilience4j:
  thread-pool-bulkhead:
    instances:
      paymentService:
        maxThreadPoolSize: 10       # at most 10 threads for payment calls
        coreThreadPoolSize: 5
        queueCapacity: 20           # buffer 20 more requests
      inventoryService:
        maxThreadPoolSize: 5        # separate pool — payment slowness can't starve inventory
        coreThreadPoolSize: 3
        queueCapacity: 10

Without Bulkhead: payment service hangs → 200 threads blocked waiting → no threads left for inventory or order calls → entire application unresponsive, even for services that are healthy.

With Bulkhead: payment thread pool fills → payment calls queue/reject → inventory threads still free → order service still works. Failure blast radius is contained.

15
What's the difference between retry with exponential backoff and the circuit breaker? When do you use both?
Medium▼
  • Retry — for transient failures (brief network blip, single 503). Retrying too aggressively worsens a struggling service
  • Exponential backoff + jitter — spreads retry load; prevents thundering herd (all services retrying simultaneously)
  • Circuit Breaker — for persistent failures. Once failure rate threshold is reached, stops retrying altogether. Gives the failing service recovery time
YAML — Retry with backoff (Resilience4j)
resilience4j:
  retry:
    instances:
      paymentService:
        maxAttempts: 3
        waitDuration: 500ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2.0   # 500ms → 1000ms → 2000ms
        randomizedWaitFactor: 0.5           # jitter: ±50% to spread retries
        retryExceptions:
          - java.net.SocketTimeoutException
          - org.springframework.web.client.ResourceAccessException
        ignoreExceptions:
          - com.example.PaymentDeclinedException  # business error, don't retry
💡
Key rule
Never retry non-idempotent operations without an idempotency key. Retrying a payment without idempotency can charge the customer twice. Always distinguish between infrastructure errors (retry-safe) and business errors (never retry).

Observability — Logs, Metrics, Traces

The three pillars of production visibility in a distributed system

16
What are the three pillars of observability and how do you implement them in Spring Boot?
Medium▼

1. Logs — structured, correlated:

YAML — Logback JSON with trace correlation
logging.structured.format.console: ecs   # Spring Boot 3.4+ ECS structured logging
# Output: {"@timestamp":"...", "traceId":"abc123", "spanId":"def456", "message":"Order created"}
# traceId + spanId injected by Micrometer Tracing automatically

2. Metrics — Micrometer + Prometheus:

Java — Custom business metrics
@Service
public class OrderService {
    private final Counter ordersCreated;
    private final Timer orderProcessingTime;

    public OrderService(MeterRegistry registry) {
        ordersCreated = registry.counter("orders.created", "type", "standard");
        orderProcessingTime = registry.timer("orders.processing.duration");
    }

    public Order createOrder(CreateOrderRequest req) {
        return orderProcessingTime.record(() -> {
            Order order = persist(req);
            ordersCreated.increment();
            return order;
        });
    }
}

3. Traces — Micrometer Tracing + Zipkin/Tempo:

YAML — Distributed tracing config
management:
  tracing:
    sampling:
      probability: 0.1   # trace 10% in prod; 1.0 for debugging
  zipkin:
    tracing:
      endpoint: http://zipkin:9411/api/v2/spans
# Add: micrometer-tracing-bridge-brave + zipkin-reporter-brave
# TraceId/SpanId propagate automatically via HTTP headers (B3/W3C Traceparent)
💡
Golden signals
Alert on: Rate (requests/sec), Errors (error rate %), Duration (p99 latency), Saturation (queue depth, CPU%). These four catch 95% of production issues. Don't alert on anything that doesn't require human action.
17
How do you trace a request across 5 microservices in production when something goes wrong?
Medium▼

Distributed tracing walkthrough:

  1. Get the traceId from the error log, from the customer support ticket (include it in error responses), or from the API Gateway log
  2. Search Zipkin/Jaeger/Grafana Tempo by traceId — see the complete waterfall of all spans across services
  3. Find the problematic span — look for spans with high duration, error status, or that were never started (service never received the request)
  4. Look at the tags on that span — HTTP method, URL, DB query, exception class
  5. Jump to structured logs — filter by traceId in your log aggregator (Loki/Elasticsearch); all log lines from all services for that request are now correlated

Setup requirement: every service must propagate the traceparent (W3C) or X-B3-TraceId (B3) header on outbound calls. Micrometer Tracing does this automatically for RestTemplate, WebClient, and Kafka consumers.

18
What is health checking in microservices? How does Spring Boot Actuator integrate with Kubernetes probes?
Medium▼
YAML — Spring Boot Actuator health groups for K8s probes
management:
  endpoint:
    health:
      probes:
        enabled: true    # enables /actuator/health/liveness and /readiness
      group:
        readiness:
          include: db, kafka, redis  # pod not ready until dependencies healthy
        liveness:
          include: ping              # only check app process, not deps

// Kubernetes deployment:
livenessProbe:
  httpGet: { path: /actuator/health/liveness, port: 8080 }
  initialDelaySeconds: 30
  periodSeconds: 10
readinessProbe:
  httpGet: { path: /actuator/health/readiness, port: 8080 }
  initialDelaySeconds: 10
  periodSeconds: 5

Why split liveness and readiness:

  • Liveness — is the process alive? If no, K8s restarts the pod. Don't include DB here — if DB is down, don't restart all pods
  • Readiness — is this pod ready to receive traffic? If no, K8s removes it from Service endpoints. Include DB, Kafka here — if dependencies are unhealthy, stop sending traffic but don't restart

Microservice Security

JWT propagation, mTLS, zero-trust, and service-to-service auth

19
How do you handle authentication and authorization across microservices? Explain JWT propagation.
Hard▼

Pattern: JWT validated at API Gateway, propagated to services as a trusted identity claim.

  1. Client authenticates with Auth Service → receives JWT signed by auth service private key
  2. Client sends JWT in Authorization: Bearer <token> header
  3. API Gateway validates JWT signature (using auth service public key); on failure → 401
  4. Gateway passes validated JWT downstream — services trust it without re-validating signature
  5. Each service reads claims from JWT: userId, roles, tenantId
Java — Spring Security Resource Server (validate JWT)
@Configuration
public class SecurityConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
          .authorizeHttpRequests(auth -> auth
              .requestMatchers("/actuator/health/**").permitAll()
              .anyRequest().authenticated())
          .oauth2ResourceServer(oauth2 -> oauth2
              .jwt(jwt -> jwt.jwtAuthenticationConverter(jwtConverter())));
        return http.build();
    }
}

// spring.security.oauth2.resourceserver.jwt.jwk-set-uri=http://auth-service/keys
// Spring fetches public key from Auth Service and validates signatures locally

Service-to-service auth (no user context): use OAuth2 Client Credentials flow. Service A gets a machine token from Auth Service and sends it to Service B. Prevents internal services from being called without authentication — zero-trust.

20
What is mTLS and why is it needed even if you have JWT?
Medium▼

JWT authenticates the user or service identity at the application layer — it's in the HTTP body/header. It doesn't prove anything about the transport itself.

mTLS (mutual TLS) authenticates both endpoints at the transport layer:

  • Server presents its certificate → client verifies
  • Client also presents its certificate → server verifies (the "mutual" part)
  • Result: both parties know exactly who they're talking to before any HTTP data is sent

Why you need both:

  • mTLS prevents a rogue pod inside your cluster from calling a service (it won't have a valid cert)
  • JWT carries user identity + roles/claims; mTLS carries service identity
  • Without mTLS: an attacker who gains internal cluster access can call any service; with mTLS: they also need a valid client cert

Implementation: a service mesh (Istio/Linkerd) handles mTLS automatically via sidecar proxies. Without a mesh, use a certificate authority (Cert-Manager + Vault) to issue short-lived certificates per service.

Monolith to Microservices Migration

Strangler Fig, Anti-Corruption Layer, and how to avoid Big Bang rewrites

21
Explain the Strangler Fig pattern. How do you migrate a monolith incrementally?
Hard▼

The Strangler Fig (named after a tropical plant that grows around a host tree until the host dies) is the safe way to migrate a monolith — never a Big Bang rewrite.

Steps:

  1. Put a proxy in front of the monolith — API Gateway or Nginx that all traffic passes through
  2. Extract one capability as a new microservice (pick low-risk, clear boundary first)
  3. Route traffic for that capability to the new service via the proxy
  4. Run in parallel — shadow mode (send to both, compare results) until confident
  5. Remove the code from the monolith
  6. Repeat for the next capability
Text — Strangler Fig timeline example
Month 1: Extract NotificationService (stateless, low risk, clear boundary)
  → Proxy: /api/notifications → NotificationService (new)
  → Monolith: email/SMS code removed

Month 3: Extract InventoryService
  → Dual-write to old DB + new service during transition
  → Migrate DB data, verify parity, cut over

Month 6: Extract PaymentService (higher risk, more careful)
  → Shadow mode for 2 weeks: send to both, compare results
  → Gradual canary: 5% → 20% → 50% → 100% traffic

Month 9: Monolith now handles only core orders → becomes OrderService
🚫
Never do this
Big Bang rewrite: "we'll freeze the monolith and rewrite everything in microservices in 6 months." This almost always fails. The monolith keeps getting feature requests; the rewrite takes 18 months; you launch with missing features and have to maintain both. Strangler Fig is slower but it always ships.
22
What is an Anti-Corruption Layer (ACL) and when do you need one?
Medium▼

An Anti-Corruption Layer is a translation layer between two bounded contexts with incompatible models. It prevents the poor design of a legacy system from "corrupting" the clean model of a new service.

Java — ACL translating legacy Customer model
// Legacy monolith's Customer: a god-object with 50 fields, null everywhere
class LegacyCustomer { String CUST_ID; String FIRST_NM; String LAST_NM; Integer STAT_CD; ... }

// New service's domain model: clean, expressive
record Customer(CustomerId id, FullName name, CustomerStatus status) {}

// ACL: translate at the boundary — new service never sees LegacyCustomer
@Component
public class CustomerAcl {
    public Customer translate(LegacyCustomer legacy) {
        return new Customer(
            new CustomerId(legacy.CUST_ID),
            new FullName(legacy.FIRST_NM, legacy.LAST_NM),
            CustomerStatus.fromCode(legacy.STAT_CD)
        );
    }
}

Use an ACL when:

  • Integrating with a legacy system you don't control
  • A third-party API's model doesn't match your domain language
  • A shared kernel would force you to adopt a design you disagree with

How do you handle API versioning in a microservice landscape?
Medium▼

Key principle: internal services should use loose coupling over versioning. External/public APIs need explicit versioning.

Three strategies:

  • URL versioning: /api/v2/orders — most common, easy to test and log. Drawback: feels "messy", proliferates routes
  • Header versioning: Accept: application/vnd.company.order.v2+json — cleaner URLs, harder to test in browser
  • Expand-contract (internal) — add new fields alongside old ones (never remove immediately). Consumers ignore unknown fields. Old field removed only after all consumers migrate. This is the preferred pattern for internal service-to-service communication.
Java — Expand-contract pattern for Kafka events
// v1 event: {orderId, customerId, totalAmount}
// v2 event: {orderId, customerId, totalAmount, currency, discountCode}
// Strategy: publish v2 format; consumers that know v1 ignore unknown fields
// Never remove customerId until NO consumer reads it

@JsonIgnoreProperties(ignoreUnknown = true)  // consumers tolerate extra fields
public record OrderCreatedEvent(Long orderId, Long customerId, BigDecimal totalAmount) {}
💡
Schema Registry
For Kafka, use Avro + Schema Registry with a BACKWARD compatibility setting — new schema must be readable by old consumers. This enforces expand-contract at the tooling level and fails fast at deploy time if you accidentally break compatibility.
24
How do you split a shared database during monolith decomposition?
Hard▼

Database decomposition is the hardest part of microservice migration — it's irreversible and has no fallback plan once you've split.

Expand-contract database split pattern:

  1. Add dual-write: monolith writes to both the shared DB and the new service's DB simultaneously
  2. Verify consistency: read from both, compare results; log discrepancies; run for 1-2 weeks
  3. Switch reads: new service reads from its own DB; stop reading from shared DB
  4. Remove from shared schema: once confident, drop the tables from the shared DB

Handling cross-service data needs:

  • If Service A needs data from Service B's domain: call Service B's API (synchronous) or replicate via events (async read model)
  • Replace DB-level foreign keys with application-level integrity: Service A stores Service B's entity ID as a plain value, and resolves it via API when needed
  • Move cross-service reports to a dedicated analytics pipeline (Kafka → data warehouse → BI tool)

How do you test microservices? What is Consumer-Driven Contract testing?
Hard▼

Testing pyramid for microservices:

  • Unit tests — fast, within service, no network
  • Integration tests — service + real DB (Testcontainers), service + real Kafka (EmbeddedKafka or Testcontainers)
  • Contract tests — verify inter-service API contracts without running both services
  • E2E tests — minimal, only critical happy paths (slow and brittle)

Consumer-Driven Contract Testing (Pact):

  1. Consumer (e.g., OrderService calling InventoryService) writes a Pact contract: "I expect GET /stock/{id} to return {productId, quantity, available}"
  2. Contract is published to Pact Broker
  3. Provider (InventoryService) runs provider verification in CI — starts its service and verifies it satisfies all consumer contracts
  4. If InventoryService changes its API in a breaking way → CI fails before deployment
💡
Why not E2E for everything?
E2E tests require all services running simultaneously, are slow (minutes), flaky (network issues, timing), and when they fail the diagnosis is hard. Contract tests give you the same protection in seconds, without a full cluster.
26
What is the "distributed monolith" anti-pattern and how do you recognize it?
Medium▼

A distributed monolith is the worst of both worlds: the operational complexity of microservices with none of the independence benefits. You've split the code across services, but the services are still tightly coupled.

Signs you have a distributed monolith:

  • Deploying service A requires deploying services B and C at the same time
  • Services share a database (direct SQL across service boundaries)
  • Synchronous call chains: A → B → C → D on every request (cascade failure)
  • A team cannot release their service without coordinating with another team
  • Schema changes require simultaneous changes in multiple services
  • You have "util" or "common" jars shared across all services that change frequently

Fix: go back to the bounded context analysis. Which capabilities are truly independent? Merge over-split services. Replace synchronous chains with events. Each service must own its data.

How do you handle distributed configuration and secrets across microservices?
Medium▼

Configuration strategy (by environment sensitivity):

  • Non-sensitive config: Kubernetes ConfigMaps → mounted as env vars or files. Spring Boot picks up via @ConfigurationProperties
  • Sensitive config (secrets): Kubernetes Secrets (base64 encoded, or better: backed by HashiCorp Vault via External Secrets Operator)
  • Spring Cloud Config Server: central Git-backed config server; services pull config at startup and on refresh. Useful for non-K8s or multi-cloud environments
YAML — External Secrets Operator pulling from Vault
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: order-service-secrets
spec:
  refreshInterval: 15m
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: order-service-secrets   # creates K8s Secret
  data:
    - secretKey: DATABASE_PASSWORD
      remoteRef:
        key: secret/order-service
        property: db-password

Never: hardcode credentials, commit secrets to Git (even in a "private" repo), put secrets in container images, or log secrets.

Summarise: when should you NOT use microservices?
Medium▼

Microservices are an organisational and operational pattern as much as a technical one. The overhead is only justified at scale.

Don't use microservices when:

  • Team is ≤5-8 engineers — the operational overhead exceeds the productivity gain. A well-structured monolith ("Modulith") with clear package boundaries is the right answer
  • Domain isn't stable — if you don't know your bounded contexts yet, splitting too early locks in wrong boundaries. Monolith first, extract services when the seams become clear
  • You don't have DevOps capability — microservices require CI/CD pipelines, container orchestration, distributed tracing, and on-call culture. Without these, deployments become a nightmare
  • The problem is a simple CRUD app — a 3-table app with 100 users doesn't need 5 services
💡
Martin Fowler's rule
"Don't start with microservices. Start with a monolith, identify the seams of independent deployability, and extract services when the monolith starts hurting your team's ability to work independently." The Modulith pattern (Spring Modulith) gives you internal module isolation without the distributed systems complexity — a good intermediate step.
