Kafka Architecture

Brokers, ZooKeeper/KRaft, topics, partitions, replication

1
Explain Kafka's core architecture. What are brokers, topics, partitions, and replicas?
Medium▼
  • Broker — a Kafka server. A cluster has multiple brokers for fault tolerance and load distribution. Each broker holds some topic partitions.
  • Topic — a named, durable, append-only log. Messages are never deleted on consume — retained by time or size policy.
  • Partition — unit of parallelism and ordering. A topic splits into N partitions distributed across brokers. Messages within one partition are strictly ordered. Consumers in a group each own a subset of partitions.
  • Replica — each partition has a leader + N-1 follower replicas on different brokers. Leader handles all reads/writes; followers replicate. If the leader fails, a follower is elected.
  • Controller — one broker elected as controller; manages partition leader elections and cluster metadata.
Text — Cluster layout
Topic: orders  (3 partitions, replication-factor 3)

Partition 0:  Leader=Broker1  Replicas=[Broker1, Broker2, Broker3]  ISR=[1,2,3]
Partition 1:  Leader=Broker2  Replicas=[Broker2, Broker3, Broker1]  ISR=[2,3,1]
Partition 2:  Leader=Broker3  Replicas=[Broker3, Broker1, Broker2]  ISR=[3,1,2]

ISR = In-Sync Replicas (replicas fully caught up to leader)
If Broker1 dies → Partition 0 leader election → Broker2 or Broker3 takes over
💡
Key interview insight
Kafka achieves high throughput by writing sequentially to disk (not random I/O) and using zero-copy transfer (sendfile syscall) to deliver messages to consumers. This is why Kafka outperforms systems that use random writes.
2
What is ZooKeeper's role in Kafka? What is KRaft mode and why was it introduced?
Hard▼

ZooKeeper (legacy, pre-3.3) stored Kafka cluster metadata: broker registration, topic configs, partition leader elections, controller election, ACLs, consumer offsets (early versions). Every Kafka operation touched ZooKeeper, creating a bottleneck and operational complexity (two systems to manage).

KRaft (Kafka Raft, Kafka 3.3+ GA) — removes ZooKeeper entirely. Kafka brokers themselves implement the Raft consensus protocol to manage metadata. Benefits:

  • Single system to operate (no ZooKeeper cluster to maintain)
  • Controller failover in milliseconds (was seconds with ZK)
  • Scales to millions of partitions (ZK was limited to ~200K)
  • Simpler security model (one auth system)
Properties — KRaft mode config
# server.properties (KRaft mode)
process.roles=broker,controller   # combined or separate roles
node.id=1
controller.quorum.voters=1@host1:9093,2@host2:9093,3@host3:9093
listeners=PLAINTEXT://:9092,CONTROLLER://:9093
log.dirs=/var/kafka/logs

# Format storage before first start:
# kafka-storage.sh format -t $(kafka-storage.sh random-uuid) -c server.properties
3
How does Kafka handle message retention? What are the retention strategies?
Medium▼

Kafka retains all messages regardless of consumer state. Consumers track their own position (offset) — re-reading is free. Two retention strategies per topic:

StrategyConfigBehaviour
Time-basedretention.ms (default: 7 days)Delete segments older than N ms
Size-basedretention.bytes per partitionDelete oldest segments when partition exceeds size
Compactcleanup.policy=compactKeep only the latest message per key — like a changelog; tombstone (null value) = deletion
Bash
# Create compacted topic — useful for event sourcing / CDC
kafka-topics.sh --create --topic user-updates \
  --config cleanup.policy=compact \
  --config min.cleanable.dirty.ratio=0.01 \
  --config segment.ms=100

# Override retention on existing topic
kafka-configs.sh --alter --topic orders \
  --add-config retention.ms=86400000  # 1 day
💡
Log compaction use case
Use compacted topics for "latest state" scenarios: user profiles, product inventory, feature flags. New consumers can bootstrap current state by reading the compacted topic — no need for a separate snapshot mechanism.
4
How do you choose the number of partitions for a topic?
Hard▼

Partitions determine maximum parallelism. Key factors:

  • Target throughput — one partition handles ~10-100 MB/s depending on message size. If you need 1 GB/s, you need at least 10-100 partitions
  • Consumer parallelism — max parallel consumers = number of partitions. 20 partitions → max 20 consumers in a group
  • Ordering requirements — if all messages must be globally ordered: 1 partition (but no parallelism). Per-key order: use key-based routing across partitions
  • Broker capacity — each partition costs ~filehandles and memory per broker. Rule of thumb: total partitions ≤ 4000 per broker, ≤ 200K per cluster
⚠️
You can't easily decrease partitions
Increasing partitions later breaks key-to-partition mapping (same key → different partition). Plan capacity upfront. A common formula: partitions = max(target_throughput_MB/s ÷ producer_throughput_per_partition, num_consumers_needed)
5
What is a Schema Registry? Why is it important in a Kafka ecosystem?
Hard▼

Schema Registry (Confluent or Apicurio) stores and enforces Avro/Protobuf/JSON Schema definitions. Producers register a schema; consumers fetch it by ID embedded in the message header. Prevents breaking schema changes from reaching consumers.

Java — Avro with Spring Kafka
// Avro schema (order.avsc)
// {
//   "type": "record",
//   "name": "OrderEvent",
//   "fields": [
//     {"name": "orderId", "type": "string"},
//     {"name": "userId",  "type": "long"},
//     {"name": "total",   "type": "double"},
//     {"name": "status",  "type": {"type":"enum","name":"Status","symbols":["PLACED","PAID","SHIPPED"]}}
//   ]
// }

// application.yml
// spring.kafka.producer.value-serializer: io.confluent.kafka.serializers.KafkaAvroSerializer
// spring.kafka.properties.schema.registry.url: http://schema-registry:8081

// Producer — sends Avro-serialized message
kafkaTemplate.send("orders", orderId, OrderEvent.newBuilder()
    .setOrderId(orderId)
    .setUserId(userId)
    .setTotal(total)
    .setStatus(Status.PLACED)
    .build());

Compatibility modes: BACKWARD (new schema reads old data), FORWARD (old schema reads new data), FULL (both), NONE. Use BACKWARD or FULL in production — lets you deploy consumers before producers.

Producer Internals & Config

acks, batching, compression, idempotence, transactions

6
Explain producer acks=0, acks=1, and acks=all. What are the trade-offs?
Hard▼
acksDurabilityLatencyRisk
0 (fire-and-forget)None — broker may not receive itLowest (no wait)Message loss on any failure
1 (leader only)Leader acknowledges writeLow (~1 RTT)Loss if leader crashes before replication
all / -1 (all ISR)All ISR replicas acknowledgeHigher (~2+ RTT)No loss as long as ≥1 ISR is alive
Properties
# Production-safe producer config
acks=all                    # wait for all in-sync replicas
retries=2147483647          # retry forever (rely on delivery.timeout.ms)
delivery.timeout.ms=120000  # 2 min overall delivery deadline
enable.idempotence=true     # exactly-once per partition (requires acks=all)
max.in.flight.requests.per.connection=5  # max with idempotence

# Performance tuning
batch.size=65536            # 64 KB batch (default 16KB — too small for high throughput)
linger.ms=5                 # wait up to 5ms to fill a batch
compression.type=snappy     # good ratio + fast decompression (lz4 also good)
buffer.memory=67108864      # 64MB producer buffer
🚫
acks=all is not enough
Also set min.insync.replicas=2 on the broker/topic. With replication-factor=3 and min.insync.replicas=2, the producer blocks if fewer than 2 replicas are in sync. Without this, acks=all with 1 ISR (the leader) = same as acks=1.
7
How does Kafka producer batching and compression work? How do you maximise throughput?
Medium▼

The producer accumulates messages in a per-partition buffer (RecordAccumulator). A batch is sent when either batch.size is reached OR linger.ms elapses (whichever first). Compression is applied per batch — larger batches compress better.

Java
// High-throughput producer with Spring Kafka
@Bean
public ProducerFactory<String, OrderEvent> producerFactory() {
    Map<String, Object> props = new HashMap<>();
    props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "kafka:9092");
    props.put(ProducerConfig.ACKS_CONFIG, "all");
    props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);
    props.put(ProducerConfig.BATCH_SIZE_CONFIG, 65536);   // 64KB
    props.put(ProducerConfig.LINGER_MS_CONFIG, 5);
    props.put(ProducerConfig.COMPRESSION_TYPE_CONFIG, "snappy");
    props.put(ProducerConfig.BUFFER_MEMORY_CONFIG, 67108864L); // 64MB
    return new DefaultKafkaProducerFactory<>(props);
}

Compression algorithms: none → gzip (best ratio) → snappy (fast, good ratio) → lz4 (fastest, less ratio) → zstd (best ratio + fast, Kafka 2.1+). For high throughput: lz4 or snappy.

What is the Kafka Transactional API? How do you implement exactly-once across multiple topics?
Hard▼

Kafka transactions allow atomically writing to multiple partitions (and marking consumer offsets). Either all writes commit or all abort — prevents partial writes on failure.

Java
// Producer config for transactions
props.put(ProducerConfig.TRANSACTIONAL_ID_CONFIG, "order-processor-1"); // unique per producer instance
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);

// Init once at startup
producer.initTransactions();

// Atomic write to multiple topics
try {
    producer.beginTransaction();

    producer.send(new ProducerRecord<>("orders", order));
    producer.send(new ProducerRecord<>("audit-log", auditEntry));
    producer.send(new ProducerRecord<>("notifications", notification));

    // Commit consumer offsets atomically with the produce
    producer.sendOffsetsToTransaction(offsets, consumerGroupMetadata);

    producer.commitTransaction();
} catch (ProducerFencedException e) {
    producer.close(); // another instance took over — fatal
} catch (Exception e) {
    producer.abortTransaction(); // rollback all writes this transaction
}

// Consumer must set isolation.level=read_committed to skip aborted transactions
⚠️
Performance cost
Transactions add ~20% latency overhead. Use them when you genuinely need cross-topic atomicity. For most cases, idempotent producers + idempotent consumers (dedup by event ID) is simpler and faster.

How does the Kafka partitioner work? When would you write a custom partitioner?
Medium▼

Default partitioner (Kafka 2.4+) — "sticky partitioning": batches messages to the same partition until the batch is full or linger.ms elapses, then switches. Better batching than round-robin.

Key-based — if a message has a key: partition = hash(key) % numPartitions. Same key always → same partition (ordering per key).

Custom partitioner use cases:

  • Route by region: EU customers → EU partition (data residency)
  • A hot key problem: one user sends 80% of traffic → all goes to 1 partition; route to random partition for that key
  • Business priority routing: VIP orders → dedicated partition consumed by faster consumer
Java
public class RegionPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        int numPartitions = cluster.partitionCountForTopic(topic);
        OrderEvent order = (OrderEvent) value;
        // Route EU orders to first half, others to second half
        if ("EU".equals(order.getRegion())) {
            return Math.abs(key.hashCode()) % (numPartitions / 2);
        }
        return numPartitions / 2 + Math.abs(key.hashCode()) % (numPartitions / 2);
    }
}

Consumer Internals & Config

Poll loop, offset management, rebalancing, lag

10
How does the Kafka consumer poll loop work? What is max.poll.records and max.poll.interval.ms?
Hard▼

The consumer's poll() call does two things: (1) fetches records from the broker, (2) sends a heartbeat to the group coordinator to stay in the group.

Java
// Manual poll loop (Spring Kafka hides this — shown for understanding)
while (true) {
    ConsumerRecords<String, OrderEvent> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<?, ?> record : records) {
        processRecord(record);  // must complete before next poll()
    }
    consumer.commitSync(); // commit after processing batch
}

// Key configs:
// max.poll.records=500          — max records returned per poll (default 500)
// max.poll.interval.ms=300000   — if poll() not called within 5min, consumer is kicked
// session.timeout.ms=45000      — heartbeat timeout (consumer crashed)
// heartbeat.interval.ms=3000    — heartbeat frequency (1/3 of session.timeout)
🚫
Common bug
If processing 500 records takes longer than max.poll.interval.ms, the consumer is kicked from the group, causing a rebalance. Fix: lower max.poll.records, increase max.poll.interval.ms, or process records asynchronously (but then you must manage offset commits carefully).
11
What is Consumer Group Rebalancing? What is Cooperative Rebalancing (Kafka 2.4+)?
Hard▼

When consumers join/leave a group, partitions are reassigned — this is a rebalance. During rebalance, all consumers in the group stop processing (Stop-the-World).

Triggers: consumer joins, consumer leaves (gracefully or timeout), partition count changes, subscription changes.

Legacy (Eager) rebalance: all consumers release all partitions, then re-assign from scratch. Processing stops for the entire group during this.

Cooperative/Incremental rebalance (Kafka 2.4+): only revoked partitions move; consumers processing unaffected partitions continue without stopping.

Properties
# Enable cooperative rebalancing (Spring Kafka 2.7+ / Kafka 2.4+)
partition.assignment.strategy=org.apache.kafka.clients.consumer.CooperativeStickyAssignor

# group.instance.id — static membership: avoids rebalance on consumer restart
# Same instance.id rejoins with its old partitions without triggering rebalance
group.instance.id=order-consumer-pod-1
💡
K8s + Kafka
In Kubernetes, set group.instance.id to the Pod name (via Downward API env var). When a pod restarts (same name), it reclaims its partitions without rebalancing the whole group — critical for production stability.
12
What is consumer lag and how do you monitor and reduce it?
Medium▼

Consumer lag = difference between the latest offset produced and the consumer's current committed offset. High lag = consumer falling behind producers.

Bash
# Check lag for a consumer group
kafka-consumer-groups.sh --bootstrap-server kafka:9092 \
  --describe --group order-processors

# Output:
# GROUP              TOPIC   PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
# order-processors   orders  0          1023400         1025000         1600
# order-processors   orders  1          987200          990000          2800

# Export lag to Prometheus via kafka-exporter or burrow
# Alert when lag > 10000 and growing

Reducing lag:

  • Add more consumers (up to partition count)
  • Increase partition count (then also add consumers)
  • Optimise processing: async I/O, batch DB writes, connection pooling
  • Increase fetch.min.bytes and fetch.max.wait.ms to reduce round trips
  • Scale consumer horizontally (multiple pods)

Java — Spring Kafka manual commit
// application.yml
// spring.kafka.listener.ack-mode: MANUAL_IMMEDIATE
// spring.kafka.consumer.enable-auto-commit: false

@KafkaListener(topics = "orders")
public void process(ConsumerRecord<String, OrderEvent> record,
                    Acknowledgment ack) {
    try {
        processOrder(record.value());
        ack.acknowledge(); // commit only after successful processing
    } catch (NonRetryableException e) {
        // Send to DLT, then ack to not block the partition
        deadLetterTemplate.send("orders.DLT", record.value());
        ack.acknowledge();
    }
    // Retryable exceptions: don't ack — message will be redelivered
}

Delivery Guarantees & Error Handling

Idempotence, DLT, retry strategies

14
How do you implement a Dead Letter Topic (DLT) pattern with Spring Kafka?
Hard▼

When a consumer fails to process a message after N retries, the message is sent to a Dead Letter Topic (e.g. orders.DLT) for later inspection and manual remediation.

Java
// DefaultErrorHandler with exponential backoff + DLT (Spring Kafka 2.8+)
@Bean
public DefaultErrorHandler errorHandler(KafkaTemplate<Object, Object> template) {
    DeadLetterPublishingRecoverer recoverer = new DeadLetterPublishingRecoverer(template,
        (r, e) -> new TopicPartition(r.topic() + ".DLT", r.partition()));

    ExponentialBackOffWithMaxRetries backoff = new ExponentialBackOffWithMaxRetries(3);
    backoff.setInitialInterval(1_000L);  // 1s, 2s, 4s
    backoff.setMultiplier(2.0);
    backoff.setMaxInterval(10_000L);    // cap at 10s

    var handler = new DefaultErrorHandler(recoverer, backoff);
    // Don't retry on deserialization errors or business logic exceptions
    handler.addNotRetryableExceptions(DeserializationException.class, BusinessRuleException.class);
    return handler;
}

// DLT consumer — for monitoring/reprocessing
@KafkaListener(topics = "orders.DLT")
public void handleDlt(@Payload OrderEvent order,
                      @Header(KafkaHeaders.EXCEPTION_MESSAGE) String errorMsg) {
    log.error("DLT received: {} reason: {}", order, errorMsg);
    alertingService.notify(order, errorMsg);
}

What is the difference between idempotent producer and transactional producer? When do you need each?
Hard▼
Idempotent ProducerTransactional Producer
GuaranteesExactly-once per partition per session (dedup by sequence number)Atomic writes across multiple partitions/topics
Configenable.idempotence=truetransactional.id=unique-id (implies idempotence)

Practical advice
Always enable idempotence (enable.idempotence=true) — it's free safety. Use transactions only for the consume-transform-produce pattern (Kafka Streams uses them automatically). For most microservices, idempotent consumer (dedup table) + idempotent producer is simpler and sufficient.

Kafka Streams

Stream processing, stateful operations, windowing, KTable

16
What is Kafka Streams? How does it differ from using Kafka consumers directly?
Hard▼

Kafka Streams is a client library for building stateful stream processing apps on top of Kafka. It runs inside your application process — no separate cluster needed (unlike Flink or Spark).

Plain ConsumerKafka Streams
Stateful operationsManual (DB, Redis)Built-in: count, aggregate, join with RocksDB state store
Exactly-onceComplex to implementBuilt-in (processing.guarantee=exactly_once_v2)
WindowingManualTumbling, hopping, sliding, session windows
Stream-Stream/Stream-Table joinsManualKStream-KStream, KStream-KTable join operators
DeploymentAny JVM appAny JVM app — scales by partition count
Java — Kafka Streams order aggregation
StreamsBuilder builder = new StreamsBuilder();

// Read from orders topic
KStream<String, OrderEvent> orders = builder.stream("orders");

// Count orders per user in a 1-hour tumbling window
KTable<Windowed<String>, Long> orderCounts = orders
    .groupBy((key, order) -> order.getUserId())     // re-key by userId
    .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofHours(1)))
    .count(Materialized.as("order-count-store"));  // RocksDB state store

// Filter high-value orders and enrich with user data (KStream-KTable join)
KTable<String, User> users = builder.table("user-updates");
orders
    .filter((k, o) -> o.getTotal() > 1000)
    .join(users, (order, user) -> new VIPOrderAlert(order, user))
    .to("vip-alerts");
17
What is the difference between KStream and KTable? What is a changelog topic?
Hard▼

KStream — unbounded sequence of events. Each record is an independent event. Reading the stream replays all events.

KTable — materialised view of the latest value per key. Like a database table updated by a stream. A new message with key K replaces the previous K entry. A null value = tombstone (delete).

GlobalKTable — like KTable but fully replicated on every application instance (useful for lookup tables like products or currencies — avoids co-partitioning requirement for joins).

Changelog topic — Kafka Streams backs every state store with a changelog topic (usually <app-id>-<store-name>-changelog). When an instance restarts, it restores state by replaying this topic into its local RocksDB store — no external DB needed.

What types of windows does Kafka Streams support? Explain tumbling vs session windows.
Medium▼
Window TypeBehaviourUse case
TumblingFixed-size, non-overlapping: 0-1h, 1-2h, 2-3hHourly sales totals, metrics aggregation
HoppingFixed-size, overlapping: window every 15min over 1h dataSliding averages, fraud detection trends
SlidingUpdated on every event; window always contains events within N ms of each otherReal-time "last N minutes" computations
SessionDynamic size; gap of inactivity N ms closes the windowUser sessions, clickstream analysis
💡
Late data
Use grace(Duration.ofMinutes(5)) to accept late-arriving events into windows. Without grace, late events are silently dropped. With Kafka Streams 3.x, you can also use ofSizeAndGrace() factory methods.

20
A consumer group's lag is constantly growing. How do you diagnose and fix it?
Hard▼

Diagnosis steps:

  1. Check lag per partition: is lag even or concentrated on one partition? → hot partition or slow consumer
  2. Check consumer CPU/memory: is the consumer process under-resourced?
  3. Check processing time per record: slow DB query, HTTP call, or lock contention?
  4. Check rebalance events: frequent rebalances reset lag measurement and waste time
  5. Check producer throughput: sudden spike in messages?

Fixes ranked by effort:

  • Add consumer instances (up to partition count) — fastest
  • Optimise processing: batch DB inserts, async I/O, connection pool tuning
  • Increase partitions (then scale consumers accordingly)
  • Offload heavy processing to a thread pool (but manage acks carefully)
  • KEDA auto-scaling: scale consumer pods on lag metric

Bash

  • For compacted topics: existing data isn't redistributed — old data stays in old partitions; only new messages go to new partitions

Spring Kafka Patterns

Listeners, retry, concurrency, testing

23
How do you configure concurrency in Spring Kafka listeners? What is ConcurrentKafkaListenerContainerFactory?
Medium▼
Java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, OrderEvent> kafkaListenerContainerFactory(
        ConsumerFactory<String, OrderEvent> cf, DefaultErrorHandler errorHandler) {

    var factory = new ConcurrentKafkaListenerContainerFactory<String, OrderEvent>();
    factory.setConsumerFactory(cf);

    // concurrency = number of consumer threads per listener
    // Should match partition count or be a divisor of it
    factory.setConcurrency(6);  // 6 threads, each owns 2 partitions (12 total)

    factory.setCommonErrorHandler(errorHandler);

    // Batch listener — process a list of records at once
    factory.setBatchListener(true);

    // AckMode — MANUAL_IMMEDIATE requires Acknowledgment param
    factory.getContainerProperties().setAckMode(AckMode.MANUAL_IMMEDIATE);

    return factory;
}

// Batch listener
@KafkaListener(topics = "orders", containerFactory = "batchFactory")
public void processBatch(List<OrderEvent> orders, Acknowledgment ack) {
    orderRepository.saveAll(orders.stream().map(mapper::toEntity).toList());
    ack.acknowledge();
}

What causes "NotLeaderForPartitionException" and how do you handle it?
Hard▼

Thrown when a producer/consumer sends a request to a broker that is no longer the leader for a partition — happens during leader elections (broker failure, controlled shutdown, rolling restart).

Handling: Kafka clients (and Spring Kafka) handle this automatically — they fetch updated metadata and retry the request. You should see this as a brief latency spike, not a failure, as long as ISR has at least one replica.

If persistent: check replica.lag.time.max.ms — followers too far behind are removed from ISR, reducing durability. Investigate broker disk I/O or network between brokers.

What is Kafka MirrorMaker 2? When do you need it?
Hard▼

MirrorMaker 2 (MM2) is a Kafka Connect-based cluster replication tool that continuously copies topics between Kafka clusters. Use cases:

  • Active-Passive DR — replicate from primary cluster to DR cluster; fail over in case of disaster
  • Active-Active — bidirectional replication (tricky — need cycle detection)
  • Data migration — move topics to a new cluster without downtime
  • Cross-region — aggregate regional clusters into a central cluster for analytics

MM2 translates offsets between clusters and adds source cluster prefix to topic names (source.orders). Consumers can checkpoint their offsets so failover doesn't lose progress.

What is the "hot partition" problem? How do you detect and fix it?
Hard▼

A hot partition occurs when one partition receives disproportionately more messages than others — usually caused by a high-cardinality key value (e.g. one user generates 80% of events).

Detection: Compare MessagesInPerSec per partition — one partition at 10x others = hot partition.

Fixes:

  • Composite key — append random suffix to hot key: userId + "_" + (random 0-9). Consumers must deduplicate by original key if ordering matters
  • Custom partitioner — detect known hot keys and spread them across multiple partitions
  • Salting — salt the key at produce time; unsalt at consume time
  • Separate topic — route traffic from VIP/power users to a dedicated topic with more partitions

Message Queue vs Event Bus vs Message Broker — core differences and when to use each
Medium▼

These three terms are often confused. They represent different patterns for asynchronous communication:

Message Queue — point-to-point channel. Producer sends a message; exactly one consumer receives and processes it. Message is removed after consumption. Classic use: job/task queue (background email sending, image processing). Examples: AWS SQS, RabbitMQ queues.

Event Bus — publish/subscribe within a single process or bounded context. Publishers emit events; multiple in-process listeners react. No persistence. Examples: Spring's ApplicationEventPublisher, Guava EventBus. Useful for decoupling layers within a service.

Message Broker — infrastructure component that routes, filters, and delivers messages between distributed services. Supports both point-to-point and pub/sub. Adds persistence, delivery guarantees, and routing rules. Examples: RabbitMQ (broker with exchanges/queues), Apache Kafka (distributed log broker).

Java — Spring in-process Event Bus
// Publisher — within the same Spring application
@Service
class OrderService {
    private final ApplicationEventPublisher publisher;

    public void placeOrder(Order order) {
        orderRepo.save(order);
        publisher.publishEvent(new OrderPlacedEvent(order)); // in-process, sync
    }
}

// Listener — decoupled but in the same JVM
@Component
class NotificationListener {
    @EventListener
    @Async // make it async to avoid blocking the caller
    public void onOrderPlaced(OrderPlacedEvent event) {
        emailService.sendConfirmation(event.getOrder());
    }
}
ℹ️
When to escalate to a broker
Use an in-process event bus for lightweight decoupling within one service. Escalate to a message broker (Kafka/RabbitMQ) when: consumers are in different services, you need durability (survive restarts), or you need replay/fan-out to multiple independent consumers.

Dead Letter Queue (DLQ) — what is it? Spring Boot implementation with @DltHandler
Medium▼

A Dead Letter Queue is a holding area for messages that failed processing after all retries are exhausted. Instead of silently dropping failed messages or blocking the queue with a poison pill, DLQ captures them for human review, alerting, or replay after fixing the bug.

Without DLQ: a bad message (e.g. malformed JSON, null pointer) causes infinite retry loops, blocking queue processing for all subsequent messages.

With DLQ: after N retries, the message is moved to the DLQ topic. Other messages continue processing. Ops team investigates the DLQ separately.

Java — Spring Kafka DLQ with @DltHandler
// Non-blocking retry + DLT (Dead Letter Topic) pattern
@Configuration
class KafkaRetryConfig {
    @Bean
    public RetryTopicConfiguration retryConfig(KafkaTemplate<?, ?> template) {
        return RetryTopicConfigurationBuilder
            .newInstance()
            .maxAttempts(3)                // 3 attempts total
            .exponentialBackoff(1000, 2, 10000) // 1s, 2s, 4s (max 10s)
            .retryOn(TransientException.class)   // only retry transient errors
            .notRetryOn(ValidationException.class) // send directly to DLT
            .includeTopic("orders")
            .create(template);
        // Creates topics: orders-retry-0, orders-retry-1, orders-dlt
    }
}

@Component
class OrderConsumer {

    @KafkaListener(topics = "orders")
    public void processOrder(String payload) {
        // Throws TransientException on DB timeout — will be retried
        orderService.process(payload);
    }

    @DltHandler
    public void handleDlt(String payload, @Header(KafkaHeaders.RECEIVED_TOPIC) String topic) {
        // Called when message lands in the dead letter topic
        log.error("Message failed all retries. Topic: {}, Payload: {}", topic, payload);
        alertService.notifyOps(payload); // PagerDuty / Slack alert
        deadLetterRepository.save(new DeadLetter(topic, payload, Instant.now()));
    }
}
✅
Non-blocking retry advantage
Spring Kafka's non-blocking retry routes failed messages to separate retry topics instead of blocking the original partition. Other messages in the same partition continue processing while the failed message waits in the retry topic — far better than blocking retry which can cascade into partition lag.

Message delivery guarantees — at-most-once, at-least-once, exactly-once with Kafka examples
Hard▼

Delivery semantics define what happens to a message when a failure occurs between produce and consume:

At-most-once — fire and forget. Message may be lost if producer/broker fails. Offset committed before processing. Use for: metrics, telemetry, non-critical logs. acks=0, auto-commit before processing.

At-least-once — message is delivered at least once but may be duplicated on retry. Offset committed after successful processing. Consumers must be idempotent. Use for: most business events. acks=all, commit after processing.

Exactly-once — each message processed exactly once, no duplicates, no loss. Requires Kafka transactions + idempotent producer. Highest overhead. Use for: financial transactions, inventory updates.

Java — Kafka exactly-once with transactions
// application.yml — exactly-once semantics
spring:
  kafka:
    producer:
      transaction-id-prefix: tx-  # enables idempotent producer + transactions
      acks: all
    consumer:
      isolation-level: read_committed  # only read committed (non-transactional) records
      enable-auto-commit: false

// Java — consume-transform-produce in a transaction
@Transactional("kafkaTransactionManager")
@KafkaListener(topics = "payments")
public void processPayment(String payload, Acknowledgment ack) {
    Payment payment = parse(payload);

    // DB write + produce to output topic in one atomic transaction
    accountRepository.debit(payment.getAccountId(), payment.getAmount());
    kafkaTemplate.send("payment-events", payment.toEvent());

    ack.acknowledge(); // offset committed as part of transaction
    // If any step fails: DB rolls back, output not visible, offset not committed
}

// Idempotency key pattern for at-least-once (simpler alternative)
public void processIdempotent(String messageId, String payload) {
    if (processedMessages.contains(messageId)) return; // deduplicate
    doProcess(payload);
    processedMessages.add(messageId); // Redis SET with TTL
}
⚠️
Exactly-once complexity
Kafka's exactly-once only covers the Kafka-to-Kafka path. If your consumer writes to a DB AND produces to Kafka, you need either Kafka transactions (which use the 2PC-like transaction coordinator) or design your consumer to be idempotent with at-least-once delivery — much simpler and often sufficient.

Pub/Sub vs Point-to-Point messaging patterns with real Java examples
Medium▼

These are the two fundamental messaging patterns. Most message brokers support both.

Point-to-Point (P2P) — one producer sends a message to a queue; exactly one consumer receives and processes it. Other consumers compete for the next message. The message is deleted after processing. Think: task distribution.

Publish/Subscribe (Pub/Sub) — one producer publishes an event to a topic; ALL subscribers receive a copy. Each subscriber processes independently. Think: event broadcasting, change notification.

Java — P2P vs Pub/Sub with Spring
// POINT-TO-POINT with RabbitMQ — competing consumers
// 3 consumers compete — only one gets each message (load balancing)
@RabbitListener(queues = "image-processing")
class ImageProcessor { // instance 1
    public void process(ImageUploadedEvent event) {
        imageResizer.resize(event.getImageId()); // this work item is done
    }
}
// instances 2 and 3 are identical — RabbitMQ round-robins messages

// PUBLISH/SUBSCRIBE with Kafka — independent consumer groups
// Both groups receive EVERY order event independently
@KafkaListener(topics = "orders", groupId = "inventory-service")
class InventoryConsumer {
    public void onOrder(String event) {
        inventory.reserveStock(event); // inventory gets every order
    }
}

@KafkaListener(topics = "orders", groupId = "billing-service")
class BillingConsumer {
    public void onOrder(String event) {
        billing.chargeCustomer(event); // billing also gets every order
    }
}
// Different groupId = independent subscription = pub/sub
// Same groupId = competing consumers = point-to-point
ℹ️
Kafka's unification of both patterns
Kafka implements both patterns through consumer group IDs. Same groupId across instances = P2P (work queue). Different groupId per service = Pub/Sub (each service sees all events). This makes Kafka exceptionally versatile — you don't choose the pattern at the broker level, you choose it at the consumer configuration level.
