🎯 System Design Interview Framework

Always follow this structure — interviewers reward process, not just the final answer.

1. Clarify (5 min)
Users, scale, SLA, read/write ratio, geographic distribution
2. Estimate (3 min)
QPS, storage, bandwidth — back-of-envelope math
3. High-level design
Draw boxes and arrows — API gateway, services, DB, cache
4. Deep dive
Pick the hardest component and go deep (data model, consistency, failover)
5. Trade-offs
Discuss what you'd change for different constraints

Distributed Systems Fundamentals

CAP, consistency, scalability, load balancing

1
Explain the CAP theorem. Give a real example of a CP vs AP system.
Hard▼

A distributed system can only guarantee 2 of 3 in the presence of a network partition:

PropertyMeaning
Consistency (C)Every read gets the most recent write or an error — all nodes see the same data at the same time
Availability (A)Every request gets a (possibly stale) response — system is always responsive
Partition Tolerance (P)System continues operating despite network failures between nodes — always required in a real distributed system

Since P is mandatory (networks fail), the real choice is CP vs AP:

  • CP systems — HBase, ZooKeeper, etcd: prefer consistency; return error rather than stale data during partition
  • AP systems — Cassandra, CouchDB, DynamoDB: prefer availability; may return stale data during partition; eventually consistent
💡
Real example
A bank balance query should be CP — better to show an error than a stale (incorrect) balance. A social media "like count" can be AP — showing 2,341 instead of 2,342 for a few seconds is acceptable.

4
Design a URL shortener (like bit.ly). Discuss trade-offs.
Hard▼

Requirements clarification: 100M URLs/day created, 10B redirects/day (read-heavy 100:1), URLs expire after 5 years, analytics needed.

High-level design:

  • Shorten API — generate unique 7-char key (Base62: a-z, A-Z, 0-9 = 62^7 = 3.5T combinations)
  • Redirect — look up key in cache → DB, respond with 301 (cached by browser) or 302 (enables analytics on repeat visits)
  • Storage — PostgreSQL for URL mapping; Redis for cache (TTL matching URL expiry)
SQL
-- URL table
CREATE TABLE urls (
  short_key  CHAR(7)     PRIMARY KEY,
  long_url   TEXT        NOT NULL,
  user_id    BIGINT,
  created_at TIMESTAMPTZ DEFAULT now(),
  expires_at TIMESTAMPTZ,
  click_count BIGINT     DEFAULT 0
);

CREATE INDEX idx_expires ON urls(expires_at);  -- for cleanup jobs

Key generation approaches:

  • Hash (MD5/SHA256) + take 7 chars — fast but collision risk; need collision handling
  • Auto-increment ID → Base62 encode — no collisions, but predictable sequential keys
  • Random key generation — check uniqueness in DB/Redis Bloom filter before saving

Scale to 10B redirects/day (~115K/sec): Cache top 20% of URLs in Redis (hot URLs = 80% of traffic). Use CDN for redirect responses. Read replicas for DB.

Database Design & Scaling

SQL vs NoSQL, sharding, replication, ACID

5
When do you choose SQL vs NoSQL? Give specific use cases for each.
Medium▼
SQL (Relational)NoSQL
Data modelStructured, schema-enforced, joinsDocument, key-value, graph, column — flexible schema
ConsistencyACID transactionsBASE (eventually consistent, varies by type)
ScalingVertical primary; horizontal via read replicasHorizontally sharded natively
Use forFinancial data, orders, user accounts — relational data with transactionsUser sessions, caching, product catalogs, real-time analytics, content
ExamplesPostgreSQL, MySQL, OracleMongoDB (doc), Redis (KV), Cassandra (wide-col), Neo4j (graph)
💡
Default choice
Default to PostgreSQL. Reach for NoSQL when you have a specific scaling or data-model reason (e.g., schema-less documents, >10TB write throughput, graph traversals). "NoSQL is faster" is a myth without a specific bottleneck.
6
What is database sharding and what are its challenges?
Hard▼

Sharding splits data across multiple databases (shards) based on a shard key. Each shard holds a subset of the data.

Sharding strategies:

  • Range-based — shard by value range (user_id 1-1M → shard 1, etc.). Simple but creates hot spots if one range is accessed more
  • Hash-based — hash(shard_key) % N_shards. Even distribution but hard to add shards (consistent hashing solves this)
  • Directory-based — lookup table maps each key to a shard. Flexible but lookup table is a bottleneck/SPOF

Challenges:

  • Cross-shard queries / joins — very expensive, avoided by denormalisation
  • Cross-shard transactions — 2PC or saga required
  • Rebalancing — adding a shard means moving data
  • Schema changes — must apply to all shards
7
Explain ACID vs BASE. What is eventual consistency?
Medium▼

ACID — Atomicity (all or nothing), Consistency (valid state before and after), Isolation (concurrent tx don't interfere), Durability (committed data survives failures). Guaranteed by most relational DBs.

BASE — Basically Available, Soft state, Eventually consistent. NoSQL trade-off: sacrifice strong consistency for availability and performance.

Eventual consistency — replicas may temporarily return stale data but will eventually converge to the same value. The window might be milliseconds (Cassandra within a data center) to seconds (cross-region).

ℹ️
ACID in microservices
ACID guarantees end at the service boundary. Cross-service operations can't use database transactions — this is why patterns like Saga and Outbox exist.
8
What is the Outbox Pattern? How does it solve dual-write problems?
Hard▼

Dual-write problem: Writing to a DB and publishing a Kafka message in two separate operations — if the app crashes between them, they go out of sync.

Outbox pattern: Write the event to an outbox table in the same DB transaction as the business data. A separate poller or CDC (Change Data Capture) tool reads the outbox and publishes to Kafka — exactly-once delivery.

SQL
-- Outbox table
CREATE TABLE outbox_events (
  id          UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  event_type  VARCHAR(100) NOT NULL,
  payload     JSONB        NOT NULL,
  created_at  TIMESTAMPTZ  DEFAULT now(),
  published   BOOLEAN      DEFAULT FALSE
);

-- In a single transaction:
BEGIN;
  INSERT INTO orders (id, user_id, total) VALUES (...);
  INSERT INTO outbox_events (event_type, payload)
    VALUES ('ORDER_PLACED', '{"orderId": "...", "userId": "..."}');
COMMIT;

-- Debezium (CDC) watches outbox_events and publishes to Kafka
-- Zero risk of message loss or duplication

Caching Strategies

Cache patterns, eviction, Redis, CDN

9
What are the main caching patterns? When do you use Cache-Aside vs Write-Through?
Hard▼
PatternHowGood forRisk
Cache-Aside (Lazy)App checks cache first; on miss, reads DB, populates cacheRead-heavy, tolerate eventual consistencyCache stampede on cold start; stale data
Write-ThroughWrite to cache and DB simultaneouslyData that will be read soon after writeWrite latency doubles; cache fills with unread data
Write-Behind (Write-Back)Write to cache; asynchronously flush to DB laterWrite-heavy workloads (gaming leaderboards)Data loss if cache fails before flush
Read-ThroughCache sits in front of DB; on miss, cache fetches from DBTransparent caching via frameworkComplex cache logic; cold start problem
⚠️
Cache stampede
When many requests hit an expired cache key simultaneously and all go to the DB. Fix: probabilistic early expiration, distributed lock (Redis SETNX), or "cache warming" before expiry.

What are Redis cache eviction policies? Which would you use for a session store vs a product catalog?
Medium▼
PolicyWhat gets evictedUse case
noevictionNothing — returns error when fullMessage queues where data loss is unacceptable
allkeys-lruLeast recently used across all keysGeneral caching — most common choice
volatile-lruLRU among keys with TTL setMix of cached data and persistent data in one Redis

Session store: volatile-ttl — oldest sessions expire first, mimicking sliding window behaviour.
Product catalog: allkeys-lfu — popular products stay in cache; rarely-viewed products evicted.

Messaging & Event Streaming

Kafka, queues, event-driven architecture

11
How does Apache Kafka work? What are Topics, Partitions, Consumer Groups, and Offsets?
Hard▼
  • Topic — named category of messages (like a table); append-only, durable log
  • Partition — unit of parallelism within a topic; messages ordered within a partition; distributed across brokers
  • Offset — each message's position within a partition; consumers track their offset (usually in __consumer_offsets topic)
  • Consumer Group — group of consumers sharing a topic's partitions; each partition consumed by exactly one consumer in the group (parallelism = # partitions)
Java
// Spring Kafka producer
@Service
public class OrderEventProducer {
    @Autowired private KafkaTemplate<String, OrderEvent> kafkaTemplate;

    public void publish(OrderEvent event) {
        kafkaTemplate.send("orders", event.getOrderId().toString(), event)
            .whenComplete((result, ex) -> {
                if (ex != null) log.error("Failed to send", ex);
            });
    }
}

// Spring Kafka consumer
@KafkaListener(topics = "orders", groupId = "inventory-service")
public void onOrderPlaced(@Payload OrderEvent event,
                          @Header(KafkaHeaders.RECEIVED_PARTITION) int partition) {
    log.info("Processing order {} from partition {}", event.getOrderId(), partition);
    inventoryService.reserveItems(event);
}
💡
Key trade-off
Kafka ordering is guaranteed only within a partition. To ensure all events for one orderId are ordered, use orderId as the message key — Kafka routes same-key messages to the same partition.
12
What is at-most-once, at-least-once, and exactly-once delivery? How does Kafka achieve exactly-once?
Hard▼
SemanticsMessages may be?How
At-most-onceLost (never duplicated)Commit offset before processing; fire and forget
At-least-onceDuplicated (never lost)Commit offset only after successful processing; retry on failure
Exactly-onceNeither lost nor duplicatedIdempotent producers + transactional API; most complex and slowest

Practically: Most systems use at-least-once + idempotent consumers (check if event already processed by storing event ID in DB). This is simpler and has most of the same guarantees.

Java
// Idempotent consumer — safe to process duplicate messages
@KafkaListener(topics = "orders")
@Transactional
public void process(OrderEvent event) {
    // Upsert — idempotent by design
    if (processedEventRepo.existsById(event.getEventId())) {
        log.info("Skipping duplicate event {}", event.getEventId());
        return;
    }
    inventoryService.reserve(event);
    processedEventRepo.save(new ProcessedEvent(event.getEventId()));
}

Architecture Patterns

Microservices, SOLID, design patterns

13
What are the SOLID principles? Give a Java example for each.
Medium▼
Java
// S — Single Responsibility: one class, one reason to change
class UserService   { void createUser()  {} }   // business logic only
class UserEmailer  { void sendWelcome() {} }   // email concern separate
class UserRepository{ void save()        {} }   // persistence separate

// O — Open/Closed: open for extension, closed for modification
interface Discount   { double apply(double price); }
class NoDiscount    implements Discount { ... }
class VIPDiscount   implements Discount { ... }   // extend without modifying

// L — Liskov Substitution: subtype must be usable in place of parent
class Rectangle  { void setWidth(int w) {} void setHeight(int h) {} }
// 🚫 class Square extends Rectangle breaks LSP — setting width changes height
// Fix: separate Shape interface, no inheritance between Square and Rectangle

// I — Interface Segregation: many small interfaces > one fat interface
interface Readable  { String read(); }
interface Writable  { void   write(String data); }
// ReadonlyFile implements only Readable, not forced to implement write()

// D — Dependency Inversion: depend on abstractions, not concretions
interface NotificationSender { void send(String msg); }
class EmailSender implements NotificationSender { ... }
class SmsSender   implements NotificationSender { ... }
class AlertService {
    private final NotificationSender sender; // inject abstraction
    public AlertService(NotificationSender s) { this.sender = s; }
}

16
Design a rate limiter. What algorithms would you use?
Hard▼

Algorithms:

  • Fixed Window Counter — count requests in fixed time window (e.g. 100/min). Simple but bursts at window edges (98 at end of minute + 98 at start of next = 196 effectively)
  • Sliding Window Log — log each request timestamp; count within sliding window. Accurate but memory-intensive
  • Token Bucket — bucket holds N tokens; each request consumes one; tokens refilled at rate R/sec. Allows controlled bursting. Most common in practice
  • Leaky Bucket — requests queue; processed at constant rate. Smooths bursts completely; old requests dropped if queue full
Java
// Token Bucket with Redis (distributed rate limiter)
// Lua script — atomic check + decrement
private static final String RATE_LIMIT_SCRIPT = """
    local tokens = tonumber(redis.call('get', KEYS[1]) or ARGV[1])
    if tokens > 0 then
        redis.call('set', KEYS[1], tokens - 1, 'EX', ARGV[2])
        return 1  -- allowed
    else
        return 0  -- rejected
    end
    """;

public boolean isAllowed(String userId) {
    String key = "rate_limit:" + userId;
    Long result = redisTemplate.execute(
        RedisScript.of(RATE_LIMIT_SCRIPT, Long.class),
        List.of(key),
        "100",   // max tokens
        "60"    // window in seconds
    );
    return Long.valueOf(1).equals(result);
}
17
How would you design a notification system (email, SMS, push) at scale?
Hard▼

Requirements: 10M notifications/day, multi-channel (email/SMS/push), user preferences, guaranteed delivery, retries.

Architecture:

  1. Notification Service API — receives notification requests, validates, stores in DB
  2. Preference Service — determines which channels a user wants for each notification type
  3. Priority Queue (Kafka topics per priority) — routes to channel-specific workers
  4. Channel Workers — Email Worker (SES/SendGrid), SMS Worker (Twilio), Push Worker (FCM/APNs)
  5. Retry with exponential backoff — failed notifications re-enqueued with delay
  6. Delivery tracking — webhooks from providers update delivery status in DB
💡
Key design decisions
Deduplication: use idempotency keys to prevent duplicate sends on retry. Rate limiting: respect per-provider limits (e.g. SES 14 emails/sec on sandbox). Templating: store template IDs + variables in DB, render at send time for localisation.
18
What is a CDN? How does it improve performance and what are its limitations?
Easy▼

A CDN (Content Delivery Network) caches static assets at edge servers geographically close to users, reducing latency and origin server load.

How it works: User requests cdn.example.com/logo.png → DNS routes to nearest PoP (Point of Presence) → If cached: serve immediately. If not: fetch from origin, cache at edge, serve.

Good for: Static assets (JS, CSS, images, videos), API responses with Cache-Control headers, geographic distribution.

Limitations:

  • Not suitable for personalised or authenticated content (though edge auth is evolving)
  • Cache invalidation is hard — purging stale content needs CDN API calls
  • Cost at high volume
  • Single point of trust — CDN sees all your traffic (mitigate with TLS, SRI hashes)
19
What is the Service Mesh pattern? What does Istio provide that Kubernetes doesn't?
Hard▼

A Service Mesh adds a sidecar proxy (Envoy) to every Pod. All traffic goes through the proxy, enabling infrastructure-level observability, security, and traffic control — without changing application code.

FeatureKubernetes aloneWith Istio
Mutual TLS (mTLS)Manual certificate managementAutomatic per-service identity and mTLS
Traffic splittingBy pod count onlyWeight-based (10%/90% canary)
ObservabilityBasic metricsGolden signals per service pair, trace spans
Retries/timeoutsApp codeDeclarative policies in YAML
Fault injectionNot availableInject delays/errors for chaos testing
⚠️
Complexity cost
Istio adds ~50ms latency per hop (proxy overhead) and significant operational complexity. Don't adopt a service mesh until you have ≥10 services and clear pain points it solves — most teams reach for it too early.
20
How would you design a system to handle 1 million concurrent WebSocket connections?
Hard▼

Challenge: 1M persistent connections — traditional thread-per-connection model won't scale (1M threads × 1MB stack = 1TB RAM).

Solution components:

  1. Non-blocking I/O — Netty/Vert.x or Spring WebFlux with Reactor Netty (event loop, not thread per connection)
  2. Horizontal scaling — 20 WebSocket servers × 50K connections each = 1M. Route with sticky L7 load balancer (same client → same server)
  3. Message fan-out — when server A needs to send to user on server B: publish to Redis Pub/Sub; all servers subscribe; the one holding the connection delivers
  4. Presence tracking — Redis hash map of userId → serverId
  5. Heartbeat + reconnect — client sends ping every 30s; server disconnects on timeout; client auto-reconnects with exponential backoff
ℹ️
Java 21 alternative
With virtual threads (Project Loom) you can use blocking code with 1M threads — JVM handles the multiplexing. Eliminates the need for reactive programming while achieving the same scale.
