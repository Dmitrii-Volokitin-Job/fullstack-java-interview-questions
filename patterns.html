SOLID Principles

The 5 object-oriented design rules that make code maintainable, testable, and extensible

Q1 Explain all 5 SOLID principles with a real Java example each Hard ›

S — Single Responsibility Principle
A class should have only one reason to change. If your OrderService validates, persists, and sends emails — three teams can break it independently.

// ❌ Too many responsibilities
class OrderService {
    void placeOrder(Order o) {
        validate(o);          // validation logic
        orderRepo.save(o);    // persistence logic
        emailService.send(o); // notification logic
    }
}

// ✅ Each class owns one concern
class OrderValidator   { void validate(Order o) { ... } }
class OrderRepository  { void save(Order o)     { ... } }
class OrderNotifier    { void notify(Order o)   { ... } }
class OrderService     { // orchestrates only
    void placeOrder(Order o) {
        validator.validate(o);
        repo.save(o);
        notifier.notify(o);
    }
}

O — Open/Closed Principle
Open for extension, closed for modification. Add behaviour by adding code, not changing existing code. Strategy and Decorator patterns are the primary tools.

// ❌ Adding a new discount requires editing this class
class DiscountCalculator {
    double calculate(Order o, String type) {
        if (type.equals("SUMMER")) return o.total() * 0.9;
        if (type.equals("VIP"))    return o.total() * 0.8;
        return o.total();
    }
}

// ✅ Add new discounts without touching existing code
interface DiscountStrategy { double apply(double total); }
class SummerDiscount implements DiscountStrategy { public double apply(double t) { return t * 0.9; } }
class VipDiscount    implements DiscountStrategy { public double apply(double t) { return t * 0.8; } }
class BlackFridayDiscount implements DiscountStrategy { ... } // NEW — zero edits to existing

L — Liskov Substitution Principle
A subtype must be usable wherever the parent is used, without surprising the caller. Violated when overriding a method weakens its contract (throws unexpected exceptions, ignores params).

// ❌ Violates LSP — Square constrains Rectangle's contract
class Rectangle { void setWidth(int w); void setHeight(int h); }
class Square extends Rectangle {
    void setWidth(int w)  { width = height = w; }  // breaks caller's assumption
    void setHeight(int h) { width = height = h; }  // that w and h are independent
}

// ✅ Fix: prefer composition or separate type hierarchy
interface Shape { int area(); }
record Rectangle(int w, int h) implements Shape { public int area() { return w * h; } }
record Square(int side)        implements Shape { public int area() { return side * side; } }

I — Interface Segregation Principle
Clients should not depend on methods they don't use. Fat interfaces force implementors to add stub/throw methods.

// ❌ Fat interface — ReadOnlyRepo is forced to throw on write methods
interface OrderRepository {
    Order findById(Long id);
    void save(Order o);
    void delete(Long id);
}
class ReadOnlyOrderRepo implements OrderRepository {
    public void save(Order o)   { throw new UnsupportedOperationException(); } // ❌
    public void delete(Long id) { throw new UnsupportedOperationException(); } // ❌
}

// ✅ Segregated — each client depends only on what it needs
interface OrderReader  { Order findById(Long id); }
interface OrderWriter  { void save(Order o); void delete(Long id); }
interface OrderRepository extends OrderReader, OrderWriter {}

D — Dependency Inversion Principle
High-level modules should depend on abstractions, not concretions. This is the foundation of Spring's IoC container — you inject interfaces, not concrete classes.

// ❌ OrderService hard-codes its dependency on a concrete class
class OrderService {
    private final MySqlOrderRepository repo = new MySqlOrderRepository(); // ❌
}

// ✅ Depend on abstraction — Spring injects the concrete implementation
@Service
class OrderService {
    private final OrderRepository repo;  // interface, not concrete class
    OrderService(OrderRepository repo) { this.repo = repo; }
}
// Spring wires MySqlOrderRepository (or InMemoryOrderRepository in tests)
Q2 What are DRY, KISS, and YAGNI — and when does following them conflict with each other? Medium ›

DRY — Don't Repeat Yourself: every piece of knowledge should have a single, authoritative representation. Violations create divergence bugs when one copy is updated and another isn't.

KISS — Keep It Simple, Stupid: prefer the simplest solution that works. Complexity is a cost — it slows reading, testing, and debugging. If a HashMap solves it, don't write a trie.

YAGNI — You Aren't Gonna Need It: don't build features until they're actually required. The cost of unused abstractions compounds as the codebase grows.

The conflict: DRY pushes you to extract shared abstractions (anti-KISS, anti-YAGNI). Classic trap: two methods look similar → you extract a shared base class → they diverge later → the abstraction fights you.

Rule of three: wait until the third duplication before extracting. Two similar things might just be coincidence. 
// DRY violation — same validation logic in 3 controllers
if (email == null || !email.contains("@")) throw ...
// Fix: @Email annotation on DTO + @Valid on @RequestBody — one source of truth

// YAGNI violation — adding a plugin system "just in case"
interface PaymentProviderPlugin { ... }       // nobody will write plugins
class PluginRegistry { ... }                  // you're maintaining this forever
// Fix: use a concrete StripePaymentService until a second provider appears

Q3 Singleton — implementation, thread safety, and why Spring beans are NOT the same thing Medium ›

Singleton ensures exactly one instance of a class exists in the JVM, with a global access point.

// Thread-safe — initialization-on-demand holder (best practice)
public class ConfigService {
    private ConfigService() {}   // private constructor

    private static class Holder {
        static final ConfigService INSTANCE = new ConfigService();
    }

    public static ConfigService getInstance() {
        return Holder.INSTANCE;  // JVM class loading guarantees thread-safety
    }
}
Spring beans vs Singleton pattern: Spring's default scope is "singleton per ApplicationContext" — one instance per container, not per JVM. If you create two contexts (e.g. in tests), you get two instances. The classic Singleton pattern enforces one per JVM classloader. In Spring, prefer injecting beans over calling getInstance().
When NOT to use: Singletons with mutable state are a concurrency bug waiting to happen. If the class holds state that changes per-request (e.g. current user), use request-scoped beans or ThreadLocal instead.
Q4 Factory Method vs Abstract Factory — difference and when to use each Medium ›

Factory Method defines an interface for creating an object but lets subclasses decide which class to instantiate. One product, one factory method.

Abstract Factory creates families of related objects without specifying concrete classes. Multiple products, one factory interface.

// Factory Method — Spring @Bean methods are factory methods
@Bean
public PaymentGateway paymentGateway(@Value("${payment.provider}") String provider) {
    return switch (provider) {
        case "stripe"  -> new StripeGateway();
        case "paypal"  -> new PayPalGateway();
        default       -> throw new IllegalArgumentException("Unknown: " + provider);
    };
}

// Abstract Factory — creates a family of UI components for a theme
interface UiFactory {
    Button createButton();
    Dialog createDialog();
    TextField createTextField();
}
class MaterialUiFactory implements UiFactory { ... }
class FluenceUiFactory  implements UiFactory { ... }
Interview tip: Factory Method = one product type, subclasses choose the concrete type. Abstract Factory = multiple product types that must be consistent (a "family"). If someone asks "which do you use more?", say Factory Method — it's everywhere in Spring (@Bean, BeanFactory, FactoryBean).
Q5 Builder pattern — why use it and how Lombok @Builder works under the hood Easy ›

Builder separates construction of a complex object from its representation. Solves the telescoping constructor antipattern — constructors with many parameters become unreadable and error-prone.

// ❌ Telescoping constructor — which arg is which?
new Order(1L, "PAID", null, true, 99.99, "EUR", LocalDate.now(), null);

// ✅ Builder — readable, optional fields are obvious
Order order = Order.builder()
    .customerId(1L)
    .status(PAID)
    .amount(BigDecimal.valueOf(99.99))
    .currency("EUR")
    .build();

// Lombok @Builder generates this at compile time:
@Builder
@Data
class Order {
    private Long customerId;
    private OrderStatus status;
    private BigDecimal amount;
    private String currency;
    // Lombok generates: static OrderBuilder builder(), all withX() methods, build()
}

// Validation in build() — use @Builder with compact validation
Order order = Order.builder()
    .amount(amount)
    .build();  // add @NonNull on fields → Lombok checks at build() time
In Spring: ResponseEntity.ok().header(...).body(dto) is the Builder pattern. UriComponentsBuilder, MockMvcRequestBuilders — all Builders. Know it cold.

Structural Patterns

How classes and objects are composed to form larger structures

Q6 Adapter pattern — real example integrating a third-party payment SDK Easy ›

Adapter converts the interface of a class into one the client expects. It lets incompatible interfaces work together — like a power adapter for different socket types.

// Target interface — what our code expects
interface PaymentGateway {
    PaymentResult charge(BigDecimal amount, String currency);
}

// Adaptee — third-party Stripe SDK with different interface
class StripeClient {                          // we can't modify this
    StripeCharge createCharge(long amountCents, String currency, String desc) { ... }
}

// Adapter — wraps Stripe, speaks our PaymentGateway language
@Component
class StripePaymentAdapter implements PaymentGateway {
    private final StripeClient stripe;

    public PaymentResult charge(BigDecimal amount, String currency) {
        long cents = amount.multiply(BigDecimal.valueOf(100)).longValue();
        StripeCharge charge = stripe.createCharge(cents, currency, "Order payment");
        return new PaymentResult(charge.getId(), charge.getStatus());
    }
}

// OrderService never knows it's talking to Stripe
@Service
class OrderService {
    private final PaymentGateway gateway;  // injected — Spring wires StripePaymentAdapter
}
Q7 Decorator pattern — how Spring AOP, caching, and security filters all use it Medium ›

Decorator wraps an object to add behaviour without subclassing. It's structurally the same as the wrapped object (implements same interface), so it's transparent to the caller. Decorators can be stacked — each adds one concern.

// Spring @Cacheable IS a decorator — it wraps your method at runtime
@Service
class OrderService {
    @Cacheable("orders")           // Spring generates a CachingDecorator proxy
    @PreAuthorize("hasRole('ADMIN')") // SecurityDecorator proxy wraps it
    public Order findById(Long id) { ... }
}

// Manual decorator — wrapping OrderRepository with audit logging
class AuditingOrderRepository implements OrderRepository {
    private final OrderRepository delegate;

    public Order findById(Long id) {
        log.info("findById({})", id);
        Order o = delegate.findById(id);     // delegate to real impl
        log.info("found: {}", o.getStatus());
        return o;
    }
}
Decorator vs Inheritance: Inheritance is static (baked in at compile time). Decorator is dynamic — you compose behaviours at runtime. Java I/O streams are the classic example: new BufferedReader(new InputStreamReader(socket.getInputStream())).

Q9 Facade pattern — simplifying a subsystem and the BFF (Backend for Frontend) connection Easy ›

Facade provides a simplified interface to a complex subsystem. It doesn't add new behaviour — it reduces complexity for the caller. A BFF (Backend for Frontend) is a Facade at the network level.

// Subsystem — 4 services the mobile app would need to orchestrate
class OrderCheckoutFacade {
    private final InventoryService inventory;
    private final PaymentService payment;
    private final NotificationService notification;
    private final OrderRepository orderRepo;

    // Mobile app calls ONE method instead of orchestrating 4 services
    public CheckoutResult checkout(Cart cart, PaymentDetails payment) {
        inventory.reserve(cart.items());
        PaymentResult paid = payment.charge(cart.total());
        Order order = orderRepo.save(Order.from(cart, paid));
        notification.sendConfirmation(order);
        return CheckoutResult.success(order.getId());
    }
}

// BFF pattern — same idea at REST level (Spring API Gateway / separate service)
@RestController
class MobileBffController {
    @PostMapping("/checkout")
    CheckoutResult checkout(@RequestBody CartDto cart) {
        return checkoutFacade.checkout(cart);  // one call, 4 services behind the scenes
    }
}

Behavioral Patterns

How objects communicate and distribute responsibility

Q10 Strategy pattern — the most important GoF pattern for Spring developers Easy ›

Strategy defines a family of algorithms, encapsulates each one, and makes them interchangeable. The client picks which strategy to use at runtime — without changing the code that uses it.

// Strategy interface — in Java 8+ this is just a functional interface
@FunctionalInterface
interface DiscountStrategy { double apply(double total); }

// Concrete strategies — can also be lambdas
class SummerDiscount     implements DiscountStrategy { public double apply(double t) { return t * .9; } }
class VipDiscount        implements DiscountStrategy { public double apply(double t) { return t * .8; } }
DiscountStrategy noDiscount = total -> total;   // lambda IS a strategy

// Context — inject strategy, don't hard-code it
@Service
class CheckoutService {
    public double calculateTotal(Order order, DiscountStrategy strategy) {
        return strategy.apply(order.subtotal());   // decoupled from strategy choice
    }
}

// Spring: select strategy at runtime from a map of beans
@Service
class DiscountRouter {
    private final Map<String, DiscountStrategy> strategies;  // injected by Spring

    public DiscountStrategy select(Customer c) {
        if (c.isVip()) return strategies.get("vipDiscount");
        if (isSummer()) return strategies.get("summerDiscount");
        return strategies.get("noDiscount");
    }
}
Q11 Observer / Event pattern — Spring ApplicationEvent, Kafka, and when to use each Medium ›

Observer lets objects subscribe to events from a publisher without the publisher knowing who's listening. Decouples the event source from event handlers.

// Spring in-process events — same JVM, synchronous by default
record OrderPlacedEvent(Long orderId, String customerEmail) extends ApplicationEvent {
    OrderPlacedEvent(Object src, Long orderId, String customerEmail) {
        super(src); this.orderId = orderId; this.customerEmail = customerEmail;
    }
}

// Publisher — doesn't know who's listening
@Service
class OrderService {
    private final ApplicationEventPublisher events;

    public Order place(Order order) {
        repo.save(order);
        events.publishEvent(new OrderPlacedEvent(this, order.getId(), order.getEmail()));
        return order;
    }
}

// Subscribers — independently handle the event
@EventListener
public void sendConfirmation(OrderPlacedEvent e) { emailService.send(e.customerEmail()); }

@EventListener
@Async   // run in separate thread pool
public void updateInventory(OrderPlacedEvent e) { inventoryService.reserve(e.orderId()); }

Q12 Template Method — how Spring's JdbcTemplate, RestTemplate, and AbstractController use it Medium ›

Template Method defines the skeleton of an algorithm in a base class, letting subclasses override specific steps without changing the overall structure. The base class calls the hook methods — subclasses fill them in.

// Base class defines the algorithm skeleton
abstract class DataExporter {
    // Template method — the algorithm skeleton
    public final void export(List<Order> orders) {
        List<String> rows = formatRows(orders);   // hook — subclass defines
        String header    = buildHeader();           // hook — subclass defines
        writeOutput(header, rows);                  // hook — subclass defines
    }

    protected abstract List<String> formatRows(List<Order> orders);
    protected abstract String buildHeader();
    protected abstract void writeOutput(String header, List<String> rows);
}

class CsvExporter  extends DataExporter { // fills in CSV-specific steps
    protected List<String> formatRows(List<Order> o) { return o.stream().map(Order::toCsv).toList(); }
    protected String buildHeader() { return "id,status,total"; }
    protected void writeOutput(String h, List<String> rows) { Files.writeString(csvPath, h + "\n" + String.join("\n", rows)); }
}

// Spring JdbcTemplate is a Template Method — you supply only the SQL and RowMapper
List<Order> orders = jdbcTemplate.query(
    "SELECT * FROM orders WHERE status = ?",
    (rs, i) -> new Order(rs.getLong("id"), rs.getString("status")),  // hook
    "PAID"
);
// JdbcTemplate handles: connection, statement, result iteration, exception translation
Q13 Chain of Responsibility — Spring Security filter chain and request validation pipelines Medium ›

Chain of Responsibility passes a request along a chain of handlers. Each handler decides to process it, pass it on, or stop the chain. Spring Security's filter chain is the prime real-world example.

// Generic handler interface
interface OrderHandler {
    void setNext(OrderHandler next);
    void handle(Order order);
}

abstract class BaseOrderHandler implements OrderHandler {
    private OrderHandler next;
    public void setNext(OrderHandler h) { this.next = h; }
    protected void passToNext(Order o) { if (next != null) next.handle(o); }
}

class StockCheckHandler extends BaseOrderHandler {
    public void handle(Order o) {
        if (!stock.available(o)) throw new OutOfStockException();
        passToNext(o);  // all good — pass along the chain
    }
}

class FraudCheckHandler extends BaseOrderHandler {
    public void handle(Order o) {
        if (fraud.isRisky(o)) throw new FraudSuspectedException();
        passToNext(o);
    }
}

// Build the chain
StockCheckHandler stock = new StockCheckHandler();
FraudCheckHandler fraud = new FraudCheckHandler();
stock.setNext(fraud);
stock.handle(order);  // enters at head, flows through

// Spring Security: filters are Chain of Responsibility — each calls chain.doFilter()
@Component
class JwtAuthFilter extends OncePerRequestFilter {
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res, FilterChain chain)
            throws IOException, ServletException {
        String token = extractToken(req);
        if (tokenValidator.isValid(token)) {
            SecurityContextHolder.getContext().setAuthentication(buildAuth(token));
        }
        chain.doFilter(req, res);  // pass to next filter in chain
    }
}
Q14 Command pattern — undo/redo, queueing operations, and CQRS connection Medium ›

Command encapsulates an action as an object. Benefits: decouple sender from receiver, queue/log commands, implement undo/redo, retry failed operations.

interface Command { void execute(); void undo(); }

record PlaceOrderCommand(Order order, OrderRepository repo) implements Command {
    public void execute() { repo.save(order); }
    public void undo()    { repo.delete(order.getId()); }
}

// Command queue — process asynchronously, supports retry
class CommandBus {
    private final Queue<Command> queue = new LinkedList<>();
    private final Deque<Command> history = new ArrayDeque<>();

    public void dispatch(Command cmd) {
        cmd.execute();
        history.push(cmd);
    }

    public void undoLast() {
        if (!history.isEmpty()) history.pop().undo();
    }
}

// PlaceOrderCommand → CommandHandler → aggregate root → domain events

Architecture Patterns

High-level blueprints for organising entire systems or services

Q15 Layered Architecture — the N-tier model and its problems at scale Easy ›

Rule: each layer only depends on the layer directly below it. A Controller must not call a Repository directly.

Problems at scale: Layered architecture becomes a distributed monolith in microservices if teams share the same database layer. Also, "anemic domain model" — all logic ends up in the service layer, entities are just data bags. Fix with Domain-Driven Design and Hexagonal Architecture.
Q16 Hexagonal Architecture (Ports & Adapters) — the cleanest separation for testable services Hard ›

The application core (domain + use cases) is completely isolated from infrastructure. It exposes ports (interfaces) and the outside world connects via adapters. Inverting dependencies means your core never imports Spring, JPA, or any framework.

// Port — defined in the core, no infrastructure dependency
interface OrderRepository {   // lives in domain layer
    Order save(Order order);
    Optional<Order> findById(Long id);
}

// Driven adapter — lives in infrastructure layer, implements the port
@Repository
class JpaOrderRepository implements OrderRepository {
    private final JpaOrderJpaRepo jpaRepo;  // Spring Data JPA — kept in infra only
    public Order save(Order o) { return jpaRepo.save(OrderEntity.from(o)).toDomain(); }
}

// Use case (core service) — depends only on the port, never on JPA
class PlaceOrderUseCase {
    private final OrderRepository repo;   // interface injection — core unaware of JPA
    Order execute(PlaceOrderCommand cmd) {
        Order order = Order.create(cmd);
        return repo.save(order);
    }
}

// In tests — swap the adapter for an in-memory fake
var useCase = new PlaceOrderUseCase(new InMemoryOrderRepository());  // zero Spring context
Q17 CQRS — Command Query Responsibility Segregation with eventual consistency trade-offs Hard ›

CQRS separates read (Query) and write (Command) models. Commands mutate state, Queries read it — from separate models, often separate datastores. Enables independent scaling and optimised read models.

// Write side — Command + Handler
record PlaceOrderCommand(Long customerId, List<CartItem> items) {}

@Service
class PlaceOrderHandler {
    @Transactional
    public Long handle(PlaceOrderCommand cmd) {
        Order order = Order.create(cmd.customerId(), cmd.items());
        orderRepo.save(order);
        eventBus.publish(new OrderPlacedEvent(order));  // triggers read-model update
        return order.getId();
    }
}

// Read side — separate query model, optimised for display
record OrderListView(Long id, String customerName, String status, BigDecimal total) {}

@Service
class OrderQueryHandler {
    public List<OrderListView> findByCustomer(Long customerId) {
        return orderViewRepo.findByCustomerId(customerId);  // read from denormalised view
    }
}
Trade-off: eventual consistency. The read model lags behind the write model by milliseconds to seconds. The caller that just placed an order might not see it in the list immediately. Always design the UI for this.
Q18 Event Sourcing — storing events instead of state, and when it's overkill Hard ›

Instead of storing the current state of an entity, Event Sourcing stores the sequence of events that led to that state. Current state is derived by replaying events.

// Events — immutable facts that happened
sealed interface OrderEvent permits OrderPlaced, OrderShipped, OrderCancelled {}
record OrderPlaced(Long orderId, Long customerId, Instant at) implements OrderEvent {}
record OrderShipped(Long orderId, String trackingCode, Instant at) implements OrderEvent {}
record OrderCancelled(Long orderId, String reason, Instant at) implements OrderEvent {}

// Event store — append-only table
-- order_events: order_id | event_type | payload (JSONB) | occurred_at
-- 1001 | ORDER_PLACED  | {customerId:1}          | 2025-06-01T10:00
-- 1001 | ORDER_SHIPPED | {tracking:"DHL123"}     | 2025-06-03T08:00

// Rebuild current state by replaying events
class Order {
    OrderStatus status;

    Order apply(OrderEvent event) {
        return switch (event) {
            case OrderPlaced   e -> { this.status = PENDING;  yield this; }
            case OrderShipped  e -> { this.status = SHIPPED;  yield this; }
            case OrderCancelled e -> { this.status = CANCELLED; yield this; }
        };
    }
    static Order rebuild(List<OrderEvent> events) {
        return events.stream().reduce(new Order(), Order::apply, (a,b) -> b);
    }
}
Use Event Sourcing whenAvoid it when
Complete audit history is required (finance, healthcare)You only need current state
You need temporal queries ("what was the state on Jan 3?")Simple CRUD with no audit needs
Multiple read models need different projectionsTeam isn't familiar with eventual consistency
Used with CQRS + Kafka in event-driven systemsSchema of events is unstable / changes often
Q20 Saga pattern — Choreography vs Orchestration for distributed transactions Hard ›

Saga manages long-running distributed transactions without 2-phase commit. Each step publishes an event/command; on failure, compensating transactions undo previous steps.

Choreography

OrderService → OrderPlaced event
    ↓
PaymentService listens → charges card
    → PaymentDone event
    ↓
InventoryService listens → reserves items
    → ItemsReserved event
    ↓
ShippingService listens → creates shipment

// On failure: PaymentFailed event
// InventoryService listens and releases
Pros: loose coupling, no central coordinator
Cons: hard to visualise full flow, cyclic dependencies creep in

OrderSaga (orchestrator)
  1. send PayCommand → PaymentService
  2. on PaymentDone: send ReserveCmd → Inventory
  3. on ItemsReserved: send ShipCmd → Shipping
  4. on any failure:

     send RefundCmd → Payment

Q20 Clean Architecture — how it relates to Hexagonal and DDD, key differences from Layered Hard ›

Clean Architecture (Robert Martin) organises code in concentric rings where the Dependency Rule states: source code dependencies must point inward. Inner rings know nothing of outer rings.

// Entities (innermost) — pure Java, no annotations, no framework
class Order {                           // knows nothing of Spring, JPA, HTTP
    private OrderStatus status = PENDING;
    void confirm() {
        if (status != PENDING) throw new InvalidOrderStateException();
        status = CONFIRMED;
    }
}

// Use Cases — orchestrate entities, define port interfaces
class ConfirmOrderUseCase {
    private final OrderRepository repo;  // port (interface) — defined here, implemented in adapters
    void execute(Long orderId) {
        Order o = repo.findById(orderId).orElseThrow();
        o.confirm();
        repo.save(o);
    }
}

// Interface Adapters — Spring controllers, JPA repositories
@RestController class OrderController { // converts HTTP ↔ use-case DTOs
    @PutMapping("/orders/{id}/confirm")
    ResponseEntity<Void> confirm(@PathVariable Long id) {
        confirmOrderUseCase.execute(id);
        return ResponseEntity.ok().build();
    }
}

Q21 Load Balancer algorithms — Round Robin, Least Connections, IP Hash, Weighted and when to pick each Medium ›
AlgorithmHow it worksBest for
Round Robin Requests cycle through servers in sequence: A → B → C → A… Homogeneous servers, stateless services, uniform request cost
Weighted Round Robin Servers get requests proportional to their weight: A(3), B(1) → A gets 3×more Heterogeneous servers (different CPU/RAM), canary deployments (give new version weight=1)
Least Connections New request goes to the server with fewest active connections Long-lived connections (WebSocket, streaming), variable request duration
IP Hash (Sticky) hash(clientIP) % numServers — same client always hits same server Stateful sessions, caches that must stay warm, WebSocket upgrades
Random Pick a server at random each request Simple, surprisingly effective when server pool is large and homogeneous
Sticky sessions (IP Hash) gotcha: If one server goes down, all its sticky clients suddenly need redistribution — you get a traffic spike on other servers. Also, IP Hash breaks behind NAT where many clients share one IP. Use a session cookie header (AWSALB, INGRESSCOOKIE) as the sticky key instead of IP when possible.

Health Checks are periodic probes the LB sends to each backend. A backend failing N consecutive checks is removed from rotation (marked unhealthy).

  periodSeconds: 10
  failureThreshold: 3   # 3 consecutive failures → kill & restart

// Readiness checks DB, Kafka, Redis connectivity

Connection Draining (Deregistration Delay): When a backend is removed (deployment, scale-down), the LB stops sending new requests but lets in-flight requests complete within a timeout (default 300s on AWS ALB). Prevents 5xx errors during rolling deploys.

A2 Two Pointers pattern — when to use it and 3 classic problems it solves Medium ›

Two pointers uses two index variables moving through the same array — either toward each other (opposite ends) or in the same direction (fast/slow). Turns O(n²) nested loops into O(n).

// Pattern 1: Opposite ends — check if a SORTED array has a pair summing to target
boolean hasPairSum(int[] arr, int target) {
    int left = 0, right = arr.length - 1;
    while (left < right) {
        int sum = arr[left] + arr[right];
        if (sum == target) return true;
        if (sum < target) left++;   // need bigger sum
        else              right--;  // need smaller sum
    }
    return false;
}  // O(n) time, O(1) space — vs O(n²) brute force

// Pattern 2: Fast/slow — detect a cycle in a linked list (Floyd's algorithm)
boolean hasCycle(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;          // moves 1 step
        fast = fast.next.next;     // moves 2 steps
        if (slow == fast) return true;  // they meet → cycle
    }
    return false;
}

// Pattern 3: Same direction — remove duplicates from sorted array in-place
int removeDuplicates(int[] arr) {
    if (arr.length == 0) return 0;
    int slow = 0;
    for (int fast = 1; fast < arr.length; fast++) {
        if (arr[fast] != arr[slow]) arr[++slow] = arr[fast];
    }
    return slow + 1;
}
Trigger words: "sorted array", "pair/triplet sum", "palindrome check", "linked list cycle", "remove duplicates in-place" → reach for two pointers first.
A3 Sliding Window pattern — longest substring without repeating characters Medium ›

Sliding window maintains a contiguous subarray/substring between a left and right pointer. You expand right to grow the window and shrink left when a constraint is violated. Avoids recalculating the whole window from scratch each step.

// Longest substring without repeating characters — O(n) time, O(k) space (k = charset size)
int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> lastSeen = new HashMap<>();
    int left = 0, maxLen = 0;

    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        if (lastSeen.containsKey(c) && lastSeen.get(c) >= left) {
            left = lastSeen.get(c) + 1;  // shrink window past the duplicate
        }
        lastSeen.put(c, right);
        maxLen = Math.max(maxLen, right - left + 1);
    }
    return maxLen;
}
// "abcabcbb" → 3 ("abc")
// "pwwkew"   → 3 ("wke")
Trigger words: "subarray/substring with condition", "maximum/minimum window", "contains at most K distinct" → sliding window.
A4 HashMap patterns — frequency count, grouping, and two-sum in O(n) Easy ›

HashMap gives O(1) average lookup/insert — the most common tool for trading space for time. Three patterns cover 80% of HashMap interview problems.

// Pattern 1: Two Sum — find indices of two numbers that add to target
int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> seen = new HashMap<>();  // value → index
    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];
        if (seen.containsKey(complement)) return new int[]{seen.get(complement), i};
        seen.put(nums[i], i);
    }
    throw new IllegalArgumentException("no solution");
}  // O(n) vs O(n²) brute force

// Pattern 2: Frequency count — is one string an anagram of another?
boolean isAnagram(String s, String t) {
    if (s.length() != t.length()) return false;
    Map<Character, Integer> freq = new HashMap<>();
    for (char c : s.toCharArray()) freq.merge(c, 1, Integer::sum);
    for (char c : t.toCharArray()) {
        freq.merge(c, -1, Integer::sum);
        if (freq.get(c) < 0) return false;
    }
    return true;
}

// Pattern 3: Grouping — group anagrams together
List<List<String>> groupAnagrams(String[] words) {
    Map<String, List<String>> groups = new HashMap<>();
    for (String w : words) {
        char[] ch = w.toCharArray(); Arrays.sort(ch);
        String key = new String(ch);            // sorted chars = canonical key
        groups.computeIfAbsent(key, k -> new ArrayList<>()).add(w);
    }
    return new ArrayList<>(groups.values());
}
A5 Binary Search — implementation, common bugs, and the "search in rotated array" variant Medium ›

Binary search halves the search space each iteration → O(log n). Works on any monotonic structure, not just sorted arrays.

// Classic binary search — returns index or -1
int binarySearch(int[] arr, int target) {
    int lo = 0, hi = arr.length - 1;
    while (lo <= hi) {                         // ≤ not < — handles single element
        int mid = lo + (hi - lo) / 2;           // avoids int overflow vs (lo+hi)/2
        if (arr[mid] == target) return mid;
        if (arr[mid] < target)  lo = mid + 1;
        else                    hi = mid - 1;
    }
    return -1;
}

// Variant: rotated sorted array [4,5,6,7,0,1,2], find target
int searchRotated(int[] arr, int target) {
    int lo = 0, hi = arr.length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if (arr[mid] == target) return mid;
        // Determine which half is sorted
        if (arr[lo] <= arr[mid]) {             // left half sorted
            if (target >= arr[lo] && target < arr[mid]) hi = mid - 1;
            else lo = mid + 1;
        } else {                               // right half sorted
            if (target > arr[mid] && target <= arr[hi]) lo = mid + 1;
            else hi = mid - 1;
        }
    }
    return -1;
}
3 classic bugs: (1) lo < hi instead of lo <= hi → misses single-element case. (2) (lo + hi) / 2 → integer overflow when lo+hi > Integer.MAX_VALUE. (3) lo = mid instead of lo = mid + 1 → infinite loop when lo==hi-1.

A7 Recursion — how to think about it, base cases, and avoiding StackOverflowError Medium ›

Every recursive solution needs: (1) a base case that stops recursion, (2) a recursive case that moves toward it. Think "trust the recursion" — assume the function works for smaller inputs and build on that.

// Fibonacci — naïve O(2ⁿ) vs memoized O(n)
int fibNaive(int n) {
    if (n <= 1) return n;                           // base case
    return fibNaive(n - 1) + fibNaive(n - 2);        // O(2ⁿ) — recalculates same values
}

int[] memo = new int[101];
int fib(int n) {
    if (n <= 1) return n;
    if (memo[n] != 0) return memo[n];              // already computed
    return memo[n] = fib(n - 1) + fib(n - 2);       // O(n) with memoization
}

// Tree height — trusting the recursion
int height(TreeNode node) {
    if (node == null) return 0;                    // base case
    return 1 + Math.max(height(node.left), height(node.right));
}  // "assume height(left) works" → just add 1 for current node
StackOverflowError — Java's default thread stack is ~512KB. Deep recursion on large input (n > ~5000) hits it. Fix: convert to iteration with an explicit Deque stack, or use tail-call optimization manually. In Spring Boot, increase stack size with -Xss2m JVM flag if needed.
A8 Stack & Queue — valid parentheses, monotonic stack, and BFS vs DFS choice Medium › 
// Classic stack — valid parentheses O(n)
boolean isValid(String s) {
    Deque<Character> stack = new ArrayDeque<>();
    for (char c : s.toCharArray()) {
        if ("({[".indexOf(c) >= 0) { stack.push(c); continue; }
        if (stack.isEmpty()) return false;
        char top = stack.pop();
        if (c == ')' && top != '(') return false;
        if (c == '}' && top != '{') return false;
        if (c == ']' && top != '[') return false;
    }
    return stack.isEmpty();
}

// BFS with Queue — shortest path in unweighted graph
int bfs(int[][] grid, int[] start, int[] end) {
    Queue<int[]> queue = new LinkedList<>();
    queue.add(new int[]{start[0], start[1], 0});
    boolean[][] visited = new boolean[grid.length][grid[0].length];
    int[][] dirs = {{0,1},{0,-1},{1,0},{-1,0}};
    while (!queue.isEmpty()) {
        int[] cur = queue.poll();
        if (cur[0] == end[0] && cur[1] == end[1]) return cur[2];
        for (int[] d : dirs) {
            int r = cur[0] + d[0], c = cur[1] + d[1];
            if (r >= 0 && r < grid.length && c >= 0 && c < grid[0].length
                && grid[r][c] == 0 && !visited[r][c]) {
                visited[r][c] = true;
                queue.add(new int[]{r, c, cur[2] + 1});
            }
        }
    }
    return -1;
}
