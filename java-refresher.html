Java 8 — LTS Functional Programming enters Java
The problem before Java 8: To filter a list of orders by status you needed a for loop, a temporary list, and an anonymous Comparator class. Three constructs for one logical idea. Java 8 let you say what you want, not how to iterate.

4 lambda syntax forms:

// 1. No params
Runnable r = () -> System.out.println("done");

// 2. One param (parens optional)
Predicate<Order> isPaid = o -> o.getStatus() == PAID;

// 3. Multiple params
Comparator<Order> byDate = (a, b) -> a.getCreatedAt().compareTo(b.getCreatedAt());

// 4. Method reference — shorthand when lambda just calls one method
Function<Order, OrderStatus> getStatus = Order::getStatus;
Consumer<Order> printer = System.out::println;
Effectively final rule: a lambda can read a local variable only if that variable is never reassigned. The compiler enforces this so lambdas are safe to pass across threads without synchronization. 
OrderStatus filter = PAID;   // effectively final — OK
// filter = SHIPPED;          // uncomment → compile error inside lambda below
Predicate<Order> p = o -> o.getStatus() == filter;

Using the 4 core interfaces in OrderService:

public class OrderService {

  // Predicate — "does this order qualify?"
  private Predicate<Order> isHighValue =
      o -> o.totalAmount().compareTo(BigDecimal.valueOf(500)) > 0;

  // Function — "transform order to summary string"
  private Function<Order, String> toSummary =
      o -> o.getId() + " — " + o.getStatus() + " — €" + o.totalAmount();

  // Consumer — "send email for this order"
  private Consumer<Order> sendConfirmation =
      o -> emailService.send(o.getCustomer().getEmail(), toSummary.apply(o));

  // Supplier — "build default order if none found"
  private Supplier<Order> emptyOrder = Order::new;

  public void processHighValueOrders(List<Order> orders) {
    orders.stream()
        .filter(isHighValue)            // Predicate
        .peek(sendConfirmation)         // Consumer — side effect in pipeline
        .map(toSummary)                 // Function
        .forEach(System.out::println);  // method reference Consumer
  }
}
Streams — Lazy Pipelines
Process collections without mutation, evaluated only on terminal call
›
UML — Stream Pipeline anatomy

Stream pipeline visualised for our order list:

orders.stream() → .filter(isPaid) → .map(Order::getItems) → .flatMap(List::stream) → .map(OrderItem::getProduct) → .distinct() → .collect(toList()) 
// All products bought in PAID orders (no duplicates)
List<Product> boughtProducts = orders.stream()
    .filter(o -> o.getStatus() == PAID)
    .map(Order::getItems)               // Stream<List<OrderItem>>
    .flatMap(List::stream)              // Stream<OrderItem> — flatten
    .map(OrderItem::getProduct)         // Stream<Product>
    .distinct()                         // by equals()/hashCode()
    .collect(Collectors.toList());
flatMap vs map: map turns each element into one thing. flatMap turns each element into a stream and merges them all. Use flatMap whenever you have "a list inside a list" you need to flatten.

Optional — Explicit Null Safety
Forces callers to handle the "nothing found" case at compile time
›
Flowchart — Optional decision tree
❌ Without Optional — NullPointerException waiting
Order order = orderRepo.findById(id);
// What if nothing found? NPE below!
String city = order.getCustomer()
                   .getAddress()
                   .getCity();
✅ With Optional — safe chain
String city = orderRepo.findById(id)
    .map(Order::getCustomer)
    .map(Customer::getAddress)
    .map(Address::getCity)
    .orElse("Unknown");
// Repository returns Optional — signals "may not exist"
public Optional<Order> findById(Long id) {
    return orders.stream()
        .filter(o -> o.getId().equals(id))
        .findFirst();   // terminal that returns Optional<T>
}

// Controller layer — 3 safe patterns

// Pattern 1: orElseThrow — most common in REST
Order order = findById(42L)
    .orElseThrow(() -> new OrderNotFoundException(42L));

// Pattern 2: orElseGet — lazy default (supplier only called if empty)
Order order = findById(42L)
    .orElseGet(() -> createDraftOrder(customerId));

// Pattern 3: ifPresentOrElse — side-effect branch (Java 9+)
findById(42L).ifPresentOrElse(
    o  -> log.info("Found: {}", o.getId()),
    () -> log.warn("Order 42 not found")
);

// Filter inside Optional
Optional<Order> paidOrder = findById(42L)
    .filter(o -> o.getStatus() == PAID);  // empty if status isn't PAID
Never call .get() without .isPresent() first. It throws NoSuchElementException — just as bad as NPE. Use orElseThrow() instead — it lets you control the exception type and message.
Default & Static Interface Methods
Add behaviour to interfaces without breaking all existing implementations
›
UML — Interface with default methods (diamond problem)
interface Discountable {
    double discountRate();   // abstract — implementors MUST provide this

    default BigDecimal discountedPrice(BigDecimal price) {
        // Default — uses discountRate() which subtype provides
        return price.multiply(BigDecimal.valueOf(1 - discountRate()));
    }

    static Discountable noDiscount() {   // factory on the interface
        return () -> 0.0;
    }
}

interface Taxable {
    default BigDecimal discountedPrice(BigDecimal price) {  // SAME method name!
        return price;
    }
}

// Diamond problem: both interfaces have discountedPrice()
class DigitalProduct implements Discountable, Taxable {
    public double discountRate() { return 0.1; }

    @Override
    public BigDecimal discountedPrice(BigDecimal price) {
        return Discountable.super.discountedPrice(price);  // pick one explicitly
    }
}

// Usage
Discountable d = Discountable.noDiscount();  // static factory
d.discountedPrice(BigDecimal.valueOf(100));   // 100.0 (0% off)
Real-world use: Java's own Comparator interface uses default methods extensively — .thenComparing(), .reversed() are defaults. You chain them without subclassing: Comparator.comparing(Order::getCreatedAt).reversed().thenComparing(Order::getId)

Java 11 — LTS Standard library polish
Context: Java 11 didn't change the language dramatically — it polished the standard library. Reading a config file went from 5 lines to 1. Checking for blank strings got a proper Unicode-aware method. The new HttpClient replaced Apache HttpClient for simple cases.
var in lambda parameters
Allows annotations on inferred lambda param types
›
Why does this exist? Before Java 11 you couldn't annotate a lambda parameter whose type was inferred. Adding var lets you write (@NonNull var order) -> ... — the type is still inferred, but annotations like @NonNull, @NotNull, or custom validation annotations now work. 
// Java 10 — var in local variables (not lambdas)
var orders = orderRepo.findAll();   // inferred as List<Order>

// Java 11 — var in lambda params (for annotations)
orders.stream()
    .filter((@NonNull var o) -> o.getStatus() == PAID)
    .forEach((@NotNull var o) -> emailService.notify(o));

// ⚠️ Rule: if you use var for one param, ALL params must use var
BiFunction<Order, Customer, String> label =
    (var o, var c) -> o.getId() + " for " + c.getName(); // OK
// (var o, Customer c) → compile error — mixed not allowed

Java 14–16 Data modelling revolution
Context: These releases introduced the features that most changed how Java code looks. Records eliminated DTO boilerplate. Pattern matching killed redundant casts. Sealed classes gave the compiler full knowledge of your type hierarchy — enabling exhaustive checks that prevent entire bug classes.
Records — Immutable Data Carriers
Auto-generates constructor · accessors · equals · hashCode · toString
›
UML — Record auto-generation
❌ Before Records — DTO boilerplate
public final class OrderSummary {
  private final Long id;
  private final OrderStatus status;
  private final BigDecimal total;

  public OrderSummary(Long id,
      OrderStatus status,
      BigDecimal total) {
    this.id = id;
    this.status = status;
    this.total = total;
  }
  public Long getId() { return id; }
  public OrderStatus getStatus()
      { return status; }
  public BigDecimal getTotal()
      { return total; }
  // + equals(), hashCode(), toString()
  // 40+ lines total
}
✅ Java 16 Record — 1 line
record OrderSummary(
    Long id,
    OrderStatus status,
    BigDecimal total
) {}

// Usage — accessors without "get"
OrderSummary s = new OrderSummary(
    1L, PAID, BigDecimal.valueOf(99.99));

s.id();       // Long — not getId()
s.status();   // OrderStatus
s.total();    // BigDecimal

System.out.println(s);
// OrderSummary[id=1, status=PAID, total=99.99]
// Compact constructor — add validation, no need to assign fields manually
record MoneyAmount(BigDecimal value, String currency) {
    MoneyAmount {   // no parentheses — compact form, fields auto-assigned after
        if (value.compareTo(BigDecimal.ZERO) < 0)
            throw new IllegalArgumentException("Amount cannot be negative");
        currency = currency.toUpperCase();   // can normalise before assignment
    }
}

// Record implementing an interface
interface Exportable { String toCsv(); }

record OrderSummary(Long id, OrderStatus status, BigDecimal total)
        implements Exportable {
    @Override
    public String toCsv() {
        return id + "," + status + "," + total;
    }

    // Static factory — conventional for records
    static OrderSummary from(Order o) {
        return new OrderSummary(o.getId(), o.getStatus(), o.totalAmount());
    }
}

// Used in service to return lightweight projection
List<OrderSummary> summaries = orders.stream()
    .map(OrderSummary::from)
    .collect(Collectors.toList());
Records vs Lombok @Value: Records are language-level (no annotation processor needed). Lombok's @Value does the same but via compile-time generation. If you're already on Java 16+, prefer records for DTOs — they're more explicit and supported natively by Jackson, Spring, and JPA projections.

Sealed Classes & Interfaces
Restrict who can extend — compiler knows all subtypes → exhaustive switch
›
UML — Sealed PaymentResult hierarchy
// Sealed interface — only these 3 records can implement it
sealed interface PaymentResult
    permits Success, Failure, Pending {}

record Success(String transactionId, BigDecimal amount, Instant paidAt)
    implements PaymentResult {}

record Failure(String reason, String errorCode)
    implements PaymentResult {}

record Pending(Duration estimatedTime, Instant retryAfter)
    implements PaymentResult {}

// Pattern switch — exhaustive because compiler knows all 3 cases (Java 21 stable)
public OrderStatus toOrderStatus(PaymentResult result) {
    return switch (result) {
        case Success s  -> {
            log.info("Payment {} confirmed: €{}", s.transactionId(), s.amount());
            yield OrderStatus.PAID;
        }
        case Failure f  -> {
            log.error("Payment failed: {} ({})", f.reason(), f.errorCode());
            yield OrderStatus.CANCELLED;
        }
        case Pending p  -> {
            log.info("Payment pending, retry after {}", p.retryAfter());
            yield OrderStatus.PENDING;
        }
        // No default needed! Compiler knows all 3 subtypes.
        // If you add Refunded to permits, this switch becomes a compile error.
    };
}

// The 3 subtype modifiers
sealed class    Vehicle permits Car, Truck {}   // continues restricting
final class     Car     extends Vehicle {}       // nobody can extend Car
non-sealed class Truck  extends Vehicle {}       // anyone can extend Truck
Sealed + Records + Pattern switch = the new Visitor pattern. Before Java 17 you'd implement the Visitor design pattern (double dispatch, lots of boilerplate) to safely handle a closed type hierarchy. Now: declare a sealed interface, make each case a record, use switch with patterns. Same safety guarantee, 80% less code.
